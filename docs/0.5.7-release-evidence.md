# 0.5.7 Release Evidence

Release date: 2026-07-22. Workspace package and binary version: `0.5.7`.
Implementation baseline: commit `6f256c6` plus the scoped 0.5.7 candidate diff.

## Root Cause And Correction

The built-in `explore`, `review`, and `check` profiles described read-only code
inspection but declared an empty tool list. Delegated profile application treats
an empty list as an explicit deny-all allowlist, so the child capability builder
correctly released neither tools nor filesystem authority.

The correction declares exactly `read`, `grep`, `find`, and `ls` for all three
helpers. The runtime security model is unchanged:

```text
effective child tools = helper allowlist ∩ parent frozen tool capabilities
```

Filesystem authority is released only when that intersection contains a
filesystem tool. The helpers receive no `write`, `edit`, `bash`, delegation,
session read/write, or UI capability. Authorization still approves only an
already admitted invocation and cannot widen a frozen capability snapshot.

## Regression Evidence

- The built-in profile owner asserts the exact four-tool list for each helper
  and the absence of mutation/shell tools.
- The delegated capability owner constructs all three helpers from a permissive
  parent and proves the four read-only tools plus filesystem are present while
  shell, mutation, delegation, session, and UI authority remain absent.
- The product operation integration test delegates to built-in `explore`, has
  the child call `read` on a workspace fixture, verifies the result reaches its
  continuation, and proves an unrelated parent-only tool is not inherited.
- Existing restricted-parent tests continue to prove intersection semantics and
  filesystem removal when no filesystem tool survives.
