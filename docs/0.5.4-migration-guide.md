# Migration Guide: 0.5.3 to 0.5.4

The 0.5.4 release changes delegation from a post-turn request/execute split to
one awaited tool invocation and adds operation-scoped child conversation pages
to the fullscreen TUI.

## Delegation Tool Results

`delegate_agent` and `delegate_team` no longer return a provisional
`{"status":"requested"}` result to the parent model. Every newly admitted tool
call returns one terminal structured result with status `completed`, `failed`,
`rejected`, or `cancelled`. The result retains the source operation/tool-call
identity, target identity, task, and (when execution starts) child operation ID.

Automatic approval executes the child before the parent model continues.
Interactive confirmation uses the existing tool-authorization waiter, so an RPC
client now approves or denies a new delegation confirmation through
`approve_tool_authorization` / `deny_tool_authorization`. The original parent
tool future remains suspended and resumes after that decision and child result.
The older `approve_delegation` / `reject_delegation` operations remain available
only for pending confirmation facts restored from a 0.5.3 session.

Print/JSON or other non-interactive callers cannot leave a new delegation in a
provisional state. A policy requiring confirmation produces a terminal
`rejected` result explaining that confirmation is required.

## Child Tool Authorization

Delegated child prompts inherit the session AuthorizationService with their own
operation capability snapshot. A child filesystem, shell, or side-effect tool
request therefore carries the child Prompt operation ID and capability
generation. Approving delegation itself does not implicitly approve child tool
permissions.

RPC continues to publish child ProductEvents with their real operation IDs.
The parent model receives only the terminal delegation tool result; permission
control events are never inserted into its conversation.

## Fullscreen TUI

The main conversation renders one compact delegation status row. Child
assistant messages, thinking, tool calls, results, and authorization previews do
not enter the main transcript. Select a delegation row and press Enter to switch
the whole conversation viewport to the child page; press Escape to return.
Child pages are conversation views, not inline expansion bodies, and their
permission controls are shown only while that child page is active. While the
main page is active, a child permission request changes only the compact status
to `waiting permission`.

Retention is bounded to 32 child conversations, 2,048 queued UI events per
conversation, and 1,024 transcript items. Snapshot recovery includes the
bounded retained child ProductEvent window and folds nested Prompt/member events
under the root operation's direct delegation child, so AgentTeam members share
one team page.

## Protocol Inventory

| Protocol family | 0.5.3 | 0.5.4 |
| --- | --- | --- |
| RPC | `2.1` | `2.1` |
| ProductEvent | `2.2` | `2.2` |
| UI snapshot | `2.1` | `2.2` |

UI snapshot `2.2` adds the bounded retained child-event projection used during
fresh-snapshot reconnect. The major version remains `2`; clients that ignore
the additive field remain wire-compatible, while clients implementing child
pages should consume the 2.2 projection.

## Embedding And Persistence

No stable facade import changes are required. Durable
`delegation.folded.updated` replay adds the `cancelled` status and now records
terminal state for awaited automatic/interactive execution and legacy restored
approval. Live ProductEvents remain notifications; SessionEvents remain the
durable source of the main folded delegation state.
