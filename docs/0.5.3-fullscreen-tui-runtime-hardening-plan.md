# Fullscreen TUI Runtime And Lint Hardening Plan For 0.5.3

## Status And Scope

Target workspace version: **`0.5.3`**. Status: **Completed 2026-07-21**.
Implementation began from annotated release `v0.5.2` (`3ca7ad8`). The
workspace advanced to `0.5.3` only after the implementation, debt ledger,
changelogs, API/protocol evidence, and release gates closed together under
`FTRH-012`.

This plan hardens the existing fullscreen interactive adapter and generic
terminal runtime. It also completes the workspace `dead_code` audit left after
the 0.5.2 lean-runtime release. It is subordinate to the normative contracts in
[`architecture/principles.md`](architecture/principles.md),
[`architecture/dependency-rules.md`](architecture/dependency-rules.md),
[`architecture/runtime.md`](architecture/runtime.md),
[`architecture/extension-platform.md`](architecture/extension-platform.md), and
[`architecture/testing.md`](architecture/testing.md).

The target owner path remains:

```text
typed input / resize / timer / client event
  -> one interactive event reducer
  -> InteractiveLocalState + ordered UiProjection
  -> bounded viewport render model
  -> pi-tui Frame / overlay / terminal transaction
  -> one terminal writer
```

`pi-tui` owns generic terminal lifecycle, input normalization, layout,
compositing, cursor placement, and rendering mechanics. `pi-coding-agent` owns
product commands, operation tasks, ProductEvent/snapshot projection, client
connection policy, transcript presentation, and fullscreen product layout.
Neither crate may acquire the other's semantics to avoid defining a boundary.

## Architectural Decisions

- The fullscreen TUI remains a peer adapter over snapshots, ProductEvents,
  reconnect cursors, typed controls, and operation outcomes. It does not read
  repositories, runners, EventHub internals, or provider streams directly.
- The interactive adapter becomes a real long-lived client connection. A task
  completion is not a substitute for replay/live event consumption, cursor
  advancement, acknowledgement, detach, and fresh-snapshot recovery.
- Exactly one owner writes terminal bytes. Progress keepalive, synchronized
  frame output, image cleanup, title updates, and terminal teardown cannot race
  through independent stdout writers.
- Input, task events, client events, and terminal work are bounded. Continuous
  deltas may be coalesced, but lifecycle, authorization, terminal, recovery, and
  acknowledgement semantics cannot be dropped.
- Fullscreen transcript rendering is viewport-first. Cache hits must avoid
  cloning and assembling the complete transcript merely to discard off-screen
  rows.
- `#[allow(dead_code)]` is not architecture documentation. Retained exceptions
  must be narrow, carry a stable `reason`, and correspond to an accepted wire,
  FFI, skipped-scope, or test-fixture contract. Module/crate-wide production
  suppression is prohibited.
- This plan does not resume skipped Extension contribution dispatch, Workbench,
  state/facts, background services, or complete Extension DX. It only preserves
  and accurately represents the accepted minimum Extension runtime.

An allowance occurrence can suppress more than one item; the count is not a
count of dead symbols. The ordinary single-lint search finds 67 occurrences;
the remaining occurrence is the Win32 module's combined
`allow(non_snake_case, dead_code)` attribute. `FTRH-001` must capture
compiler-selected builds and the actual warnings revealed when each allowance
group is removed.

Current logic risks frozen by this plan are:

- terminal progress writes concurrently with synchronized fullscreen frames;
- stopping progress can synchronously join a sleeping thread for nearly one
  second, and changing `show_progress` while running can skip cleanup;
- PromptTask deltas cross an unbounded adapter channel;
- input events remaining in the same parsed chunk are discarded once an event
  starts a task;
- stdin EOF during a running task does not flush `StdinBuffer`;
- the interactive `CodingAgentClientConnection` is connected/detached but its
  reconnect/live receiver and acknowledgements are not consumed;
- terminal I/O failure after synchronized-update start may omit the matching
  end sequence;
- shutdown input drain sleeps without actually draining and the reader thread
  cannot be cancelled;
- every fullscreen render walks and clones all transcript blocks and clones the
  complete per-block view maps before selecting the visible viewport;
- fullscreen differential rendering clears and rewrites the complete suffix
  even when the changed range is small;
- running and idle event-loop branches duplicate input, resize, render, theme,
  EOF, detach, and exit behavior and have already diverged.

## Objectives

- make terminal output single-owner, cancellation-safe, and error-safe;
- preserve every parsed input event and give stdin shutdown one explicit owner;
- bound/coalesce high-rate task and client event delivery without losing
  semantic events;
- consume the canonical client snapshot/reconnect/live/ack boundary throughout
  the interactive lifetime;
- unify the running and idle event-loop mechanics around typed effects;
- replace primary resize polling with typed resize events while retaining a
  bounded compatibility fallback where required;
- make fullscreen transcript cost proportional to the viewport and changed
  blocks rather than total session history;
- preserve overlay focus, scroll anchoring, mouse hit regions, Unicode width,
  images, terminal cleanup, and machine-output isolation;
- remove stale and redundant dead-code suppressions, consume protocol fields
  that carry correctness semantics, and document the few retained exceptions;
- complete version, changelog, architecture evidence, snapshots, pressure/fault
  matrices, and release validation together.

## Non-Goals

- replacing `pi-tui` with another terminal framework;
- moving product commands, ProductEvents, sessions, profiles, authorization, or
  transcript semantics into `pi-tui`;
- moving terminal components, concrete layouts, or adapter state into the
  stable product API;
- changing the SessionEvent/outbox/finalization owner graph;
- introducing an unbounded queue elsewhere to hide backpressure;
- using random sleeps or uncontrolled chaos as the primary correctness gate;
- requiring a live or paid provider in ordinary tests;
- deleting provider wire fields without a fixture-backed consume/validate/
  ignore decision;
- resuming skipped Extension contribution dispatch or Workbench work merely to
  make retained boundary types appear used;
- retaining obsolete code solely to keep an old test helper or private method.

## Detailed Acceptance

### `FTRH-001`: Reproducible Baseline And Failure Inventory

- Record idle, streaming, tool-update, overlay, resize, and long-transcript
  render latency, allocations, bytes written, full/differential redraw counts,
  event queue high-water marks, and shutdown latency under
  `target/perf-baseline/0.5.3-fullscreen-tui/`.
- Use at least short, 1,000-block, and 10,000-block deterministic transcripts;
  measure unchanged spinner/status frames separately from changed blocks.
- Inventory every PromptTask producer, ProductEvent producer, stdin producer,
  render wake, terminal writer, progress transition, connection cursor, and ack.
- Remove allowance groups in compiler-selected diagnostic builds and record the
  exact symbols each group suppresses. Do not infer symbol counts from source
  attribute counts.
- Freeze queue capacities and quantitative regression thresholds before changing
  behavior. Correctness assertions must not depend on uncontrolled wall time.

### `FTRH-002`: Single Terminal Writer And Progress Lifecycle

- Remove the progress stdout keepalive thread. Progress updates are scheduled by
  the interactive loop or a generic terminal command source consumed by the
  sole `Tui<T>` owner.
- Track actual progress activation independently from mutable settings. Start,
  finish, abort, setup failure, input EOF, render error, and exit clear it
  exactly once even when `show_progress` changes during a task.
- Do not block Tokio workers with `thread::sleep` or `JoinHandle::join` during
  normal operation completion or shutdown.
- Keep terminal progress generic in `pi-tui`; keep the policy deciding when an
  operation is active in `pi-coding-agent`.
- Add a terminal-operation trace proving progress writes cannot occur inside a
  synchronized frame transaction and that task completion has no keepalive
  interval-sized latency tail.

### `FTRH-003`: Input Ownership And Lossless Parsing

- Preserve parsed events remaining after an earlier event starts a task. Queue
  them for the next reduction or continue processing them under running-state
  policy; never silently drop the remainder of a chunk.
- Route running and idle EOF through one handler that flushes `StdinBuffer`
  exactly once, processes the resulting events, and then chooses drain/wait/exit.
- Replace the detached blocking stdin reader with a cancellable input owner.
  Shutdown disables terminal protocols, consumes pending key-release input to an
  idle/deadline boundary, closes the channel, and joins without hanging.
- Use a bounded channel or a directly awaited event source. Paste/IME payloads
  remain intact and are not split by byte-oriented backpressure.
- Add deterministic cases for multiple events in one read, partial escape EOF,
  bracketed paste plus submit, submit plus abort/follow-up, input closure during
  a task, and TUI re-entry in one process.

### `FTRH-004`: Bounded Task And Client Event Pressure

- Eliminate the PromptTask event channel in favor of the canonical client
  receiver. If an operation-local bridge is ever reintroduced, it must use one
  reviewed bounded contract with recorded capacity and saturation behavior.
- Classify events as non-droppable semantic events, coalescible deltas, or
  replaceable latest snapshots. Terminal, authorization, recovery, tool
  lifecycle, operation association, and cursor evidence are non-droppable.
- Do not coalesce already sequenced ProductEvents in the PromptTask bridge;
  their cursors are correctness evidence. Any adjacent text/thinking/tool
  coalescing must happen before product sequence allocation or after projection
  as render invalidation, remain within the same admitted operation and semantic
  block, and be fenced by cancellation and terminal generations.
- Drain a bounded batch per wake so high-rate output does not starve input,
  cancellation, resize, authorization, client control, or render deadlines.
- Expose test-only queue high-water evidence without adding production-global
  mutable metrics or public adapter internals.

### `FTRH-005`: Canonical Interactive Client Connection

- Establish the interactive connection when a session becomes active and seed
  the root from its public snapshot/cursor.
- Consume replay plus live ProductEvents in sequence order, apply them to the
  single `UiProjection`, and acknowledge only after successful application.
- Handle retained gaps and stream/protocol mismatch with typed fresh-snapshot
  recovery. Never repair storage or infer a terminal in the adapter.
- Switching/forking sessions detaches the old generation and installs the new
  snapshot/receiver atomically from the adapter's perspective.
- PromptTask completion returns owned operation/session results but does not
  create a parallel per-operation projection truth. Remove duplicated temporary
  subscriptions after parity is proven.
- Cover idle background publication, concurrent embedding mutation, disconnect,
  replay/live handoff, duplicate delivery, stale generation, slow receiver,
  fresh snapshot, and shutdown drain.

### `FTRH-006`: One Interactive Event Loop

- Replace duplicated running/idle select blocks with one event-source set and
  one typed reducer. Optional task event/completion/spinner sources are enabled
  only while a task exists; input, client, resize, theme, render, and shutdown
  remain single paths.
- Represent event-loop effects explicitly: update local state, submit/abort,
  apply client event, request render, update terminal progress, switch session,
  detach, or exit.
- Preserve explicit mutable owners and borrow scopes; do not introduce a generic
  workflow engine, service container, or string action bus.
- Prefer typed resize events from the input/terminal owner. A portability
  fallback may poll at a bounded low rate only when event delivery is unavailable.
- Prevent closed theme/input/client sources from causing a ready-loop spin.

### `FTRH-007`: Failure-Safe Terminal Transactions

- Introduce a synchronized-render guard so every successful start has a
  best-effort end on normal return, I/O error, panic unwind, and `Tui::stop`.
- `ProcessTerminal::stop` defensively disables synchronized output, progress,
  mouse capture, keyboard protocols, bracketed paste, alternate screen, cursor
  hiding, and raw mode without one cleanup failure skipping later cleanup.
- Suspend or emit an empty valid frame for zero-column/zero-row terminals;
  overlays cannot manufacture a one-column line when available width is zero.
- Preserve cursor and Kitty image cleanup when a changed-range write fails.
- Add fail-on-Nth-terminal-operation matrices for full redraw, differential
  redraw, overlay composition, image deletion, progress, stop, and resize.

### `FTRH-008`: Viewport-First Fullscreen Rendering

- Use cached row metadata to identify viewport-intersecting blocks before
  cloning/rendering lines. Render only visible blocks plus a small fixed
  overscan; retain exact scroll anchors and block selection.
- Avoid cloning `TranscriptViewSnapshot` maps every frame. Borrow immutable view
  state or share a revisioned snapshot that changes only with view revision.
- Separate conversation-panel invalidation from spinner/status/composer/overlay
  invalidation so an unchanged transcript is not reassembled at 120 ms cadence.
- In a fixed-height fullscreen frame, rewrite the actual changed range rather
  than clearing the entire suffix unless image/protocol constraints require a
  reviewed fallback.
- Preserve ANSI boundaries, Unicode display width, Markdown/theme changes,
  image rows, mouse hit regions, collapsed/expanded block rows, and new-output
  indicators.
- Prove steady-state fullscreen rendering is proportional to viewport rows and
  changed blocks, not total transcript history.

### `FTRH-009`: Pressure And Fault Injection

- Use typed test failpoints at terminal writes, input closure, task-event send,
  client replay/live handoff, acknowledgement, resize, and shutdown boundaries.
- Use virtual clocks and deterministic schedules for render, spinner, progress,
  resize fallback, input fragment, and client event ordering.
- Test slow terminal, event saturation, resize storms, continuous provider/tool
  deltas, cancellation, authorization while saturated, session switch, render
  error, disconnect/reconnect, and shutdown at each owned phase.
- Assert bounded memory/queues, no deadlock, no lost semantic event, no duplicate
  terminal, monotonic cursor/generation, fenced late results, clean terminal
  teardown, and protocol-clean stdout for non-interactive modes.
- Add a bounded offline soak gate. Live-provider testing remains optional and
  explicitly authorized, never a normal release prerequisite.
  `scripts/tui-runtime-soak.sh` owns the fixed `0.5.3-mixed-v1` schedule and
  defaults to 20 iterations with a per-command hard deadline.

## `dead_code` Decision Ledger

The action vocabulary is:

- **Retain**: the item is required by an accepted current or explicitly skipped
  architecture contract; keep only a narrow reasoned allowance when production
  reachability cannot honestly represent that contract.
- **Update**: the item/module is required, but the allowance hides missing use,
  missing validation, wrong ownership, or overly broad suppression.
- **Delete**: the item exists only for obsolete/private/test convenience and has
  no supported architecture or wire obligation. Move to `cfg(test)` counts as
  deletion from the ordinary production path.
