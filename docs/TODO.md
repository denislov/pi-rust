# pi-rust TODO

This file is the working checklist for the Flow-centered `pi-rust` architecture.

`pi-rust` is no longer treated as a direct TypeScript `pi` port. The TypeScript `pi` repo remains a behavioral and product reference. PocketFlow remains an architectural reference for explicit graph-shaped orchestration. The Rust project should use both references to build a clearer, more principled agent runtime.

## Update Rule

Update this TODO whenever a task completes meaningful progress against an item below.

Required updates:

- Mark completed items with `[x]`.
- Mark active partial work with `[~]` and add a short note.
- Add new discovered work under the relevant phase instead of leaving it implicit.
- Add a dated progress note when a phase boundary, guide step, or major design decision changes.
- Include TODO updates in the same commit as the implementation or documentation change that changes status.

Do not let this file become historical fiction. If implementation changes the plan, update this file and the relevant guide/spec together.

- 2026-07-07: Provider usage/cache accounting correctness fix. Root cause of the fast-growing footer `input` token count: the Rust Anthropic provider only set `cache_control: ephemeral` on the system prompt and never on the conversation history, so every turn re-sent the full history as non-cached `input_tokens` (billed at the full input rate), and the accumulated `usage.input` grew with conversation length. Anthropic's `input_tokens` already excludes cached tokens, so the statistics were faithfully reporting what was billed — the fix is to enable history caching, not to subtract history from the stat. `anthropic::convert::build_request` now attaches `cache_control: ephemeral` to the last content block of the last user-role message (mirroring TS `anthropic-messages.ts` `addCacheControlToLastConversationMessage`), so history is served from the prompt cache on subsequent turns and `input_tokens` holds only genuinely new tokens. Secondary fix: OpenAI completions `map_usage` set `input = prompt_tokens`, but OpenAI's `prompt_tokens` includes the cached subset, double-billing cached tokens at both input and cache_read rates; it now sets `input = prompt_tokens - cached_tokens`. Regression coverage added for both providers. Follow-up (not in this slice): gate caching behind a TS-style `cacheRetention` option, and cache the last tool definition via `supports_cache_control_on_tools`.
