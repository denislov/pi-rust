# Extension Services, State, And Durability Plan For 0.4.3

## Status And Scope

Target workspace version: **`0.4.3`**. Status: **Skipped by the 2026-07-19
Extension scope freeze**.

All `ESS-*` and `REL-043-001` tasks in this plan are **Skipped**. The detailed
acceptance is retained only as historical design input; none of it is implied to
be implemented or release-blocking. Resumption requires a new reviewed plan.
No `0.4.3` workspace version or changelog release is produced by this skipped
plan; the train remains at the completed `0.4.2` release.

This release adds stateful and long-lived extension services on the sole
TypeScript/Wasm kernel: asynchronous operations, typed hooks, subscriptions,
managed background jobs, scoped state, durable facts, coordinated package update,
and operable diagnostics. It does not implement the portable Workbench.

See the [`0.4.x` roadmap](0.4.x-architecture-extension-platform-roadmap.md).

## Objectives And Non-Goals

- global/workspace/session/branch/ephemeral state with explicit durability;
- `ExtensionFact` as opaque historical session evidence;
- session/branch mutations through the single SessionCoordinator transaction;
- async operations/tools/hooks with progress, typed errors, bounds, and leases;
- host-owned subscriptions and managed background jobs;
- crash-recoverable, generation-fenced package update, migration, drain,
  activation pointer, pre-activation rollback, and post-activation forward
  recovery without a fictional cross-store transaction;
- version negotiation completed for state/facts/Host API families;
- audit/tracing/log/source-map/trap/redaction contract (`ADR-010`).

Workbench views and Application Profiles belong to `0.4.4`; final SDK/DX and
performance hardening belong to `0.4.5`. No second extension runtime or session
store is permitted.

## Detailed Acceptance

### `ESS-001`: Decisions

- Accept five scopes: global, workspace, session, branch, ephemeral invocation.
- Global/workspace KV uses the extension state store; session/branch state and
  facts use generic SessionEvents.
- No cross-store atomic transaction exists between global/workspace and session.
- Freeze state/fact schema compatibility, package update, migration, and rollback.
- Dependency permissions never transfer; cross-extension calls use admission.
- Accept `ADR-010` correlation identities, channel separation, phase/error/trap
  taxonomy, redaction, visibility, and rate/size/retention limits before state,
  job, and update implementations embed diagnostics.
- Define package activation as a durable coordinated state machine. Atomicity is
  per store; cross-store correctness comes from phase records, idempotence,
  generation fencing, an admission barrier, and recovery.

### `ESS-002`: Global And Workspace State

- Namespace by extension and scope; enforce encoded-byte/key/mutation quotas.
- Atomic per-scope writes use schema versions and compare/transaction semantics.
- Migration writes a candidate namespace, validates it, and atomically switches
  the active version pointer; failure leaves the old pointer active.
- Migration is bounded pure-data work with no model/network/shell/secret/extension
  or arbitrary operation access.
- Missing/disabled extensions do not block inspection/export/deletion.

### `ESS-003`: Session, Branch, And Facts

- Generic `ExtensionStateMutation` SessionEvents rebuild opaque session/branch KV.
- Branch state inherits ancestor-visible mutations and diverges copy-on-write;
  deletes are tombstones.
- Session state is shared across branches.
- `ExtensionFact` is immutable historical evidence with extension/schema/event/
  lineage/payload envelope.
- Core replays/exports opaque state/facts without loading extension code.
- Core facts, session/branch state, ExtensionFacts, outbox, and cursor may commit
  atomically through SessionCoordinator.
- Session migration appends new-version mutations/markers and never rewrites
  history.

### `ESS-004`: Executable Services

- Commands become admitted async extension operations with typed input/output,
  progress, cancellation, resource claims, deadlines, errors, and durability.
- Tools support structured content/details/artifacts with output bounds.
- Hooks split into observation and closed point-specific interception patches.
- Ordering, conflict, timeout, cancellation, and fail-open/fail-closed policy are
  deterministic.
- Hooks never receive mutable operation contexts or raw services.
- Durable extension writes join canonical session transaction/outbox.

### `ESS-005`: Subscriptions And Background Jobs

- Instance grants register filters/job definitions; every callback/tick is a
  child operation with a lease.
- No raw EventHub receiver, detached task, or forever-running operation is exposed.
- Job concurrency is one of `SkipIfRunning`, `QueueOne`, or `ReplaceRunning`;
  unbounded queues are forbidden.
- Revoke/update/session close/shutdown cancels callbacks/jobs deterministically.
- Buffers, frequency, deadlines, progress, and output are bounded.

### `ESS-006`: Coordinated Update, Activation, And Recovery

```text
quarantine -> validate -> resolve -> review grants
  -> prepare candidate artifact and per-store candidate state
  -> validate candidate while externally non-admissible
  -> close old generation admission -> drain/cancel old generation
  -> commit durable activation record/pointer
  -> open candidate generation admission -> retire old artifact/state later
```

- Candidate code cannot receive externally visible callbacks, ticks, commands,
  tools, hooks, or view/data requests before the activation record commits.
- Old instances stop new admission and drain/cancel under a deadline before the
  new generation opens admission. Generation checks reject late old results.
- Artifact registry, global/workspace state, and each SessionEvent store commit
  atomically only within their own boundaries; no cross-store atomic transaction
  is claimed.
- Global/workspace migration uses candidate namespaces and version pointers.
  Session/branch migration appends idempotent versioned mutation/marker events;
  candidate-generation events remain inert until the durable activation record
  selects that generation, and committed history is never erased to simulate
  rollback.
- The update protocol declares whether all targeted session stores must prepare
  before activation or whether migration is lazy. Lazy migration makes an
  unmigrated scope explicitly unavailable/migrating until its idempotent forward
  migration completes; it never serves old state to new code silently.
- Failure before externally visible activation rolls back/discards prepared
  candidates, marks any append-only candidate facts inert/abandoned, reopens old
  admission, and retains the old generation. Failure after activation uses
  recorded forward recovery; rollback is allowed only through another explicit
  generation transition whose data compatibility is proven.
- Recovery handles process death at each phase and converges dependency lock,
  grants, per-store state versions, generation, and active pointer without
  admitting two generations.

### `ESS-007`: Observability And Conformance

- Implement the `ADR-010` contract accepted in `ESS-001`; later implementation
  cannot invent incompatible correlation IDs or diagnostic phases.
- Correlate operation/root/parent, session/client, extension/instance/contribution,
  request, grant generation, and contract versions.
- Keep audit, ProductEvent, trace, extension log, diagnostic, and SessionEvent
  channels distinct.
- Enforce log/diagnostic rate/size/retention/visibility/redaction.
- Source maps improve developer traps without exposing secrets/host internals.
- Test missing extension, replay, fork, migration/downgrade, quota, revoke,
  background cancellation, update crash, rollback, and durable outbox recovery.
- Capture and enforce provisional no-major-regression limits for state replay,
  migration, update recovery, jobs/subscriptions, diagnostics, and pressure.
