# Architecture And Extension Platform Roadmap For The 0.4.x Train

## Status

This document is the cross-version roadmap for the `0.4.x` development train.
It replaces the former single-release interpretation of
`0.4.0-architecture-extension-platform-plan.md`. Each linked version plan is an
independent release contract with its own tasks, debt closure, version metadata,
changelogs, validation, and completion record.

Current status: **`0.4.0` and `0.4.1` are complete; `0.4.2` is in progress with
Extension scope reduced to a minimum viable framework. Later Extension platform
tasks in `0.4.2` through `0.4.5` are explicitly Skipped as recorded below**.

The normative document split is complete. This roadmap and its plans remain
subordinate to the contracts indexed by `docs/architecture.md` and accepted ADRs.

The workspace begins this train at `0.3.1`. Releases advance strictly in order:

```text
0.4.0 -> 0.4.1 -> 0.4.2 -> 0.4.3 -> 0.4.4 -> 0.4.5
```

### Train-Entry Gate

`0.4.0` implementation cannot begin merely because workspace metadata says
`0.3.1`. The `0.3.1` release must first have a dated completion record, a stable
source commit/tag, frozen public-API and protocol evidence, current changelogs,
and passing release validation. Runtime, protocol, or lifecycle work present
after that baseline must be assigned explicitly to either the `0.3.1` closure or
`0.4.0`; it cannot remain an unversioned input to this train.

Gate evidence: annotated tag `v0.3.1` identifies source commit `870d4bb`; the
dated changelog record is `180f219`; formatting, strict Clippy, full workspace
tests, public API/protocol evidence, and the offline tmux TUI smoke suite pass.

## Release-Train Policy

### Extension Scope Freeze (2026-07-19)

The Extension platform currently requires only a minimum viable framework:
versioned Manifest/WIT/schema candidates, offline TypeScript componentization,
immutable package quarantine/install, grants and operation leases, explicit
core/extension handler references, and one real isolated Wasm Component
invocation through a lease-backed Host API. This framework is infrastructure,
not a promise of end-user contribution parity or a complete extension product.

After that minimal `EKR-004` vertical slice, later Extension productization is
out of scope for the current `0.4.x` train and is marked **Skipped**, not
Completed. This includes contribution parity, full runtime/parity/performance
baselines, extension state/facts/services/jobs, extension-driven
Workbench/Application Profiles, and Extension SDK/CLI/DX hardening. Legacy
cleanup is not skipped: `0.4.2` must still delete Lua/native extension paths and
obsolete trait-object contribution providers so the minimum framework does not
add a second runtime beside dead infrastructure. Restarting any skipped
capability requires a new reviewed version plan.

The same convergence rule applies to the generic Flow implementation replaced
by typed runners in `0.4.1`: `0.4.2` removes obsolete production/compatibility
Flow scaffolding rather than retaining redundant concepts indefinitely.

The train follows a framework-first convergence rule:

1. repair lifecycle correctness before structural migration;
2. establish owners, descriptors, single-writer consistency, and publication
   seams before changing workflows;
3. converge Agent/workflow execution before exposing it to extensions;
4. establish the minimum TypeScript/Wasm/WIT extension framework;
5. delete superseded Lua/native extension and generic Flow infrastructure;
6. skip later Extension productization until a new plan explicitly resumes it.

Every release must leave the workspace compiling, testable, documented, and free
of debt assigned to that release. A later plan cannot be used to waive an earlier
release gate. Intentional later capabilities are roadmap scope, not execution
debt in an earlier release.

Because the project is pre-product, unstable internal and extension contracts may
break across this train. Supported durable user data and public product protocols
still require explicit compatibility decisions and migration evidence.

## Stable Train-Wide Architecture Decisions

The following decisions are fixed for the train and may be changed only by a
superseding ADR with updated downstream plans.

### Runtime Ownership

The runtime uses a one-way collaborator graph:

```text
Adapter
  -> OperationSupervisor
  -> Typed Workflow
  -> SessionCoordinator
  -> committed outbox
  -> OutboxPublisher
  -> EventHub
  -> ClientProjectionCoordinator
  -> TUI / RPC / JSON / Embedding
```

- `RuntimeHost` is the composition/lifetime root, not a service locator.
- `OperationSupervisor` owns admission, execution identity, cancellation,
  permits, terminal authority, and `RecoveryPending` ownership.
- `SessionCoordinator` owns the per-session writer, transaction, committed read
  model, durable ProductEvent outbox, and snapshot cursor.
- `OutboxPublisher` delivers committed obligations but cannot decide outcomes.
- `EventHub` fans out events but cannot mutate sessions or invent durable facts.
- `ClientProjectionCoordinator` owns client projection and transient overlays but
  cannot repair durable state.
- provider, tool, extension, publication, and client work never executes inside
  the session-writer consistency point.

Terminal authority crosses the session consistency point through one typed
finalization protocol:

```text
OperationSupervisor
  -> immutable FinalizationDecision / CommitIntent
  -> SessionCoordinator validates and commits facts + terminal outbox
  -> Committed | DefinitelyFailed | InDoubt(recovery_id)
  -> OperationSupervisor resolves terminal or RecoveryPending
```

`SessionCoordinator` may validate and persist a decision but cannot reinterpret
the business outcome. `OperationSupervisor` cannot publish durable success before
commit. An in-doubt result carries durable recovery evidence and is a non-terminal
status/receipt, not a speculative terminal outcome.

### Capability Lifetime

The extension security model has three levels:

```text
GrantRecord
  -> revocable ExtensionInstanceGrant
  -> frozen OperationCapabilityLease
```

Activation, contribution registration, and subscription definitions use the
instance grant. Every privileged tool, command, hook, view refresh, data fetch,
event callback, or background tick is an admitted operation with a bounded lease.
No extension receives ambient product authority, a raw receiver, or a detached
background future. Dependency permissions never transfer implicitly.

### Wasm Invocation

- TypeScript is the sole third-party authoring language.
- Wasm Component is the sole installed third-party runtime artifact.
- WIT plus language-neutral schemas are the sole stable extension ABI.
- compiled immutable components may be cached globally;
- each operation receives an isolated Wasm Store/Instance and resource table;
- mutable guest memory is not durable state and is never shared across
  workspaces;
- invocation uses async host bindings, operation cancellation, Tokio deadlines,
  epoch interruption, deterministic fuel, host-call allocation limits, and
  memory/output quotas;
- cancelled, trapped, out-of-fuel, or over-limit instances are destroyed and
  never reused;
- installed extensions run without Node.js or build tooling.

### Extension State And Facts

The state scopes are:

```text
global
workspace
session
branch
ephemeral invocation
```

Global/workspace key-value state uses an extension state store. Session/branch
state mutations are generic SessionEvents through `SessionCoordinator`, so no
second durable session source exists. `ExtensionFact` is immutable historical
session evidence, not mutable key-value state.

Core SessionEvents, session/branch extension state, `ExtensionFact`, committed
ProductEvent outbox records, and snapshot cursor advancement may share one
session transaction. Global/workspace state never participates in that atomic
transaction. Migrations are bounded pure-data transformations and cannot invoke
model, network, shell, secrets, other extensions, or arbitrary operations.

Package activation across the artifact registry, global/workspace state store,
and SessionEvent stores is a durable coordinated state machine, not a fictional
cross-store transaction. Each store changes atomically within its own boundary;
candidate work is not externally admissible before the activation barrier;
session migrations append idempotent versioned facts; crash recovery converges a
recorded phase. Rollback means pre-activation rollback where possible and
explicit forward recovery after externally visible activation.

### Workbench Protocol

Workbench is a host-rendered semantic retained tree:

```text
ViewSnapshot(view_instance_id, revision, root)
ViewPatch(view_instance_id, base_revision, typed operations)
```

Nodes have stable IDs. The host validates revisions, references, depth, size,
rate, paging, and action bindings. A revision gap causes `ViewResyncRequired` and
a fresh snapshot; stale patches are not merged. Focus, selection, scroll,
viewport, and transient form input are client-local. Shared data invalidation
causes each client to refresh its own view instance. Lists, trees, tables, and
logs use paging/lazy loading/virtualization and bounded buffers.

Workbench is not a virtual DOM, CSS engine, raw TUI widget API, HTML/WebView, or
arbitrary JavaScript UI runtime.

## Cross-Version Gates

- `0.4.1` cannot begin from a runtime that fails the `0.4.0` lifecycle matrix.
- `0.4.2` cannot expose extension execution through unresolved Agent/workflow
  cancellation or message identity semantics.
- `0.4.2` exposes only the minimum Wasm framework vertical and does not claim
  contribution parity or publish a complete extension product, but it must
  remove Lua/native extension paths and obsolete generic Flow scaffolding.
- Skipped `0.4.3` through `0.4.5` Extension work cannot be treated as completed
  evidence or resumed without a new reviewed plan.

Before the train begins, the train-entry gate above must pass. Each release also
enforces a provisional, evidence-based no-major-regression guard for every path
it introduces. `0.4.5` freezes the final `ADR-009` budgets; it is not the first
release that may reject an architectural performance regression.

The phrase "no compatibility shim" in a release gate means no unresolved,
unowned, temporary forwarding or alternate-runtime shim. Owned versioned durable
decoders, schema migrations, and negotiated public-protocol compatibility remain
allowed when their support window and tests are explicit.

Every version runs at least:

```bash
cargo fmt --all --check
cargo clippy --workspace --all-targets --all-features -- -D warnings
cargo test --workspace --all-features
scripts/tui-smoke.sh
```

Additional owning-crate, protocol, conformance, security, performance, and
generated-artifact checks are listed in the corresponding version plan. Ordinary
and smoke tests remain offline.

## Train Completion

The architecture and extension platform initiative is complete only after
`0.4.5` passes and records:

- all six version completion records;
- accepted `ADR-001` through `ADR-012` and supersession history;
- no open train debt;
- one runtime identity/terminal/recovery model;
- one TypeScript/Wasm/WIT third-party extension path;
- no Lua or Rust-as-plugin runtime/ABI;
- durable state/fact and package update/rollback evidence;
- portable Workbench/Application Profile conformance;
- TypeScript SDK/tooling, security, performance, protocol, version, and changelog
  evidence.
