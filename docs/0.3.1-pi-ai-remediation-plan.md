# `pi-ai` 0.3.1 Reliability And Capability Remediation Plan

## Status

This document is the version advancement plan for the workspace `0.3.1`
release. It corrects the reliability, protocol, and capability gaps identified
in the `pi-ai` review performed on 2026-07-17 and refreshes that review against
the released `0.3.0` source baseline.

Current status: **completed on 2026-07-17; release gates passed**.

Target workspace version: **`0.3.1`**.

The version is complete only when every task and execution-debt entry in this
document is closed, the workspace and crate changelogs describe the resulting
behavior, and the release gates pass. A partial wave may be committed during
development, but it must not be presented as the completed `0.3.1` release.

This plan is subordinate to the normative ownership and dependency contracts in
[`architecture.md`](architecture.md). It does not authorize provider policy,
session behavior, product events, CLI configuration, RPC, or TUI concerns to
move into `pi-ai`.

## Version Scope

`0.3.1` is a focused provider-runtime reliability release. It may intentionally
correct public surfaces that advertise behavior with no production execution
path, provided the change has migration guidance and public API coverage. It
does not reopen the completed `0.3.0` product-runtime or TUI plan.

The release advances these artifacts together:

- tracked workspace package version metadata;
- root `CHANGELOG.md` and `crates/pi-ai/CHANGELOG.md`;
- this task ledger and its completion record;
- public API tests and snapshots for intentional facade changes;
- downstream provider-consumer tests in `pi-agent-core` and
  `pi-coding-agent`.

The repository intentionally ignores `Cargo.lock`. When a local lockfile is
generated for validation, every workspace package must resolve to `0.3.1`, but
the ignored file is not a release artifact.

## Starting `0.3.0` Baseline

The remediation does not start from an empty implementation. Existing behavior
must be retained and extended rather than duplicated:

- Anthropic detects EOF without `message_stop`, but the shared SSE processor
  still wraps the resulting `StopReason::Error` message in `Done`.
- Bedrock rejects incomplete binary event-stream frames, but does not require an
  observed `messageStop` before emitting `Done` when the decoder buffer is empty.
- OpenAI Completions and Responses use the shared HTTP retry/hook helper;
  Anthropic, Google, Mistral, Azure, Bedrock, and Codex still own divergent send
  and body-wait paths.
- The SSE parser has basic chunk, comment, and CRLF coverage, but joins multiple
  `data:` fields without the required newline and manufactures a trailing event
  from incomplete buffered bytes.
- OpenAI Completions consumes a subset of compatibility metadata through typed
  resolution, and Mistral maps `session_id` to affinity; other retained fields
  remain incomplete, inconsistent, or without an explicit disposition.
- Bedrock supports explicit credentials, session tokens, bearer tokens, and a
  hand-written SigV4 path, but an advertised `AWS_PROFILE` cannot execute a
  standard AWS credential-provider chain.
- Image-generation DTOs and a private OpenRouter mapper exist, but no production
  image provider/client/registry path owns them.
- Codex WebSocket URL/frame helpers exist, but production execution is SSE-only.
- The generated catalog still contains a negative dynamic-price sentinel and
  records whose `maxTokens` exceeds `contextWindow`.
- Existing provider fixtures and transport contract tests are useful owner
  tests, but they are not a uniform cross-provider behavioral matrix.

## Objective

Make `pi-ai` a truthful, cancellation-aware, and provider-neutral model runtime
whose public capability surface matches its production behavior.

Completion requires all of the following:

- a truncated or malformed provider stream can never be reported as success;
- cancellation wakes blocked request and body-stream waits promptly;
- timeout, retry, hooks, and transport options have defined support semantics;
- provider stream parsers handle terminal, failure, and unknown events safely;
- model compatibility metadata is either enforced or removed;
- advertised authentication and image/transport capabilities are executable or
  removed/rejected with typed migration diagnostics;
- catalog and cost data cannot produce invalid runtime values;
- every built-in provider is covered by the same contract-oriented test matrix.

## Non-Goals

- Adding coding-agent retry, admission, or user-interaction policy to `pi-ai`.
- Adding live-provider tests to the default offline test suite.
- Persisting `StreamOptions`, credentials, provider clients, or transport state.
- Moving model selection, profile resolution, or provider configuration
  precedence out of `pi-coding-agent`.
- Preserving inert fields solely because they existed in the `0.3.0` public API.

## Remediation Principles

1. Terminal state is explicit. EOF is transport state, not provider success.
2. Cancellation is part of every async wait, not a condition checked only
   between waits.
3. A public option must be supported, rejected with a typed diagnostic, or
   removed. Silent ignore is not acceptable.
4. Provider-specific wire events remain private and are converted into the
   closed provider-neutral event contract.
5. Unknown non-terminal wire events should be forward-compatible when they can
   be ignored safely; unknown terminal or failure states must fail closed.
6. Retries are allowed only before any response event has been exposed to the
   caller. Mid-stream replay requires a separate resumability contract and is
   outside this plan.
7. Secrets must not appear in `Debug`, serialized snapshots, diagnostics, test
   failures, or protocol output.
8. Generated catalog metadata is input data and must be validated before it is
   treated as a runtime contract.

## Execution Order

### Wave 1: Correctness Contracts

Implement `PAIR-001`, `PAIR-002`, `PAIR-005`, `PAIR-006`, and `PAIR-011`.
These tasks establish trustworthy terminal outcomes, responsive cancellation,
wire robustness, and valid catalog data without depending on a transport
refactor.

### Wave 2: Transport Convergence

Implement `PAIR-003` and `PAIR-004`. Move every HTTP/SSE built-in provider onto
one transport execution contract while keeping request conversion and stream
parsing provider-owned.

### Wave 3: Capability Truthfulness

Implement `PAIR-007`, `PAIR-008`, `PAIR-009`, and `PAIR-010`. Each optional
capability must end in one of two states: production-backed and tested, or
removed from the supported facade with migration notes.

### Wave 4: Full Validation

Implement `PAIR-012`, run downstream regression suites, update public API
snapshots when required, and close every acceptance criterion in this plan.

## `PAIR-001`: Typed Terminal Stream Outcomes

### Problem

The shared SSE processor currently turns body EOF into `Done`. Provider parsers
cannot distinguish a valid provider terminal event from a cleanly truncated
connection. `complete()` accepts every `Done`, including messages whose
`stop_reason` is `Error`.

### Required Design

- Introduce an internal terminal decision returned by each stream handler, for
  example `Continue`, `ProviderDone`, `ProviderError`, and `ProtocolError`.
- Track whether the provider's required terminal event was observed.
- Treat EOF before a required terminal event as `AssistantMessageEvent::Error`.
- Reserve `Done` for successful terminal reasons: `Stop`, `Length`, and
  `ToolUse`.
- Treat cancellation as `Error { reason: Aborted }`, unless the public event
  model is intentionally expanded with a separate aborted terminal variant.
- Make `complete()` reject `Done` messages carrying `Error` or `Aborted` for
  defensive compatibility with custom providers.
- Define the terminal requirement per built-in protocol:
  - Anthropic: `message_stop`;
  - OpenAI Completions: `[DONE]` after a usable finish reason;
  - OpenAI Responses/Azure/Codex SSE: completed, failed, or incomplete response;
  - Google: a candidate finish reason or an explicit documented terminal frame;
  - Mistral: `[DONE]` or an equivalent documented terminal marker;
  - Bedrock: `messageStop` and no incomplete event-stream frame.

### Tests

- Add one successful terminal fixture and one cleanly truncated fixture for
  every built-in streaming protocol.
- Add EOF-before-first-event, EOF-mid-text, EOF-mid-thinking, and
  EOF-mid-tool-arguments cases.
- Prove `complete()` returns `Err` for every error/aborted terminal shape.
- Prove exactly one terminal event is emitted.
- Prove no content end event is emitted after a terminal error unless the
  provider-neutral ordering contract explicitly requires it.

### Acceptance Criteria

- No built-in parser can produce `Done` without observing its valid provider
  terminal condition.
- A truncated fixture fails deterministically with provider/API/model context.
- Downstream agent tests cannot commit a truncated provider message as a
  successful assistant turn.

## `PAIR-002`: Cancellation-Aware Async Waits

### Problem

Cancellation is checked before awaiting the next body item. A stalled request or
body stream therefore cannot be interrupted until the network produces another
result.

### Required Design

- Wrap request-send waits in `tokio::select!` against the cancellation token.
- Wrap every body/SSE/event-stream `next()` wait in the same cancellation select.
- Centralize the select logic in transport helpers rather than duplicating it in
  provider modules.
- Preserve cancellation reason and provider/API/model context in the emitted
  terminal event.
- Drop the request/response/body stream immediately after cancellation so the
  underlying connection can be released.
- Ensure payload and response hooks also have a cancellation boundary. A hook
  must not make operation cancellation wait forever.

### Tests

- Use pending futures and never-yielding streams; cancel after the future starts.
- Assert cancellation completes within a bounded test deadline.
- Cover cancellation while sending, waiting for headers, waiting for the first
  body chunk, waiting between chunks, waiting before retry, and running hooks.
- Cover all transport families: shared HTTP/SSE, Bedrock event stream, and any
  retained WebSocket path.

### Acceptance Criteria

- Cancellation wakes every blocked provider operation without network activity.
- Every cancellation path emits one `Aborted` terminal event and releases its
  body/request resources.
- No provider implements a private polling-only cancellation loop.

## `PAIR-003`: Timeout Scope And Enforcement

### Problem

`timeout_ms` currently covers only selected `request.send()` calls and generally
stops applying once response headers arrive.

### Required Design

- Decide and document the public meaning of `timeout_ms`.
- Recommended contract: one end-to-end deadline from provider invocation until a
  terminal provider event, including hooks, retries, and body streaming.
- Derive one deadline at operation start; retries consume the same remaining
  budget rather than resetting it.
- Distinguish timeout from cancellation and HTTP/network failure in
  `ProviderErrorKind`.
- Include timeout scope and elapsed limit in diagnostics without including
  secrets or payloads.
- Apply the same deadline contract to every built-in provider.

### Tests

- Timeout before headers, between stream chunks, during retry delay, and in a
  hook.
- Timeout after partial output must produce an error terminal, not `Done`.
- A cancellation racing a timeout must produce one deterministic terminal event.
- `timeout_ms = 0` behavior must be explicitly defined and tested.

### Acceptance Criteria

- All providers interpret `timeout_ms` identically.
- A stalled body cannot outlive the configured deadline.
- Timeout behavior is documented on `StreamOptions` and in the public API tests.

## `PAIR-004`: Unified Transport Policy And Option Support

### Problem

Only OpenAI Completions and Responses use the complete shared transport helper.
Other providers silently ignore some combination of retry, timeout, hooks, and
transport selection.

### Required Design

- Refactor the shared HTTP execution layer to accept:
  - an already-built request;
  - a serializable or byte payload for hook processing;
  - provider/API/model identity;
  - cancellation and deadline;
  - retry configuration;
  - a provider-owned body parser.
- Route Anthropic, Google, Mistral, Azure, Codex SSE, and eligible Bedrock HTTP
  setup through the common execution contract.
- Define a capability check for options that are not meaningful for a provider.
- Reject unsupported explicit options before network I/O with a typed provider
  error. Do not silently discard them.
- Invoke payload hooks on the actual payload sent on the wire.
- Invoke response hooks for every HTTP response, including retryable and final
  error responses, with a documented once-per-attempt or once-per-operation
  contract.
- Retry only before exposing a provider-neutral stream event. Never replay a
  partially emitted response.
- Preserve explicit header precedence while preventing overrides of headers that
  would invalidate Bedrock signatures or leak/replace credentials accidentally.

### Provider Option Matrix

Create a checked-in table in the owner tests covering these options:

| Option | Anthropic | OpenAI Completions | OpenAI Responses | Azure | Google | Mistral | Bedrock | Codex |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| `temperature` | Map/compat | Map | Map | Map | Map | Map | Map | Map/compat |
| `max_tokens` | Map | Map | Map | Map | Map | Map | Map | Map or reject |
| `thinking` | Map/compat | Map/compat | Map/compat | Map/compat | Map | Map | Map | Map |
| `tool_choice` | Map | Map | Map | Map | Map | Map | Map | Map or reject |
| `session_id` | Compat/header/cache | Compat/header/cache | Cache/header | Cache/header | Reject or map | Affinity | Reject | Header/cache |
| `timeout_ms` | Enforce | Enforce | Enforce | Enforce | Enforce | Enforce | Enforce | Enforce |
| retry fields | Enforce | Enforce | Enforce | Enforce | Enforce | Enforce | Enforce before body | Enforce |
| hooks | Enforce | Enforce | Enforce | Enforce | Enforce | Enforce | Enforce with signed-body rules | Enforce |
| `transport` | Validate | Validate | Validate | Validate | Validate | Validate | Validate | Select |

The exact Map/Reject decisions may change during implementation, but every cell
must have owner tests and must not remain implicit.

### Acceptance Criteria

- Every public `StreamOptions` field has documented semantics.
- Every built-in provider either implements an explicitly supplied option or
  rejects it before network I/O.
- All HTTP providers share one retry/cancellation/timeout/hook implementation.

## `PAIR-005`: OpenAI Responses Event Coverage

### Problem

The Responses parser uses a closed serde enum that rejects every unrecognized
event and does not model failed or incomplete response terminals.

### Required Design

- Refresh the event inventory against the provider protocol used by OpenAI,
  Azure OpenAI Responses, and Codex SSE.
- Add explicit terminal handling for completed, failed, incomplete, cancelled,
  and provider error events that can occur on the supported endpoints.
- Preserve structured provider error code/message/type in internal error data;
  expose a secret-free message through the existing provider-neutral contract.
- Add a safe unknown-event representation carrying only event type and raw JSON.
- Ignore unknown non-terminal bookkeeping events only when doing so cannot lose
  content, usage, tool arguments, or terminal state.
- Fail closed for unknown events that advertise terminal/error semantics.
- Track multiple concurrent output items by provider item/index rather than a
  single text slot and a single tool slot.
- Finalize tool arguments with strict JSON parsing at the provider terminal.
- Keep the parser shared by OpenAI Responses, Azure, and Codex only where their
  event contracts are proven compatible; add adapters when endpoint-specific
  differences exist.

### Tests

- Full event fixtures including bookkeeping and `*.done` events.
- Failed and incomplete response fixtures with structured errors.
- Unknown harmless event, unknown content-bearing event, and unknown terminal
  event cases.
- Interleaved/multiple text and function-call output items.
- Missing `response.created`, duplicate terminal, malformed final tool JSON, and
  completed-without-usage cases.

### Acceptance Criteria

- Supported production event sequences do not fail because of harmless extra
  events.
- Provider-declared failure/incomplete responses always become error terminals.
- Multiple tool calls and output items preserve IDs, ordering, and arguments.

## `PAIR-006`: SSE Parser Compliance

### Problem

Multi-line `data:` fields are concatenated without newlines, and EOF fallback can
emit raw buffered text including the `data:` prefix.

### Required Design

- Implement the SSE field-processing rules for `event`, `data`, `id`, comments,
  CRLF/LF/CR line endings, blank-line dispatch, and final EOF dispatch.
- Join multiple data lines with `\n`, removing the final inserted newline at
  dispatch time.
- Decode UTF-8 across arbitrary byte chunk boundaries without lossy corruption.
- Bound the maximum buffered line and event size; return a typed transport error
  when limits are exceeded.
- Do not reinterpret arbitrary trailing bytes as an event payload.
- Separate SSE parsing errors from provider JSON/wire errors.

### Tests

- Multi-line JSON/text data preserving newlines.
- CRLF, LF, and CR-only line endings.
- UTF-8 scalar split at every byte boundary.
- EOF with and without a final blank line.
- Comments, empty data, repeated event fields, unknown fields, `id`, and retry
  fields.
- Oversized line/event rejection and bounded-memory behavior.

### Acceptance Criteria

- The SSE parser passes a table-driven protocol suite independent of providers.
- Provider fixtures behave identically regardless of network chunk boundaries.
- EOF never manufactures a valid-looking provider event from an incomplete line.

## `PAIR-007`: Provider Compatibility Metadata

### Problem

The model catalog contains compatibility settings with inconsistent runtime
dispositions. OpenAI Completions already consumes a typed subset, and Mistral
maps session affinity, but Anthropic temperature/adaptive-thinking flags and
several OpenAI-compatible store, affinity, cache, routing, and tool-stream flags
remain inert or only partially enforced. The current public surface does not
distinguish runtime-enforced metadata from catalog-only documentation.

### Required Design

- Inventory every field in `ModelCompat`, `AnthropicMessagesCompat`,
  `OpenAICompletionsCompat`, and `OpenAIResponsesCompat`.
- Assign each field one disposition:
  - enforced by request conversion;
  - enforced by transport/header policy;
  - catalog-only documentation with no runtime behavior;
  - obsolete and removed.
- Add typed compatibility resolution helpers owned by the relevant provider
  family; do not scatter raw JSON/optional flag checks across modules.
- At minimum address:
  - unsupported temperature suppression/rejection;
  - forced adaptive thinking;
  - cache-control support on prompts, messages, and tools;
  - long cache-retention support;
  - session-affinity/session-ID headers;
  - strict tool schema support;
  - store support;
  - provider routing payloads;
  - provider-specific tool-stream formats.
- Remove public fields that have no supported behavior and no documented
  catalog-only purpose. Update migration notes and API snapshots for removals.

### Tests

- One positive and one negative model fixture for every retained flag.
- Generated-catalog audit proving every non-null compatibility field has a
  registered runtime disposition.
- Request snapshots proving unsupported fields are absent and required fields
  are present.

### Acceptance Criteria

- No non-null compatibility field is behaviorally inert without an explicit
  catalog-only designation.
- Models marked as not supporting a parameter never receive that parameter.
- Compatibility behavior is deterministic and provider-family owned.

## `PAIR-008`: Bedrock Credential Chain

### Problem

The authentication facade recognizes `AWS_PROFILE`, and public types expose a
Bedrock profile field, but the provider only accepts explicit access-key pairs or
a bearer token.

### `0.3.1` Disposition

Implement the standard AWS credential provider chain. `AWS_PROFILE` and
`bedrock_profile` remain supported only because `0.3.1` makes them executable.
Use a proven AWS credential/signing library rather than
  expanding the hand-written credential loader.

### Required Design

- Support environment credentials, session tokens, shared config/credentials
  profiles, and refreshable credentials used by the selected library.
- Keep credential loading behind `ProviderAuthResolver` or a provider-owned
  credential source; do not let provider modules read product configuration.
- Ensure selected region and credentials are frozen for one request attempt.
- Preserve explicit bearer-token behavior where supported.
- Redact access keys, session tokens, bearer tokens, signed authorization, and
  credential-source details that could expose account secrets.

### Tests

- Explicit access key/secret/session token.
- Named profile using isolated temporary AWS config files.
- Missing profile, partial credentials, expired/refresh failure, bearer token,
  and explicit-over-profile precedence.
- SigV4 request tests using an authoritative library/test vector.
- No tests may read the developer's real AWS configuration.

### Acceptance Criteria

- A credential source reported as available can execute the signing path.
- `AWS_PROFILE` is either fully operational or no longer advertised.
- No credential value appears in Debug output or failure messages.

## `PAIR-009`: Remove Or Defer The Image-Generation Execution Surface

### Problem

The stable facade exports image-generation DTOs and the private implementation
contains an OpenRouter request/response mapper. However, there is no production
image provider trait, client, authentication path, or provider registry path.
The facade therefore advertises a data shape without an executable capability.

### `0.3.1` Disposition

**Remove/defer** image generation from the stable facade until a production
owner and provider contract exist. No current downstream product requires a
standalone image-generation client for `0.3.1`; multimodal conversation images
remain part of the existing conversation content contract and are not removed.

### Requirements For Any Future Implementation

- Add a provider-neutral `ImagesProvider`/client contract separate from chat
  streaming unless one interface can represent both without optional-method
  ambiguity.
- Add authentication, timeout, retry, cancellation, hooks, error classification,
  and output-size limits.
- Validate MIME types and base64 payloads without logging image data.
- Provide at least one production provider implementation and offline transport
  contract tests.
- Define cost and usage semantics independently from text-token usage where the
  provider bills per image.

### `0.3.1` Requirements

- Remove the image category from the stable `api` facade.
- Keep internal DTO experiments private or behind an explicitly experimental
  feature.
- Update public API tests, snapshots, changelog, and migration guidance.

### Acceptance Criteria

- The supported public facade does not advertise DTO-only functionality.
- Any retained image execution path has the same cancellation, timeout, retry,
  redaction, and error guarantees as text providers.

## `PAIR-010`: Codex WebSocket Surface Removal And Rejection

### Problem

Codex WebSocket URL/frame helpers exist, and `StreamOptions.transport` is public,
but production execution always uses SSE and no WebSocket dependency exists.

### `0.3.1` Disposition

Do not add a production WebSocket stack solely to preserve speculative helpers.
Remove dead WebSocket URL/frame helpers and make explicit unsupported transport
selection fail before network I/O. Retain `transport="sse"` only if the field
has a documented compatibility purpose; otherwise remove the field with public
API migration guidance.

### Requirements For Any Future Implementation

- Use a proven async WebSocket library.
- Define authentication headers, connection deadline, idle/read deadline,
  cancellation, ping/pong, close-frame, frame-size, and reconnect behavior.
- Reuse the hardened Responses event parser through a frame-to-event adapter.
- Do not retry/reconnect after exposing partial provider-neutral output unless a
  response resume contract is implemented.
- Emit diagnostics when automatic fallback changes transport, without exposing
  credentials or payloads.

### Tests

- Prove explicit `transport=websocket` is rejected before request construction
  or network I/O and never silently uses SSE.
- Prove unknown transport values receive the same typed unsupported-option
  treatment.
- Prove omitted transport and any retained explicit `transport=sse` select only
  the supported SSE path.
- Remove tests that validate otherwise unowned WebSocket URL/frame helpers.

### Acceptance Criteria

- `transport` is either removed or documents SSE as its only accepted value and
  rejects every unsupported value before network I/O.
- No WebSocket-only helper remains public or test-only without an owning runtime
  path.

## `PAIR-011`: Model Catalog And Cost Validation

### Problem

The generated catalog accepts semantically invalid values, including negative
price sentinels and models whose maximum output exceeds their context window.
Unknown dynamic pricing can therefore produce negative calculated costs.

### Required Design

- Introduce catalog validation run by the generator and owner tests.
- Validate non-empty identity/API/provider fields, supported API registration,
  positive context/output limits, input capabilities, finite prices, and
  compatibility-type consistency.
- Represent unknown or dynamic price explicitly. Preferred options:
  - make cost rates optional; or
  - add a typed `Known`/`Unknown` price state.
- Do not use negative numeric sentinels in runtime calculations.
- Define whether `max_tokens > context_window` is invalid or whether the fields
  have different semantics; encode and test the decision.
- Keep Azure's runtime-resolved empty base URL valid only through an explicit
  provider-specific validation exception.
- Make catalog generation deterministic and fail generation on invalid records.

### Tests

- Full generated catalog validation.
- Unknown-price calculation must produce unknown/no cost, never a negative cost.
- Non-finite, negative, duplicate, unsupported API, empty base URL, and invalid
  limit fixtures.
- Ensure every catalog API appears in `builtin_provider_apis()` or is explicitly
  marked external-only with a supported registration path.

### Acceptance Criteria

- No supported model produces a negative or non-finite cost.
- Invalid generated records fail CI with model/provider-specific diagnostics.
- Catalog invariants are documented beside the generator.

## `PAIR-012`: Cross-Provider Contract Matrix And Release Gates

### Problem

Existing tests cover ideal fixtures and request serialization but do not enforce
uniform failure, cancellation, timeout, option, terminal, and redaction
semantics across all built-in providers.

### Required Design

- Add a provider contract test harness that can run the same behavioral cases
  against every built-in provider using local transports and deterministic
  fixtures.
- Keep provider-specific request/response fixtures in provider-owned tests while
  enforcing shared behavior through the common harness.
- Cover every API in `builtin_provider_apis()` and prove the built-in registration
  set matches the catalog's executable API set.

### Required Contract Cases

- missing credentials;
- malformed request serialization where applicable;
- network failure before headers;
- non-retryable HTTP status;
- retryable status with zero retries and with bounded retries;
- invalid and excessive `Retry-After`;
- timeout before headers and during body streaming;
- cancellation at every async phase;
- malformed wire event;
- unknown non-terminal event;
- provider failure terminal;
- clean truncation before terminal;
- successful text, thinking, tool call, usage, and cost mapping;
- malformed final tool arguments;
- hook success, mutation, failure, and cancellation;
- header precedence and secret redaction;
- exactly one terminal event and no events after terminal.

### Public API And Downstream Validation

- Update `crates/pi-ai/tests/public_api.rs` for every intentional facade change.
- Update API snapshots when public types or exports change.
- Run focused downstream suites for `pi-agent-core` provider consumption and
  `pi-coding-agent` prompt, compaction, branch-summary, print/JSON, and RPC paths.
- Confirm machine-readable stdout/JSONL never contains provider diagnostics with
  secret material.

### Acceptance Criteria

- Every built-in provider passes the common contract matrix.
- No provider-specific implementation silently weakens terminal, cancellation,
  timeout, retry, hook, or redaction guarantees.
- All retained public capabilities have at least one production execution test.

## Completion Record

The plan was executed as one `0.3.1` provider-runtime convergence wave on
2026-07-17. No execution-debt item was opened or deferred.

Selected implementation dispositions:

- `timeout_ms` is one end-to-end deadline. Cancellation wins a deadline race;
  both are applied to hooks, request sends, retry delays, response hooks, body
  reads, SSE decoding, and Bedrock event streams.
- All nine built-in APIs use the shared HTTP transport contract. Bedrock signs
  the hook-mutated payload for each request factory invocation and uses the AWS
  official `aws-config`/`aws-sigv4` 0.56.1 family for standard credentials and
  signing, with explicit credentials and bearer tokens retained.
- Compatibility fields consumed by conversion are `Request`; fields without a
  production execution path are explicitly `CatalogOnly` through
  `compatibility_field_disposition`. No field remains implicitly inert.
- Standalone image-generation DTOs are removed from the stable facade. Ordinary
  multimodal conversation images remain supported.
- Codex is SSE-only. Unsupported transport values fail with a typed diagnostic
  before request construction.
- Unknown model pricing is represented by `known = false`; transcript,
  product-event, and RPC aggregates preserve that state. RPC protocol is now
  `2.1` and ProductEvent protocol is now `2.2`, both additive minor updates.

## Secret Handling Audit

The implementation must include a dedicated audit of types containing secrets.

- Replace derived `Debug` on `StreamOptions`, `ProviderAuth`, and any credential
  carrier with redacted manual implementations, or wrap secrets in a redacting
  type.
- Reassess `Serialize` on `StreamOptions.api_key`. If serialization is required
  for a private request shape, separate the serializable non-secret options from
  runtime credential material.
- Ensure provider header maps redact `authorization`, `api-key`, `x-api-key`,
  cookies, AWS signature headers, and configured custom secret headers in all
  diagnostics.
- Add tests that serialize and debug-print credential-bearing values and assert
  the secret bytes are absent.

This audit is part of `PAIR-004` and `PAIR-008`; it is not optional cleanup.
