# Fullscreen TUI Visual Hierarchy And Interaction Polish Plan For 0.5.6

## Status And Scope

Target workspace version: **`0.5.6`**. Status: **Completed 2026-07-22**.
This is the completed successor to the 0.5.5 operation-tree pressure and
fault-hardening plan. The product vocabulary, responsive thresholds, semantic
style roles, marker ownership, and immutable 0.5.5 baseline source are frozen in
[`architecture/fullscreen-visual-contract-0.5.6.md`](architecture/fullscreen-visual-contract-0.5.6.md).

Package versions, Cargo.lock, binary output, changelogs, protocol/API snapshots,
and release scripts advanced together after `FVP-001` through `FVP-010` closed;
the complete 0.5.6 release gate and evidence close `FVP-011`.

This plan gives the fullscreen interactive TUI one coherent visual and spatial
language. It fixes incorrect overlay anchoring and missing separators, clarifies
conversation/context/composer/status hierarchy, makes responsive modes behave
like explicit pages or drawers, and applies theme/accessibility rules without
moving product semantics into `pi-tui`.

The plan is subordinate to the normative contracts in
[`architecture/principles.md`](architecture/principles.md),
[`architecture/dependency-rules.md`](architecture/dependency-rules.md),
[`architecture/runtime.md`](architecture/runtime.md), and
[`architecture/testing.md`](architecture/testing.md). Visual changes must
preserve terminal cleanup, cursor placement, Unicode display width, bounded
rendering, resize behavior, unrelated scrollback, input/focus ownership,
machine-protocol cleanliness, and child-agent isolation.

## Motivation And Current Defects

The 0.5.4 fullscreen shell is functionally structured but visually inconsistent.
The primary problems are spatial semantics rather than a lack of colors:

- slash-command suggestions are a composer-assistance surface, but the shared
  transient overlay policy anchors them at `BottomCenter` with a 72-column
  maximum, so wide terminals show the menu detached in the middle of the screen;
- wide layout reserves a one-row gap between Context and Tips but does not draw
  the separator, leaving the right rail visually unstructured;
- panel focus, composer prompt, list selection, autocomplete selection, and
  transcript selection all reuse `>` even though they represent different
  states;
- Context title and tabs form one crowded plain-text line, with only brackets
  identifying the active tab;
- Tips has a fixed four-row allocation but can generate more actions, then
  truncates the tail without preserving the most important current action;
- the composer, autocomplete surface, and status bar use competing full-width
  lines and ad hoc separators rather than one vertical hierarchy;
- the status bar reads like a raw diagnostic string, mixing `|` and `/` while
  dropping rightmost segments under width pressure;
- Context usage is rendered only as token text/percentage even though both
  current context tokens and context-window capacity are available; the status
  bar lacks an immediately readable bounded progress gauge;
- `session_label` falls back to the durable session ID and is included in the
  status bar, consuming scarce width with an implementation identity that is
  already available through `/session` and Context detail;
- the empty conversation repeats keybindings already available through Tips and
  `/help`;
- startup/welcome and slash-command notices use unindented `System` transcript
  rendering, while assistant Markdown adds a one-column indent and selectable
  blocks add a separate two-column gutter, so conversation content has no common
  left baseline;
- Markdown tables compute cell widths and padding correctly, then pass completed
  table rows through the global whitespace-collapsing word wrapper; padding is
  normalized and vertical separators can move to different visible columns;
- medium/narrow Context behaves partly like an overlay and partly like a page,
  leaving unclear focus, composer, and back-navigation semantics;
- plain/color-disabled rendering does not yet have a documented hierarchy of
  markers independent of theme color.

The current implementation already contains the necessary generic primitives:
`OverlayAnchor`, explicit margins/width, bounded `Frame` rectangles, focus rings,
theme tokens such as `accent`, `borderAccent`, `borderMuted`, `muted`, and
`selectedBg`, virtual-terminal tests, and tmux smoke captures. 0.5.6 should use
these owners instead of introducing a second rendering system.

## Desired Visual Direction

The fullscreen TUI should feel like a restrained coding interface rather than a
boxed dashboard:

- align assistance with the control that invoked it;
- use few, meaningful separators rather than borders around every block;
- reserve accent color for focus, selection, active work, and actionable state;
- render inactive structure, empty states, IDs, timestamps, and hints as muted;
- distinguish focus, selection, input prompt, and runtime status with different
  markers;
- preserve the same information hierarchy with color disabled;
- treat wide Context as a rail, medium Context as a drawer, and narrow Context
  as a page;
- keep the main conversation visually dominant;
- make empty space intentional and stable instead of filling it with repeated
  help text;
- keep child conversations, main conversations, and Context pages within one
  consistent navigation and header language.

The initial wide composition target is:

```text
▌ Conversation                                      │ Context  [Ops] Changes Agents Usage
                                                    │
Assistant                                           │ ◌ prompt
Here is the current result...                       │   running · 12s
                                                    │
Tool · read                                         │ ✓ read
src/main.rs                                         │   completed
                                                    │
                                                    ├─ Tips ─────────────────────────
                                                    │ ↑ / ↓       Select
                                                    │ Enter       Open detail
                                                    │ Ctrl+G      Toggle context

  /help       Show commands and keybindings
  /model      Select the active model
  /settings   Open settings
────────────────────────────────────────────────────────────────────────────────────────
> /
────────────────────────────────────────────────────────────────────────────────────────
 ● idle   ctx [███░░░] 28%   default   deepseek-v4-flash · xhigh                  .
```

This wireframe expresses relationships, not exact glyph or color commitments.
The implementation may refine it while preserving the acceptance rules below.

## Architectural Decisions

- `pi-coding-agent` owns product shell layout, panel labels, status composition,
  slash-command presentation, child-page semantics, and responsive page policy.
- `pi-tui` owns generic overlay anchors, frame/layout primitives, editor/input,
  Unicode width, terminal lifecycle, styling, and virtual-terminal rendering.
  Do not add agent, command, context, or product status types to `pi-tui`.
- Composer assistance and modal dialogs use distinct overlay policies. A single
  boolean such as `non_capturing` cannot remain the positioning authority for
  semantically different surfaces.
- Slash/file completion is anchored to the composer text origin. Dialogs may be
  centered; Context detail follows the responsive Context owner; authorization
  presentation follows its operation/page scope.
- `ShellLayout` explicitly owns every drawn structural region, including the
  Context/Tips divider. Render code must not infer structural rectangles from
  incidental neighboring coordinates.
- Theme application uses the existing resolved token vocabulary. Do not embed
  terminal-specific RGB values or introduce a second product palette.
- Plain rendering is a first-class contract. Color can reinforce hierarchy but
  cannot be the only signal for focus, selection, failure, or pending approval.
- Snapshot tests assert stable semantic layout and exact critical coordinates;
  they should not freeze incidental ANSI ordering or every blank cell.
- Performance remains viewport-first and differential. A visual redesign may
  not restore full-transcript formatting or terminal writes on unchanged frames.

## Non-Goals

- Do not change provider, agent, delegation, authorization, session, ProductEvent,
  or operation-runtime semantics.
- Do not fold the 0.5.5 runtime fault-hardening work into this visual version.
- Do not add an ASCII logo, animation framework, mouse-only controls, or a large
  dashboard header.
- Do not make all surfaces modal or wrap every surface in a box.
- Do not replace the existing theme format or break user-defined themes.
- Do not require truecolor; ANSI-256, basic color, and color-disabled terminals
  remain supported.
- Do not expose private layout/style state through the stable embedding facade.
- Do not make visual snapshots depend on a live provider, credentials, host font,
  or uncontrolled terminal emulator behavior.
- Do not redesign inline/non-fullscreen mode beyond changes needed to preserve
  shared component correctness.
- Do not reactivate Extension Workbench or placeholder web UI scope.

## Baseline Evidence

Before implementation, freeze the exact 0.5.5 source commit plus representative
captures and metrics. Source immutability is the exhaustive reconstruction
authority; do not spend release time producing every permutation of a known
defective appearance. The final evidence must cover at least 60x18, 80x24,
120x32, and 160x40 terminals in dark, light, ANSI-256/basic, and color-disabled
modes, including:

- empty main conversation;
- non-empty assistant/tool transcript;
- slash suggestions after `/` and filtered query;
- multiline composer with file completion;
- wide Context for every tab and focus state;
- medium Context drawer;
- narrow Context page;
- Tips under Conversation, Context, and Composer focus;
- idle, running, waiting-permission, failed, and new-output-below status;
- child conversation page and child authorization;
- settings/model/session/tree modal surfaces;
- Unicode/CJK/emoji, long paths/IDs/model names, and no-color output;
- resize across wide/medium/narrow thresholds while an overlay is open.

Record current layout facts alongside each baseline rather than treating the
current appearance as the target. In particular, the centered slash menu and
blank Context/Tips row are defects to be replaced, not golden output.

## Detailed Acceptance

### `FVP-001`: Baseline And Visual Contract

- Record the immutable 0.5.5 source baseline and representative captures under
  `target/perf-baseline/0.5.6-fullscreen-visual/baseline/`; retain the exhaustive
  final matrix under its `final/` directory and human-readable smoke captures
  under `target/tui-smoke/`.
- Define one layout vocabulary for conversation, context rail/drawer/page,
  composer, status, support overlay, modal overlay, divider, header, tabs, and
  child page.
- Define semantic style roles using existing theme tokens: focused border/header,
  inactive border/header, selected row, primary text, secondary text, muted
  metadata, running, waiting, success, warning, and failure.
- Define marker ownership so panel focus, row selection, composer input,
  transcript selection, and autocomplete selection do not all use `>`.
- Record responsive thresholds and behavior at threshold-minus-one, threshold,
  and threshold-plus-one widths.
- Add source/boundary guards preventing product style or layout vocabulary from
  moving into `pi-tui`.

### `FVP-002`: Overlay Placement By Role

- Replace the shared transient positioning helper with explicit roles or typed
  builders for at least composer assistance, non-capturing support prompts,
  modal dialogs, and any Context-owned detail surface.
- Slash-command and file-completion surfaces use `BottomLeft` semantics and
  align their first content column exactly with the composer text origin after
  the prompt prefix.
- On wide terminals the assistance surface is no wider than 72 columns unless
  a measured content rule justifies another bound; on narrow terminals it fits
  inside the viewport with symmetric/minimal margins.
- Assistance remains immediately above the actual composer height, including a
  scrolled multiline composer and resize while visible.
- Modal settings/model/session/tree surfaces keep deliberate modal placement
  and focus capture; changing slash alignment must not move them accidentally.
- Overlay stacking, z-order, capture/non-capture behavior, visibility, focus
  restoration, and hidden-overlay cleanup remain deterministic.
- Add exact-coordinate virtual-terminal tests for 60-, 80-, 120-, and 160-column
  slash menus, including filtered queries, Unicode descriptions, and resize.

### `FVP-003`: Shell Grid And Structural Dividers

- Make every structural row/column an explicit `ShellLayout` rectangle,
  including the Context/Tips divider and any header divider required by the
  chosen composition.
- Draw a visible horizontal separator between Context and Tips in wide mode.
  Its intersection with the conversation/context vertical separator uses an
  appropriate junction glyph and remains valid without color.
- Prefer a titled divider such as `├─ Tips ─` when it saves a redundant header
  row; do not render both a divider title and a second `Tips` title.
- Panel headers have stable padding and focus treatment. Content columns align
  beneath their headers, and no focus glyph shifts the body horizontally.
- Conversation stays visually dominant and receives all width not explicitly
  assigned to the side rail and divider.
- Zero-size, 1x1, tiny-height, and narrow-width layouts produce bounded lines
  without underflow, overflow, panic, or terminal scroll.
- All emitted lines have visible width at most their frame rectangle.

### `FVP-004`: Responsive Context Behavior

- Wide mode uses a persistent right rail with a vertical separator, Context
  body, Context/Tips divider, and compact Tips region.
- Medium mode uses a right-side drawer with an explicit left border/header and
  `Esc` close behavior. The drawer must look overlaid rather than like silently
  overwritten conversation content.
- Narrow mode uses a complete Context page replacing the conversation viewport.
  Composer visibility/focus is explicitly decided; the preferred target hides
  or disables the composer while Context owns the page.
- Context page/drawer navigation restores the previous conversation focus,
  selected block, and scroll position.
- Tabs fit by compact spacing, controlled abbreviation, or horizontal tab
  selection—not by splitting wide graphemes or silently overlapping labels.
- Context detail follows the owning Context presentation: rail detail, drawer,
  or page. It does not appear as an unrelated centered surface.
- Resize across responsive modes while Context/detail is open preserves the
  selected tab/item and produces a valid focus ring.

### `FVP-005`: Contextual Tips

- Replace fixed-order tail truncation with priority-based action selection.
  Current-region primary action and close/cancel behavior outrank global help.
- Tips content changes for Conversation, Context, Composer, child page, pending
  authorization, and modal state without showing actions unavailable in the
  current capability/focus state.
- Wide Tips uses the available rail width efficiently, aligning key labels and
  action descriptions where possible.
- The divider owns the `Tips`/`Shortcuts` label, leaving all allocated Tips body
  rows for actions.
- Medium/narrow modes move essential hints into the page header/status or `/help`
  rather than reserving a cramped rail that does not exist.
- User-remapped keybindings remain the displayed source; no hard-coded key text
  replaces the keybinding registry.
- Long key labels and no-color mode truncate or wrap deterministically without
  hiding the primary current action.

### `FVP-006`: Composer And Completion Polish

- Establish one vertical relationship: support suggestions, composer border,
  input content, composer/status divider, then status bar.
- Keep the composer prompt marker for the first logical/visual input line;
  continuation lines use aligned indentation instead of implying multiple
  prompts where practical within the editor contract.
- Focused and unfocused composer borders use existing active/muted tokens and
  remain distinguishable without color.
- Slash rows visually separate command label, arguments when useful, and a muted
  description. Selection uses a dedicated marker/background distinct from panel
  focus and transcript selection.
- File completion and slash completion share alignment and navigation behavior
  while preserving their different completion edits.
- Menus remain bounded under many commands, long descriptions, multiline input,
  Unicode, terminal resize, and explicit cancellation.
- Composer rendering continues to reserve the correct cursor cell and does not
  exceed `MAX_COMPOSER_HEIGHT` or corrupt overlay bottom margins.

### `FVP-007`: Semantic Theme Hierarchy

- Apply resolved theme tokens to focused/inactive panel headers and borders,
  active/inactive Context tabs, selected rows, Tips, empty states, status states,
  IDs/paths/timestamps, and secondary metadata.
- Do not hard-code RGB values. Built-in dark/light themes and user themes obtain
  the hierarchy through existing token resolution.
- Accent is reserved for focus, selection, active work, and actionable state;
  inactive structure uses border/muted tokens.
- Running, waiting-permission, failed, cancelled, and completed states are
  semantically distinct without painting entire large regions.
- Replace the plain `ctx used/window` status segment with an adaptive context
  gauge derived from the existing latest-turn `context_tokens` and model
  `context_window` projection. Wide mode shows a bounded bar plus percentage,
  medium mode uses a shorter bar, and narrow mode retains at least `ctx N%`.
- Reuse the same ratio/rounding owner in the Context Usage tab, where the wider
  rail/page may show a longer gauge together with exact used/window token values;
  status and Context must never disagree because of separate calculations.
- Clamp only the filled bar geometry to its capacity; preserve explicit text for
  over-capacity/invalid states instead of hiding them. A zero or unknown context
  window renders an unavailable/omitted gauge without division or panic.
- Provide a tested ASCII gauge for terminals where block glyphs are disabled or
  unreliable. Every glyph used by the default gauge must have stable one-column
  width under the project's width contract.
- Remove `session_label` from the fullscreen status composition entirely. A
  durable session ID must never appear there, and an explicit human session name
  remains discoverable through `/session`/Context rather than taking permanent
  status width.
- Reorder status priority to keep runtime state and context pressure visible,
  followed by active profile and model/thinking; cwd is lowest priority and
  abbreviates or disappears first.
- Color-disabled output preserves hierarchy with markers, brackets, case,
  spacing, and separators; it cannot collapse into indistinguishable text.
- ANSI sequences do not affect width calculations, cursor placement, clipping,
  overlay splicing, or differential frame equality.

### `FVP-008`: Transcript And Child-Page Convergence

- Define consistent role headers/markers for User, Assistant, Tool, Delegation,
  diagnostic/error, and system/compaction blocks without boxing every item.
- Establish one fixed transcript content origin. Welcome/startup content,
  slash-command output, System notices, assistant Markdown, tools, errors, and
  delegation rows reserve the same base gutter before any role-specific styling.
- Reserve selection-marker columns for every transcript row, including
  non-selectable System rows, so selecting a block does not change neighboring
  content alignment. A selected marker replaces reserved space rather than
  inserting a new prefix.
- Remove the assistant renderer's independent one-column indent once the common
  gutter owns alignment; avoid double indentation from item styling plus the
  selection gutter.
- Simplify the startup message and render it through the same System/slash
  content path. Slash-command notices, multiline `/session` or `/hotkeys`
  output, and normal chat content must share the same visible first-content
  column and continuation-line policy.
- Running/completed/failed/cancelled tool and delegation states use the same
  semantic status vocabulary as Context and the global status bar.
- Long tool arguments/results, IDs, paths, timing, and token/cost metadata use
  primary/secondary/muted hierarchy and retain existing disclosure behavior.
- The main delegation row remains status-only; visual polish cannot reintroduce
  child assistant/tool/authorization content into the root transcript.
- Child-page header uses the same focused page language as Conversation and
  Context, includes a concise target/status/short ID, and presents `Esc back`
  without crowding narrow widths.
- Main/child page switching restores selection and scroll, and child permission
  presentation remains scoped to the active child operation.
- Markdown, code blocks, tables, diff/tool output, and images remain owned by
  their existing renderers; product styling composes rather than forks them.

### `FVP-009`: Markdown Table Geometry

- Treat a completed table as an atomic, already-wrapped block. Table border and
  row lines must bypass the generic paragraph `split_whitespace()` wrapper so
  cell padding cannot be collapsed after column layout.
- Introduce one table-line emission helper or equivalent typed block marker;
  every top/header/separator/body/bottom line follows the same no-rewrap path.
- Make the column allocator exact: the sum of cell widths plus border overhead
  never exceeds the available rectangle. Minimum word widths are preferences,
  not permission to overflow a narrow viewport.
- Define a deterministic narrow fallback when there is insufficient room for at
  least one visible cell column per table column. Prefer a readable stacked or
  simplified representation over a clipped/misaligned box.
- Normalize tab characters inside cells to spaces before both measurement and
  rendering. Do not measure a tab as a fixed width while emitting a terminal tab
  whose tab-stop width depends on its absolute column.
- Honor Markdown left/center/right alignment using visible-column padding after
  ANSI styling and wrapping. Styled content, OSC hyperlinks, and reset sequences
  remain zero-width for geometry.
- Use grapheme-aware display widths for CJK, combining marks, emoji/ZWJ, inline
  code, links, and styled text. Document terminal-dependent ambiguous-width
  limitations and provide conservative ASCII/no-color evidence.
- Add geometry assertions that collect the visible-column positions of every
  `│`, `┬`, `┼`, and `┴`; corresponding boundaries must match across all table
  rows and borders.
- Cover natural fit, proportional shrink, extremely narrow width, unequal/missing
  cells, long unbroken words, multiline wrapping, tabs, ANSI styles, links,
  CJK, combining marks, and emoji in both color and no-color modes.

## Responsive Contract

Exact thresholds are frozen by `FVP-001`; behavior at each class is normative:

| Mode | Conversation | Context | Tips | Composer |
| --- | --- | --- | --- | --- |
| Wide | dominant left pane | persistent right rail | rail section with divider | full-width bottom owner |
| Medium | full work area beneath drawer | right drawer with left border | essential hints only | full-width, remains available unless modal policy blocks it |
| Narrow | page owner | full replacement page | header/status hints | hidden or disabled while Context page owns focus |
| Child | full conversation page | optional responsive Context by same rules | child-relevant actions | no parent composer input |

Resize preserves logical state—active page, Context tab/item, child ID, focus
restore target, editor text/cursor, transcript selection/scroll—and recomputes
only rectangles and bounded rendering.

## Marker And Style Contract

The final glyphs may be adjusted for portability, but their semantic ownership
must remain distinct:

| Meaning | Preferred visual | No-color fallback |
| --- | --- | --- |
| focused panel/page | accent header or `▌` | `▌`/explicit title marker |
| selected list row | selected background plus `›` | `›` |
| composer prompt | `>` | `>` |
| autocomplete selection | background plus `›` | `›` |
| transcript selection | left accent bar | `▌` |
| running | spinner/accent | spinner/text |
| waiting permission | warning token | `! waiting permission` |
| completed | compact success token | `✓ completed` |
| failed | compact error token | `× failed` |
| cancelled | muted/warning token | `– cancelled` |

If a terminal cannot reliably display a preferred glyph, the implementation
must use a tested ASCII fallback without changing alignment semantics.

## Completion Invariants

0.5.6 is complete only when all of the following are true:

- composer assistance is spatially attached to and aligned with the composer;
- modal dialogs and composer assistance no longer share an accidental placement
  policy;
- Context and Tips have an explicit visible structural divider in wide mode;
- focus, selection, prompt, autocomplete, and runtime status use distinct visual
  semantics with no-color fallbacks;
- status shows an adaptive, bounded context-usage gauge and never spends width
  on a durable session ID/session-label segment;
- welcome/startup, slash-command output, System notices, and chat transcript
  roles share a stable content gutter and continuation-line baseline;
- Markdown tables preserve cell padding as atomic prewrapped output, allocate
  exact widths, align borders by visible columns, and remain bounded for ANSI,
  tabs, CJK, combining marks, emoji, and narrow fallbacks;
- Context behaves as rail, drawer, or page according to the responsive contract;
- Tips always prefers the current primary action over lower-priority global help;
- composer, completion, status, panel headers/tabs, empty states, transcript,
  delegation rows, and child pages share one semantic theme hierarchy;
- all layouts remain bounded and Unicode-safe at normal, threshold-adjacent,
  tiny, and resized viewports;
- terminal cleanup, synchronized output, cursor placement, input capture, focus
  restore, scroll state, child isolation, and protocol cleanliness remain intact;
- virtual-terminal, snapshot, tmux smoke, no-color, dark/light, and steady-frame
  performance evidence covers the completed appearance;
- version, changelog, migration, API/protocol decisions, release evidence, and
  architecture current-state match the implementation;
- the final release gate passes from the complete tree and the committed worktree
  is clean.
