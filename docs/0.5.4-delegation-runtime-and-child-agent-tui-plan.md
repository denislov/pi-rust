# Delegation Runtime And Child-Agent TUI Plan For 0.5.4

## Status And Scope

Target workspace version: **`0.5.4`**. Status: **Completed 2026-07-21**.
Implementation started from the completed `0.5.3` release at commit
`d933140`. The workspace advanced to `0.5.4` only after this plan's
protocol/runtime evidence, tests, and release gates closed together.

This plan addresses the delegation defects found after the 0.5.3 fullscreen
runtime hardening release:

- `delegate_agent` and `delegate_team` return a provisional `requested` tool
  result to the parent LLM before authorization and child execution finish;
- the child final result is folded into product state but is not returned to the
  original parent tool call, so the parent LLM cannot continue with it;
- manual approval runs the child operation but does not complete the parent's
  suspended tool call or persist a completed folded delegation update;
- child ProductEvents share the parent's stream but the interactive bridge
  discards `operation_id`, `parent_operation_id`, and `root_operation_id`;
- child assistant/tool/authorization output consequently pollutes the main
  transcript and child authorization can appear as a main-agent permission
  request;
- the fullscreen adapter has a generic expandable delegation tool block, but no
  child-operation conversation page, navigation stack, or child transcript
  projection.

The plan is subordinate to the normative contracts in
[`architecture/principles.md`](architecture/principles.md),
[`architecture/dependency-rules.md`](architecture/dependency-rules.md),
[`architecture/runtime.md`](architecture/runtime.md), and
[`architecture/testing.md`](architecture/testing.md). It does not weaken
operation identity, capability, authorization, cancellation, persistence, or
adapter boundary rules in those documents.

## Architectural Problem

The current delegation path is split across two incompatible lifetimes:

```text
parent LLM tool call
  -> returns {status: requested}
  -> parent PromptTurn finishes
  -> product layer authorizes and runs child
  -> child final_text is emitted as a live event/folded update
  -> no parent tool continuation
```

The 0.5.4 path must instead preserve one logical tool invocation:

```text
parent LLM tool call
  -> typed delegation admission
  -> optional confirmation waiter
  -> child operation with parent/root lineage
  -> child final result or typed failure
  -> original tool call receives one final result
  -> parent LLM continues with that result
```

Authorization requests and delegation lifecycle events are control-plane/UI
facts. They are not provisional tool results for the parent model. A parent
model must see exactly one terminal delegation tool result, even when the child
was paused for confirmation or child-tool authorization.

## Architectural Decisions

- The delegation tool remains product-owned, but its execution callback becomes
  an awaitable typed handoff to the operation runtime. It cannot return a
  provisional `requested` string and rely on a post-turn side effect.
- Parent, child, and root operation identities remain immutable and are carried
  through admission, capability snapshots, ProductEvent envelopes, tool
  results, and UI projections.
- Confirmation-required delegation suspends the child request at a typed
  control boundary. Approval or rejection resolves the waiter; it does not
  start an unrelated operation that loses the original tool-call continuation.
- Child tool authorization is scoped to the child operation. The main page may
  show only a compact `waiting permission` state; authorization detail is
  rendered in the child page and is resolved against the child operation.
- Main and child conversations are separate projections. Main transcript keeps
  one compact delegation status item; child messages and tool output never
  enter that transcript.
- Enter/open switches the chat viewport to the selected child conversation;
  Escape/back returns to the previous conversation. This is a page transition,
  not an inline expansion of the main delegation item.
- Child conversation state must be retained for the lifetime of the interactive
  client and have a bounded replay/hydration representation for reconnect. A
  child operation may remain non-persistent as a session transcript only if its
  bounded UI operation log is durable/replayable enough to satisfy the client
  contract; otherwise the operation transcript becomes a typed persisted fact.
- The public RPC/ProductEvent/UI snapshot versions remain unchanged unless the
  new result or child-page state cannot be represented compatibly. Any required
  version change needs an explicit migration record and API snapshot diff.

## Non-Goals

- Do not add generic workflow graph nodes or a second event bus.
- Do not move product delegation policy into `pi-agent-core` or `pi-tui`.
- Do not expose repositories, EventHub internals, provider streams, or runtime
  dispatch authority through the stable embedding facade.
- Do not make the child transcript a second copy of the parent's durable
  session transcript without an ownership decision and bounded retention.
- Do not auto-approve child tool permissions merely because delegation itself is
  configured with `require_confirmation = never`.
- Do not make the main page render arbitrary child output as a collapsed/expanded
  tool-body shortcut.

## Baseline Evidence

The post-0.5.3 source audit records:

| Measure | Current behavior | Required 0.5.4 behavior |
| --- | --- | --- |
| Delegation tool result | provisional `requested` JSON | one terminal typed result |
| Parent continuation | ends before child execution | resumes after child result |
| Auto-approved delegation | child runs after parent turn | child runs inside awaited tool path |
| Manual approval | separate approval operation | resolves original waiter |
| Manual completion persistence | approval fact plus live completed event | durable completed folded update |
| ProductEvent routing | event body only; envelope discarded in bridge | operation-aware routing |
| Main transcript | child assistant/tool events included | summary/status only |
| Child conversation page | absent | explicit page/navigation state |
| Child authorization UI | can appear in main overlay | child-scoped page/operation state |
| Child prompt storage | non-persistent child session | bounded reconnectable child projection |

`operations/delegation/mod.rs`, `operations/prompt/mod.rs`,
`operations/delegation/execution.rs`, `operations/agent_invocation/runner.rs`,
`adapters/interactive/event_bridge.rs`, and
`adapters/interactive/transcript.rs`. These facts are the starting point for
the implementation; they are not the target contract.

## Progress Evidence

Completion evidence (2026-07-21):

- `UiProjection` now consumes ProductEvent operation lineage and routes events
  with a parent operation into bounded per-child queues. Root pending UI events
  no longer receive child assistant/tool/authorization events.
- `InteractiveRoot` retains at most 32 child conversations, 2,048 queued child
  UI events per conversation, and 1,024 rendered transcript items. Enter opens
  the selected child as a full conversation page, the child page is read-only
  apart from scoped controls/navigation, and Escape restores the main page.
- Prompt runtime construction installs a typed delegation executor recursively.
  Automatic delegation and interactive confirmation both remain inside the
  original tool future; interactive confirmation reuses the generation-checked
  tool-authorization waiter. Non-interactive confirmation returns terminal
  `rejected` instead of `requested`. The 0.5.3 pending-confirmation operation is
  retained only to reopen and resolve already-persisted legacy requests.
- `DelegationToolResult` provides the only new parent result statuses:
  `completed`, `failed`, `rejected`, and `cancelled`. Child tasks run on isolated
  Tokio tasks to bound recursive polling stack growth while cancellation aborts
  the child and returns one terminal result.
- Terminal delegation results update folded session state, including recovered
  legacy approval completion. Replay/export coverage proves the final state is
  durable. UI snapshot `2.2` adds bounded retained child ProductEvents so a
  completed or waiting child page can be reconstructed after reconnect.
- Operation, RPC, recursive-depth/cycle, child-authorization, persistence,
  lineage folding, bounded retention, fullscreen navigation, and main-transcript
  isolation regressions pass. `scripts/delegation-runtime-soak.sh` freezes the
  deterministic repeated schedule.
- The 20-iteration delegation and fullscreen soak schedules, offline tmux TUI
  smoke, strict workspace Clippy, all-feature workspace tests, Extension
  contract/SDK/real-Wasm gates, and checked-in public API manifest verification
  passed. Full command evidence is retained in `0.5.4-release-evidence.md`.
- Release testing also hardened the Wasmtime epoch clock: it runs on the
  blocking pool so Guest deadline/cancellation progress does not depend on a
  current-thread Tokio executor.

## Detailed Acceptance

### `DCR-001`: Typed Parent-Facing Result

- Define a product/runtime-owned result with explicit `status` values at least
  `completed`, `failed`, `rejected`, and `cancelled`.
- Include `child_operation_id`, target identity, and either `final_text` or a
  typed redacted error. Do not expose internal capability or repository data.
- Make the result serializable for agent-core tool result consumption without
  making product policy types depend on `pi-agent-core`.
- Add a single-result invariant: one accepted delegate tool call produces one
  terminal tool result, regardless of approval or child authorization retries.

### `DCR-002`: Awaited Runtime Handoff

- Replace `handle_delegation_request`'s provisional `requested` result with a
  typed request sent to the operation runtime.
- Validate target, policy, depth, lineage, capacity, and parent capability
  before admitting child execution.
- Keep the tool future pending while a confirmation decision or child runtime
  action is outstanding.
- Preserve cancellation and shutdown behavior: aborting the parent cancels the
  child and resolves the tool future exactly once.
- Do not run the child after the parent PromptTurn has already finalized.

### `DCR-003`: Confirmation And Resume

- Auto-approved policies execute through the same waiter path as interactive
  approval.
- Manual approval resolves the original delegation request by operation ID and
  tool-call ID; it must not create an unrelated orphan approval execution.
- Rejection resolves the original tool call with a terminal rejected result.
- Stale, duplicate, expired, and generation-mismatched approvals fail closed.
- Restart recovery must either restore a replayable waiting request or emit a
  typed non-terminal recovery state; it must never claim that a child completed
  when only approval was persisted.

### `DCR-004`: Parent Result Delivery

- The parent LLM receives the child result in the same logical tool-call
  continuation and can produce a follow-up assistant message using it.
- Child result delivery must preserve tool-call ID, operation association, and
  ordering relative to other parent tools.
- A child failure, cancellation, or denied authorization returns a typed result
  rather than silently ending the parent turn.
- Add tests proving no result is emitted twice when both live completion and
  terminal operation handling observe the child outcome.

### `DCR-005`: Durable Folded State

- Automatic, interactive, RPC, and recovered approvals all persist one final
  `DelegationFoldedUpdated` record.
- The folded record carries status, child operation ID where available, and a
  bounded/redacted summary suitable for the main status row.
- Replay reconstructs completed, failed, rejected, cancelled, and waiting
  states consistently.
- Live ProductEvents remain notifications; replayable session facts remain the
  durable source of delegation state.

### `DCR-006`: Child Authorization Isolation

- Every authorization request carries the child operation identity and lineage.
- Authorization resolution validates child operation ID and capability
  generation before waking the child tool.
- Parent LLM receives only the final delegation tool result; authorization
  required/approved/denied events are not injected as parent model messages.
- Main UI receives a compact child delegation state update, not the child
  authorization preview or tool body.

### `DCR-007`: Operation-Aware Event Routing

- `UiProjection` and `CodingEventBridge` consume the ProductEvent envelope,
  including operation, parent, and root IDs.
- Root conversation receives root events and delegation summary events only.
- Child events are routed to the matching child projection; unknown/stale child
  events are retained in a bounded orphan/recovery bucket or fail closed.
- Snapshot hydration and reconnect replay use the same routing rules as live
  events.
- Add a boundary guard preventing a child Message/Tool event from mutating the
  root transcript.

### `DCR-008`: Child Conversation Projection

- Introduce a bounded `ChildConversationProjection` keyed by child operation ID.
- Preserve assistant deltas, thinking, tool calls, tool results, errors, usage,
  and authorization status in child scope.
- Keep memory and replay bounds explicit; a child page must not clone the whole
  parent transcript or retain unlimited child output.
- Define behavior when a child completes before the user opens its page: the
  summary remains available and the child page opens from retained/replayable
  content.
- Define behavior for child cancellation, failure, gap recovery, session switch,
  and reconnect.

### `DCR-009`: Main/Child Fullscreen Navigation

- Main page renders one compact delegation row with target, task summary,
  status, elapsed time, and child operation ID/short label.
- Main page never renders child assistant text or tool output inline.
- Enter/open on a selected delegation row switches the entire chat viewport to
  the child conversation; it does not toggle a tool block body.
- Escape/back returns to the previous conversation and restores its scroll,
  selection, and input focus state.
- Child page has a clear breadcrumb/header and preserves normal transcript
  scrolling, selection, image, Unicode, and tool rendering behavior.
- Status transitions are visible without opening the child page:
  `queued`, `awaiting approval`, `waiting permission`, `running`, `completed`,
  `failed`, and `cancelled`.

### `DCR-010`: Child Permission Controls

- Child authorization overlay is scoped to the active child page and operation.
- If a child asks for permission while the main page is active, main shows only
  `waiting permission`; no child preview is appended to the main transcript.
- Opening the child page exposes the pending authorization and allows approve
  once, approve for child operation, or deny with reason.
- Approval/rejection returns to the child operation and updates both child page
  state and the compact main delegation row.
- Permission state survives reconnect or is represented as an explicit recovery
  state; it cannot silently disappear.

### `DCR-011`: Regression And Fault Matrix

Required focused coverage includes:

- parent tool call waits for automatic child completion and receives final text;
- parent tool call waits across manual approval and then receives final text;
- rejection, child failure, cancellation, stale approval, and duplicate approval;
- child tool authorization waits and resumes without parent transcript/model
  leakage;
- manual approval persists completed folded state and replay restores it;
- parent/child/root event envelope routing and out-of-order replay;
- child output never mutates root transcript content or root assistant deltas;
- main page status-only rendering and Enter/Escape page transitions;
- child page streaming, scroll, resize, tiny viewport, Unicode, images, and tool
  output;
- reconnect before child open, during child execution, after child completion,
  and after authorization request;
- bounded child projection memory and event queue pressure;
- shutdown/abort while parent tool waits and while child authorization is open;
- RPC/print/JSON adapters receive the same terminal delegation result contract.

### `DCR-012`: Version And Release Convergence

- Refresh `docs/architecture/current-state.md` with the completed delegation
  result, operation-aware projection, and child-page facts.
- Add a 0.5.4 migration guide describing result/projection behavior and any
  protocol or snapshot changes.
- Advance workspace packages, lockfiles, binary version, and changelogs to
  `0.5.4` only after `DCR-001` through `DCR-011` close.
- Review stable facade, RPC, ProductEvent, and UI snapshot diffs. Preserve
  versions when the wire contract remains compatible; otherwise add an explicit
  migration and API snapshot update.
- Run clean detached API snapshot generation, full workspace validation,
  architecture/Extension gates, TUI smoke, deterministic delegation soak, and
  `git diff --check`.
- Preserve all skipped Extension productization decisions; this plan does not
  reactivate contribution dispatch or Workbench work.

gap-recovery remain covered by the existing runtime/client and interactive
matrices invoked by the workspace and TUI gates.

## Completion Invariants

0.5.4 is complete only when all of the following are true:

- no parent-facing delegation tool result uses provisional `requested` as its
  terminal answer;
- every accepted delegation has exactly one terminal parent tool result;
- child final text is consumed by the parent LLM before the parent operation
  finalizes;
- manual and automatic approval share the same execution/resume semantics;
- all approval paths persist the correct folded delegation terminal state;
- child ProductEvents cannot mutate the root transcript projection;
- main fullscreen view is status-only for child runs;
- child chat opens as a separate page and returns with Esc/back;
- child authorization is operation-scoped and never becomes a parent model
  message or root transcript item;
- child projection retention, reconnect, cancellation, and memory bounds have
  deterministic evidence;
- public/API/protocol/version/changelog evidence matches the implementation;
- all release gates pass from a clean detached audit tree.
