## Program Execution Principles

### Preserve The Target Architecture

Implementation decisions must be made with the complete runtime and product
shape in view, not only the nearest failing test or adapter call site. Each
change should move ownership toward the contracts in `architecture.md` and the
task-level boundaries in this plan.

- Prefer one authoritative runtime path with multiple adapters over duplicated
  adapter-specific implementations.
- Preserve stable external behavior and protocol contracts intentionally, but
  do not preserve obsolete internal structure merely because a test encodes
  it.
- When a test protects a superseded implementation detail rather than an
  agreed compatibility contract, update or replace the test instead of adding
  redundant production code, compatibility facades, duplicate state, or
  permanent shims solely to make it pass.
- Compatibility code is justified only by a named user-visible, embedding, or
  negotiated-protocol contract with an explicit retirement decision.
- Prefer completing an owning runtime contract before projecting it into TUI,
  RPC, print, JSON, or embedding adapters.
- Shared product events, snapshots, operation controls, and services are the
  convergence points. Adapters must not become temporary alternative owners
  without an execution-debt record and a removal trigger.

Tests remain mandatory, but they validate the intended contract. Removing or
rewriting an obsolete test must be accompanied by replacement coverage for the
new contract when behavior remains supported. This principle must not be used
to weaken safety, persistence, cancellation, terminal-cleanup, public API, or
protocol guarantees.

### Execution Debt

Temporary divergence is allowed when it keeps independently testable work
moving, but it must be explicit execution debt rather than hidden unfinished
architecture. An execution-debt entry must record:

- a stable debt ID and the task or phase that introduced it;
- the target contract that is not yet fully reached;
- why immediate convergence would block useful delivery;
- the exact removal trigger or dependent phase;
- the tests or verification required to close it;
- its current owner and status.

Execution debt may postpone convergence between phases; it may not redefine
the final acceptance criteria. A task cannot be marked complete while debt
owned solely by that task remains open. The `0.3.0` plan cannot be declared
fully converged or release-ready while any debt entry is `Open`, `Deferred`,
or lacks a verified disposition. Debt is closed by removing the temporary
path, completing the target contract, and updating tests and documentation,
not by relabeling the temporary behavior as permanent.

#### Execution Debt Ledger

| Debt ID | Introduced by | Deferred convergence and reason | Removal trigger | Verification | Owner | Status |
| --- | --- | --- | --- | --- | --- | --- |
| None | - | No execution debt recorded during planning | - | - | - | Closed |

New debt entries must be added when the temporary decision is made, not during
the final cleanup pass. Closed entries remain in the ledger as a concise record
of how convergence was completed.

## `V03-001`: Model-Visible Delegation Target Inventory

### Summary

The runtime loads agent and team profiles and can enforce delegation policy,
but the model-visible `delegate_agent` and `delegate_team` tool definitions do
not identify the targets that the model may select. The model receives an
unconstrained string field and must guess a profile ID.

This is a context-projection defect, not a profile-loading or delegation-
execution defect.

### Current Behavior

The current path is:

```text
ProfileRegistry loads agent and team profiles
        |
        v
DelegationPolicy stores allowed_agents and allowed_teams
        |
        v
delegation_tools creates delegate_agent and delegate_team
        |
        v
the provider receives a generic string target field
        |
        v
the runtime validates the selected ID only after a tool call
```

For example, `delegate_agent.agent_id` is currently described only as a
configured profile ID. Its JSON schema has no `enum`, and the tool description
does not expose profile display names or capability descriptions.

The built-in default profile authorizes `explore`, `review`, and `check`, but
its system prompt is unset and the delegation tool schema does not identify
those profiles. External adapters can list profiles through interactive and
RPC commands, but those adapter capabilities are not available to the model.

### Existing Policy Semantics

The implementation must preserve these existing meanings:

- `allow_delegate_agent = false` means `delegate_agent` is not available.
- `allow_delegate_team = false` means `delegate_team` is not available.
- A non-empty `allowed_agents` or `allowed_teams` list is an allowlist.
- An empty allowlist means all loaded profiles of that target kind are allowed.
- Loading a custom profile does not add it to the built-in default profile's
  non-empty `allowed_agents` allowlist.
- Model-visible schema constraints do not replace operation capability checks,
  depth checks, confirmation policy, profile existence checks, cycle/lineage
  protection, or execution-time authorization.

### Required Design

Delegation tool construction must have access to the active `ProfileRegistry`
or to an equivalent immutable projection owned by the product runtime. For
each enabled delegation tool, it must derive an effective target inventory by
intersecting registry contents with the active profile's policy:

```text
effective targets = loaded targets of the requested kind
                    filtered by the policy allowlist when it is non-empty
```

The effective inventory must be projected into the provider-visible tool
definition:

1. `delegate_agent.agent_id.enum` contains the effective agent profile IDs.
2. `delegate_team.team_id.enum` contains the effective team profile IDs.
3. Each tool description identifies the available targets using profile ID,
   display name, and description when present.
4. Inventory ordering is deterministic so provider requests and tests are
   stable across filesystem and registry load order.
5. Agent and team delegation use the same filtering and presentation rules.
6. IDs present in a non-empty policy allowlist but absent from the registry are
   not advertised as callable targets. Existing configuration diagnostics or
   execution-time rejection remain responsible for reporting invalid IDs.
7. Runtime validation remains authoritative even when a provider validates the
   schema before emitting a tool call.

The registry-derived tool schema is the canonical model-visible inventory for
this task. Duplicating the same inventory in the system prompt is not required
unless provider compatibility evidence shows that schema projection alone is
insufficient. Adding model-callable `list_agents` or `describe_agent` tools is
also outside this task because the inventory is bounded and already available
when the provider request is built.

### Ownership And Boundaries

- `pi-coding-agent` owns profile policy, registry resolution, delegation tools,
  and provider-facing runtime assembly; the change stays in this crate.
- `pi-agent-core` continues to receive ordinary `AgentTool` definitions and
  must not gain product profile or delegation-policy types.
- `pi-ai` continues to map the supplied tool schema to provider requests and
  must not resolve product profiles.
- Interactive and RPC profile-listing behavior is not changed by this task.
- The schema describes the operation-local effective inventory, not every
  profile installed on disk.

### Expected Implementation Areas

The implementation is expected to touch the smallest necessary subset of:

- `profiles/` for a deterministic, kind-specific registry projection if an
  existing iterator is insufficient;
- `operations/delegation/` for inventory filtering and schema/description
  construction;
- runtime/admission assembly call sites that currently invoke
  `delegation_tools` with only profile ID and policy;
- focused product tests that inspect the actual provider request.

This list is guidance, not authorization for unrelated refactoring. The final
implementation should follow the current ownership paths discovered at that
time.

### Acceptance Criteria

`V03-001` is complete when all of the following hold:

- A prompt run under the built-in default profile sends a `delegate_agent`
  schema whose `agent_id.enum` contains exactly `check`, `explore`, and
  `review`, in deterministic order.
- The same tool definition exposes enough display-name and description context
  for the model to choose among those targets without guessing IDs.
- A custom profile with a non-empty agent allowlist advertises only loaded
  agents in that allowlist.
- A custom profile with an empty agent allowlist advertises all loaded agent
  profiles allowed by the existing semantics.
- Equivalent cases pass for `delegate_team` and `allowed_teams`.
- Disabled delegation kinds do not expose their tools.
- A stale/missing allowlisted profile ID is not included in the schema enum.
- A forged or provider-generated disallowed ID is still rejected by the
  existing runtime authorization path.
- No product profile types or registry dependencies are introduced into
  `pi-agent-core` or `pi-ai`.

### Test Plan

The change requires tests at both construction and request boundaries:

- focused unit tests for effective inventory filtering, empty-allowlist
  expansion, deterministic ordering, descriptions, and missing registry IDs;
- delegation tool tests that assert the generated JSON schema for agents and
  teams;
- a provider request regression test proving that the model actually receives
  the effective inventory, rather than only proving that a locally constructed
  tool can execute;
- retained execution tests proving that runtime policy rejects disallowed or
  invalid targets independently of schema constraints;
- focused `pi-coding-agent` checks during implementation, followed by the
  workspace validation required by the final change's blast radius.

### Non-Goals

- Changing the meaning of an empty delegation allowlist.
- Automatically authorizing every newly installed custom agent for the built-in
  default profile.
- Replacing runtime policy enforcement with JSON Schema validation.
- Adding a general model-facing profile discovery API.
- Changing interactive `/agents` or RPC `list_agent_profiles` contracts.
- Redesigning profile persistence, team execution, delegation confirmation, or
  operation capability snapshots.

### Planning Decisions

- The inventory source is the active registry filtered by the active
  delegation policy.
- Tool schemas, rather than default system-prompt text, are the primary
  projection surface.
- Agent and team inventories are addressed together to keep delegation
  semantics consistent.
- Provider-request visibility is part of the acceptance contract.
- Execution-time authorization remains mandatory defense in depth.

### Completion Record

Completed on 2026-07-16 without execution debt.

- The active profile now freezes a registry-derived, policy-filtered agent and
  team inventory into its operation-local runtime snapshot.
- `delegate_agent` and `delegate_team` schemas expose deterministic target
  enums, while tool descriptions include target IDs, display names, and
  descriptions.
- Empty allowlists expand over loaded profiles, missing allowlist IDs are not
  advertised, and delegation tools with no effective target are not projected
  to the provider.
- Runtime policy and child profile resolution remain authoritative for forged,
  stale, or otherwise invalid tool calls.
- Unit, provider-request, delegation execution, boundary, full
  `pi-coding-agent --all-features`, and strict Clippy validation passed.

## `V03-002`: Full-Screen TUI And Interactive Transcript Blocks

### Summary

Migrate the interactive adapter from a vertically appended line-oriented TUI
toward a full-screen application shell with bounded panels, independent
scrolling, responsive layout, keyboard focus, and mouse interaction.

The first product-facing result must keep the conversation and composer as the
primary workflow while allowing long thinking sections, tool arguments, and
tool results to be selected and individually collapsed, previewed, or expanded.

This is an application-shell and interaction-model change, not a redesign of
session facts, product events, provider streaming, or the non-interactive
adapters.

### Current Constraints

The current interactive root renders the transcript, editor, menus, and footer
as one sequential `Vec<String>`. The generic `Component` contract renders by
width and does not receive a rectangular viewport. Input supports keys, paste,
raw terminal sequences, and resize events, but has no typed mouse event or hit
testing. Terminal startup enables raw mode, bracketed paste, and keyboard
protocol negotiation, but does not own alternate-screen or mouse-capture
lifecycle.

Transcript rendering already has useful foundations that should be preserved:

- stable per-item metadata and revisions;
- cached rendering of individual transcript blocks;
- row-count metadata for scroll preservation;
- streaming updates for assistant, thinking, tool, delegation, error, and
  system blocks;
- Unicode-width, ANSI, Markdown, theme, and Kitty image handling.

The migration must extend these foundations rather than replace them without a
measured reason.

### Target Product Layout

The initial full-screen layout consists of four bounded regions:

```text
+--------------------------------------------------------------+
| session / profile / model / operation status                 |
+---------------------------------------+----------------------+
|                                       | Context              |
| Conversation                          |                      |
|                                       | operation / changes  |
| assistant / thinking / tool blocks    | agents / usage tabs  |
|                                       |                      |
+---------------------------------------+----------------------+
| composer                                                     |
+--------------------------------------------------------------+
| contextual key hints / diagnostics                           |
+--------------------------------------------------------------+
```

The conversation panel is the primary surface and owns transcript selection
and scrolling. The context panel is secondary and uses tabs rather than
showing many small panels simultaneously. The composer remains visible and
usable while output streams.

Responsive behavior must follow capability and available space:

- wide terminals show conversation and context side by side;
- narrower terminals hide the context panel behind a tab, command, or overlay;
- low-height terminals compress status and contextual hints before reducing
  the conversation to an unusable size;
- the minimum layout always retains a usable conversation viewport and
  composer;
- panel dimensions are stable within a frame and dynamic content must not
  shift unrelated controls.

Exact breakpoints are implementation details and should be selected through
virtual-terminal and smoke testing rather than encoded as release contracts.

### Interactive Transcript Model

Expansion is per transcript block or block section, not a single global tool-
output flag. Long content uses a three-state display model:

```text
Collapsed -> Preview -> Expanded
```

The view state is keyed by stable transcript item identity or an equivalent
stable call/message ID. Array indices are not sufficient because streaming,
replay, delegation updates, and future filtering can change presentation
order.

Recommended initial defaults:

| Block or section | While running | After completion |
| --- | --- | --- |
| Assistant answer | Stream expanded | Expanded |
| Thinking | Tail preview | Collapsed or preview |
| Tool arguments | Summary | Collapsed |
| Tool result | Tail preview | Preview |
| Tool error | Key error visible | Expanded |
| Bash output | Command and recent output | Preview with exit summary |
| Delegation | Target and live status | Preview with final summary |
| System notice | Single-line display | Not foldable |

Thinking is a foldable section inside an assistant block. Tool arguments and
tool results may be independently foldable when that distinction improves
readability. A collapsed header must still identify the block and communicate
useful state, such as target, command, exit status, error status, or hidden line
count.

Expansion state is adapter-local view state. It must not be persisted as a
durable `SessionEvent` or treated as a product fact. Session replay may restore
the transcript and then apply the default folding policy.

### Keyboard And Mouse Interaction

Every mouse action must have an equivalent keyboard path. The initial
interaction model should support:

- moving focus between visible panels;
- moving selection between transcript blocks;
- toggling the selected block or section between collapsed, preview, and
  expanded states;
- scrolling the focused or pointer-local panel without moving unrelated
  panels;
- opening and closing the context panel on narrow layouts;
- retaining a command for expand-all or collapse-all behavior without making
  it the primary interaction.

Suggested defaults include `Tab`/`Shift+Tab` for focus, arrow keys or `j`/`k`
for block selection, and `Enter` or `Space` for disclosure. Existing `Ctrl+O`
may be retained as a current-block toggle or migrated with an explicit
compatibility decision.

Mouse support must use typed events and frame-local hit regions. Clicking a
block selects it; clicking its header or disclosure control toggles it. Normal
text clicks must not unexpectedly collapse content. Mouse wheel input affects
the panel under the pointer. Unsupported or disabled mouse capture must leave
the complete keyboard workflow available.

### Generic TUI Foundations

`pi-tui` should gain the minimum reusable primitives needed by full-screen
applications:

- `Inline` and `Fullscreen` terminal modes;
- alternate-screen and mouse-capture lifecycle with reliable cleanup;
- rectangular geometry and horizontal/vertical layout constraints;
- bounded viewport rendering, clipping, and stable cursor placement;
- focus routing across components or panels;
- typed mouse events including coordinates, buttons, movement when needed,
  and wheel input;
- frame-local hit regions mapped to opaque interaction IDs;
- panel-local scrolling and overlay composition above rectangular layouts.

The existing line renderer remains useful for leaf components. The component
API should evolve additively where practical: existing width-based components
can be adapted into a bounded viewport and clipped to its height while new
components can return cursor and hit-region metadata.

Introducing a replacement TUI framework and rewriting the existing renderer
is not part of this task. Such a change would require separate evidence and a
separate migration decision because the current implementation already owns
terminal lifecycle, ANSI width, Markdown, differential rendering, cursor
placement, input normalization, images, and deterministic terminal tests.

### Ownership And Boundaries

- `pi-tui` owns generic terminal modes, layout geometry, viewport rendering,
  input normalization, focus, hit testing, and terminal cleanup.
- `pi-coding-agent` owns the product layout, conversation/context/composer
  panels, transcript selection, folding defaults, block summaries, and
  product-specific interactions.
- `pi-agent-core` must not gain panel, focus, mouse, or folding concepts.
- `pi-ai` must remain independent of all terminal presentation concerns.
- `SessionEvent` remains the durable source of session facts; folding and
  panel state remain ephemeral adapter state.
- Product adapters continue to consume product events and snapshots. The TUI
  must not bypass product contracts to obtain panel data.
- Print, JSON, and RPC stdio behavior must remain unchanged and protocol-clean.

### Delivery Phases

The task is delivered in independently testable phases. Later phases depend on
the contracts established by earlier phases.

#### Phase 1: Full-Screen Terminal Substrate

- Add explicit `Inline` and `Fullscreen` modes.
- Add alternate-screen entry/exit and full-frame rendering.
- Preserve cursor, raw mode, keyboard protocol, bracketed paste, progress, and
  image cleanup behavior.
- Restore terminal state after normal exit, abort, setup failure, runtime
  error, and panic cleanup paths.
- Keep the existing product layout operational before introducing panels.

#### Phase 2: Rectangular Layout And Focus

- Add `Rect`, bounded viewport rendering, clipping, and deterministic split
  layout primitives.
- Add focus identity and focus traversal.
- Place status, conversation, composer, and footer in fixed regions.
- Provide a generic overlay host for selectors, settings, confirmations, and
  plugin dialogs. Existing inline menus may remain temporarily only when they
  are recorded as execution debt with a migration trigger.
- Add responsive collapse behavior for small terminal sizes.

#### Phase 3: Interactive Transcript Blocks

- Expose or derive stable view identities for transcript blocks.
- Add transcript selection and per-block display state.
- Add thinking, tool argument, tool result, bash, and delegation previews.
- Render supported image content through `pi-tui` image components and honor
  the existing `show_images` and `image_width_cells` settings.
- Preserve the user's visual anchor when an earlier block changes height.
- Preserve bottom-follow behavior only while the user remains at the bottom.

#### Phase 4: Mouse And Hit Testing

- Add terminal mouse capture and SGR mouse parsing.
- Add typed mouse input and frame-local hit regions.
- Support pointer-local scrolling, block selection, and disclosure controls.
- Ensure keyboard-only operation remains complete.
- Disable mouse capture and restore terminal state on every exit path.

#### Phase 5: Context Panel And Responsive Product Layout

- Add a secondary context panel with tabs for operation state, changes,
  delegated agents, and usage where those facts already exist in product
  events or snapshots.
- Do not invent direct service reads from the adapter when the product event or
  snapshot contract should own the data.
- Hide or overlay the context panel on narrow terminals.
- Retain conversation and composer priority at all supported sizes.

### Migration And Compatibility Strategy

The migration begins with both inline and full-screen modes available through
CLI/configuration. Full-screen should be opt-in during initial implementation
and compatibility testing. Making it the default requires evidence that the
terminal lifecycle and interaction contracts pass across supported terminals,
tmux, SSH sessions, resize behavior, and offline smoke tests.

The exact option names are not frozen by this plan, but the user-visible model
must remain equivalent to:

```text
tui mode = inline | fullscreen
```

Inline mode must not silently regress while full-screen remains experimental.
This requirement protects user-visible terminal and scrollback behavior; it
does not require two independent product state machines or rendering models.
Inline and full-screen modes should share transcript state, actions, product
events, leaf components, and modal commands wherever their presentation
constraints permit.

Tests that protect supported inline behavior remain valid. Tests that assert a
superseded internal line composition or require duplicate production branches
must be updated to the shared target design instead of driving permanent
compatibility code. A later explicit completion decision may retire inline
mode after its supported use cases and terminal evidence are reviewed; this
task does not assume that outcome in advance.

### Acceptance Criteria

`V03-002` is complete when all of the following hold:

- Full-screen mode owns the complete terminal viewport without corrupting the
  primary screen or pre-existing shell scrollback.
- Normal exit, abort, error, and panic cleanup restore raw mode, cursor,
  alternate screen, keyboard protocol, bracketed paste, progress state, mouse
  capture, and image state as applicable.
- Conversation, composer, status, footer, overlays, and optional context panel
  render in bounded non-overlapping regions after startup and resize.
- Conversation and context panels can scroll independently.
- Streaming output follows the bottom only while the user has not scrolled
  away; new output is indicated without stealing the viewport.
- Thinking and tool blocks can be individually collapsed, previewed, and
  expanded using both keyboard and mouse.
- Changing a block's height preserves a coherent visual anchor rather than
  unexpectedly jumping to another part of the transcript.
- Collapsed blocks retain useful identity, status, and hidden-content
  information.
- Narrow and low-height layouts retain a usable conversation and composer and
  do not overlap text or controls.
- Unicode, CJK, emoji, ANSI styling, Markdown wrapping, cursor placement, and
  supported terminal images remain correctly framed.
- `show_images = false` suppresses terminal image rendering, while
  `image_width_cells` bounds rendered image width when images are enabled.
- Keyboard workflows remain complete when mouse support is unavailable.
- Inline mode continues to satisfy its existing scrollback and lifecycle
  contracts during the compatibility period.
- Print, JSON, and RPC adapters produce no new TUI output and retain protocol
  cleanliness.
- No TUI presentation types leak into `pi-agent-core` or `pi-ai`.

### Test Plan

The implementation requires layered tests proportional to the terminal risk:

- unit tests for rectangle splitting, clipping, responsive layout, focus
  traversal, mouse parsing, hit testing, and block display-state transitions;
- virtual-terminal tests for exact panel geometry, cursor placement, resize,
  independent scrolling, streaming follow behavior, and visual-anchor
  preservation;
- transcript rendering tests for collapsed, preview, expanded, streaming,
  error, Unicode, Markdown, tool, bash, thinking, and delegation states;
- terminal lifecycle tests for alternate-screen and mouse enable/disable
  sequences on every supported cleanup path;
- adapter tests proving view state is not persisted as a session fact and that
  product events remain the source of displayed runtime state;
- expanded `scripts/tui-smoke.sh` coverage for inline and full-screen modes,
  tmux capture, resize, abort, exit, scrollback preservation, and offline
  streaming fixtures;
- full workspace formatting, strict Clippy, tests, and the applicable release
  gates before changing the default mode.

### Non-Goals

- Replacing `pi-tui` with another TUI framework as part of this task.
- Persisting panel focus, scroll positions, or folding state as session facts.
- Adding an embedded shell, terminal emulator, general file browser, or full
  IDE workspace in the first full-screen release.
- Moving product operation, profile, delegation, change, or usage semantics
  into `pi-tui`.
- Allowing the interactive adapter to read lower-level runtime internals in
  place of product events or snapshots.
- Removing inline mode before full-screen lifecycle and compatibility evidence
  is complete.

### Planning Decisions

- The target is a full-screen application shell with interactive transcript
  blocks, not a cosmetic restyle of the inline renderer.
- Conversation and composer remain the first-viewport priority.
- The initial product layout uses one primary conversation panel and one
  optional tabbed context panel.
- Folding is per block or section and uses collapsed, preview, and expanded
  states.
- Folding and panel state are ephemeral TUI state.
- Mouse interaction is additive; every action retains a keyboard equivalent.
- The migration extends the existing renderer and terminal abstractions unless
  separate evidence justifies replacement.
- Full-screen mode is opt-in until terminal lifecycle and smoke evidence is
  sufficient to consider changing the default.
- Inline and full-screen presentations share one product interaction model;
  compatibility tests do not justify duplicated runtime or adapter state.

## `V03-003`: Truthful RPC Runtime Surface

### Summary

Make RPC protocol capabilities and successful responses describe behavior that
the RPC adapter can actually execute. Remove silent state-only acknowledgements
and connect existing runtime functionality where the public protocol already
models it.

The current RPC surface has four related defects:

- generic runtime capabilities are projected without intersecting them with
  RPC command support;
- some setters update only adapter fields and then report success;
- session statistics contain hard-coded zero values despite available usage
  and message data;
- several DTO fields or command variants are parsed but rejected or ignored at
  the adapter boundary.

### Required Scope

#### Adapter-specific capability projection

RPC capability status must be the intersection of:

```text
runtime capability
AND RPC protocol support
AND current adapter/session state
```

An operation must not be advertised as `available` unless the negotiated RPC
version accepts a command that can invoke it. This applies at minimum to
`compact`, `fork`, `cloneSession`, `branchSummary`, `switchSession`, and
`export`.

For every capability, implementation must choose one explicit outcome:

1. add the typed RPC command and execute the existing runtime operation;
2. report `unsupported` with a stable reason;
3. remove the capability from a future protocol-major revision.

Capability projection must not silently reuse a broader embedding facade when
the adapter exposes a narrower operation set.

#### Runtime-backed mutable settings

The following commands must either affect subsequent runtime behavior or
return an unsupported/error response:

- `set_steering_mode`;
- `set_follow_up_mode`;
- `set_auto_compaction`;
- `set_session_name`.

`set_thinking_level` already reaches subsequent prompt options and serves as a
reference for a runtime-backed setter. State echo in `get_state` is not
sufficient acceptance evidence.

Queue-mode changes must affect queue admission/drain semantics. Auto-compaction
changes must affect the `CompactionConfig` used by later prompt operations.
Session naming must receive one defined semantic: durable session metadata or
explicit adapter-local display state. The command name and response must match
that choice.

#### Accurate session statistics

`get_session_stats` must derive values from authoritative runtime/session data:

- user, assistant, tool-call, and tool-result counts;
- input, output, cache-read, cache-write, and total token counts;
- accumulated cost using the existing usage representation;
- active session and leaf identity.

Unknown data must be represented as absent/unknown according to the protocol,
not fabricated as zero. Bash, custom, and branch-summary records must not be
misclassified as user messages merely to produce a count.

#### Existing modeled-but-disconnected inputs

Resolve the following protocol gaps:

- `prompt`, `steer`, and `follow_up` image content should use the existing
  multimodal `PromptInvocation::Content` and prompt-control representation, or
  the fields must be explicitly removed at a protocol-major boundary;
- `new_session.parentSession` must create the defined parent/fork relationship,
  be validated and rejected when invalid, or be removed from the protocol;
- `compact` must invoke the existing manual compaction operation when the
  session capability is available instead of always returning an M5 error.

### Ownership And Boundaries

- RPC DTOs and adapter-specific capability projection stay in `protocol/` and
  `adapters/rpc/`.
- Durable usage and session facts continue to come from session/runtime owners;
  the RPC adapter must not create a second accounting model.
- Runtime settings must enter operations through existing prompt/runtime
  options rather than mutable global state in lower crates.
- Multimodal request handling reuses existing `ContentBlock` and
  `PromptInvocation` types.
- No RPC compatibility fallback may corrupt JSONL output or protocol
  negotiation.

### Dependencies And Sequencing

Capability auditing, statistics, multimodal input, and runtime-backed setting
work may begin independently. RPC abort behavior must consume the uniform
operation cancellation contract from `V03-004`. RPC list/approve/deny support
for pending tool authorization must consume the request, decision, event, and
snapshot contracts from `V03-008`; it must not define competing RPC-local
authorization state.

When an existing protocol test encodes a response that was accepted but never
executed, the implementation must choose and test a truthful supported or
unsupported outcome. It must not retain the false acknowledgement through a
shadow field or no-op compatibility branch solely to preserve that test.

### Acceptance Criteria

- Every RPC capability marked `available` has a successfully parseable and
  executable command in the negotiated protocol version.
- Unsupported RPC operations are reported as `unsupported`, not `available` or
  silently ignored.
- Steering and follow-up mode changes alter queue behavior in a provider-backed
  request test.
- Auto-compaction enable/disable changes the config used by later prompt runs.
- Session-name behavior is defined, tested across reconnect/reopen as
  applicable, and no longer exists only as an unverified adapter field.
- `get_session_stats` reports non-zero recorded usage and tool counts from a
  deterministic provider/session fixture.
- RPC multimodal prompt input reaches the provider request and durable
  transcript using the same content semantics as CLI/interactive input.
- `parentSession` is either functional and validated or absent from the next
  protocol contract; it is never accepted and discarded.
- RPC manual compaction follows runtime admission, persistence, events,
  cancellation, and terminal outcome contracts.

### Test Plan

- command-by-capability matrix tests for idle, busy, persistent, and
  non-persistent sessions;
- behavioral tests for each mutable setting, not only `get_state` echo tests;
- provider request tests for queue modes, auto-compaction configuration, and
  image content;
- session/replay-backed statistics tests with non-zero tokens, cache usage,
  cost, and tool calls;
- `parentSession` success, missing-parent, and invalid-parent tests;
- JSONL cleanliness and protocol-version tests for every added or changed
  command.

### Non-Goals

- Duplicating runtime operations inside the RPC adapter.
- Returning placeholder values to preserve a misleading response shape.
- Advertising future commands before their typed protocol and execution path
  exist.

## `V03-004`: Uniform Asynchronous Operation Cancellation

### Summary

Extend typed cancellation beyond prompt operations so every asynchronous
operation exposed by interactive, RPC, or embedding APIs has a coherent abort
path and terminal outcome.

The runtime already owns operation identities, guards, cancellation tokens,
parent/child lineage, and operation-specific events. Adapters must use that
authority instead of dropping futures and reporting adapter-specific
`UnsupportedMode` errors.

### Required Scope

Cancellation must cover at minimum:

- prompt and manual compaction;
- agent and team invocation;
- branch summary and branch-summary navigation;
- session fork/navigation operations;
- self-healing edit;
- plugin load and plugin command;
- delegation approval/rejection operations where cancellation remains
  meaningful before commit.

The operation owner must decide whether cancellation is cooperative, immediate,
or no longer possible after a commit boundary. Adapters report that typed
result rather than inventing their own semantics.

Parent cancellation must cancel active children without allowing them to
outlive the root. Cancellation must preserve durable transaction and
operation-event invariants: an operation cannot disappear without a terminal
or recoverable interrupted state.

### Acceptance Criteria

- Interactive and RPC abort controls can target every active cancellable
  asynchronous operation by operation identity.
- Capability snapshots mark abort available for cancellable non-prompt
  operations and keep steer/follow-up prompt-specific.
- Cancellation produces a typed cancelled/aborted outcome and the appropriate
  product event family.
- Session-write operations leave no partially committed success record.
- Child operation cancellation propagates through delegation and team lineage.
- Terminal cleanup and adapter ownership are restored after cancellation.
- No production path contains `interactive ... abort is not implemented yet`.

### Test Plan

- deterministic blocked-operation tests for each operation family;
- cancellation before work, during provider streaming, during tool execution,
  before commit, and after the point where cancellation must be rejected;
- parent/child and team cancellation propagation tests;
- interactive and RPC adapter tests asserting typed terminal behavior;
- recovery tests for a process interruption during cancellation.

### Non-Goals

- Making steer or follow-up valid for non-prompt operations.
- Treating future-drop alone as a complete cancellation contract.
- Hiding incomplete cleanup behind adapter error strings.

### Completion Record

Completed on 2026-07-16 without execution debt.

- Runtime-owned root and child operations now expose identity-scoped
  cancellation handles. Parent cancellation cascades through admitted child
  lineage, repeated cancellation is idempotent, and the owner closes a shared
  commit gate when cancellation is no longer safe.
- Public client control stores concurrent operation bindings by operation ID,
  preserves them across connection-generation takeover, and reports the typed
  `NoLongerCancellable` rejection after the commit boundary.
- Prompt, compaction, agent/team invocation, delegation children, branch
  summary and navigation, session fork/navigation, self-healing edit, plugin
  load/command, and meaningful pre-commit delegation mutations use the same
  owner-side cancellation contract.
- Interactive abort no longer drops asynchronous workflow futures or emits
  operation-specific "not implemented" errors. RPC abort accepts an optional
  operation ID so foreground and background work can be targeted independently.
- Abort capability projection now applies to every active cancellable root,
  while steer and follow-up remain prompt-specific. Self-healing cancellation
  has a typed aborted product/protocol event, and cancelled branch-summary
  navigation no longer proceeds to fork.
- Deterministic owner, commit-gate, reconnect, concurrent-root, workflow,
  parent/child, interactive, RPC, recovery, event-contract, and boundary tests
  passed. `scripts/tui-smoke.sh`, strict workspace Clippy, formatting, and the
  full workspace all-features test suite also passed.

## `V03-006`: Executable Plugin UI Actions

### Summary

Give registered plugin UI actions an explicit execution contract. A generic UI
action currently reaches the interactive dispatcher but only prints a system
notice unless its `action_id` happens to match a plugin command or dialog.

### Required Design Decision

Choose and enforce one of these models:

1. **Reference-only actions:** every UI action references a registered command
   or dialog, and plugin load rejects unresolved action IDs.
2. **Executable actions:** `UiProvider` exposes a typed action executor behind
   `PluginService`, with capability checks, diagnostics, cancellation, and
   product events.

The repository must not continue accepting a third, non-executable action
category that appears successful in the UI.

Keybindings, menus, dialogs, and future full-screen controls must resolve the
same action inventory. Dialog submission should invoke the declared action
directly rather than manufacturing editor text as an internal transport.

The design decision may be completed before `V03-004`, but executable plugin
actions that perform asynchronous work must use the shared operation identity,
cancellation, and terminal-outcome contract. Full-screen plugin controls in
`V03-002` must consume this action inventory rather than preserving the current
notice-only path as a compatibility fallback.

### Acceptance Criteria

- Plugin load rejects unresolved action or keybinding targets with actionable
  diagnostics.
- Activating every registered UI action either executes its declared target or
  returns a typed failure.
- Capability denial, plugin failure, panic isolation, cancellation, and output
  are handled through `PluginService` and product events.
- Interactive tests verify actual execution, not only pending-action state or a
  rendered notice.
- The contract is reusable by the full-screen TUI without embedding plugin
  runtime logic in `pi-tui`.

### Non-Goals

- Moving product plugin semantics into `pi-tui`.
- Treating a transcript notice as successful action execution.

### Completion Record

Completed on 2026-07-16 without execution debt.

- Selected and documented the reference-only model. Commands are executable
  leaves, dialogs submit to commands, UI actions target commands or dialogs,
  and keybindings reference targets exposed by registered UI actions.
- `PluginService` now validates the complete command/dialog/action/keybinding
  graph before installation. Unresolved references fail plugin load with
  deterministic diagnostics containing the owning plugin, contribution ID,
  target ID, and expected target kind.
- Plugin registration failures and panics remain isolated by `PluginService`
  and are included in the plugin-load diagnostic outcome. Invalid loads do not
  publish loaded IDs or install a partial plugin service.
- Interactive action activation now has one reusable target router for current
  keybindings and future full-screen controls. Command targets use the
  canonical capability-aware `PluginCommand` operation; dialog targets open
  the typed form and submit its JSON arguments directly to that command path.
- Removed the notice-only `PendingPluginUiAction` path and the editor-text
  transport that manufactured `/plugin-command` input for dialog submission.
- Capability denial, plugin execution failure, command/UI-provider panic
  isolation, cancellation ownership, command output, product terminal events,
  and adapter ownership continue through the existing plugin service and
  operation contracts.
- Valid and invalid reference graph tests, direct action activation,
  keybinding-to-command execution, dialog opening/submission, capability,
  panic, event, API, RPC, and boundary tests passed. `scripts/tui-smoke.sh`,
  strict workspace Clippy, formatting, and the full workspace all-features test
  suite also passed.

## `V03-008`: Interactive Tool Authorization And Approval Workflow

### Summary

Add a first-class authorization workflow for tool invocations that require a
user decision before execution. The interactive adapter must automatically
present pending requests in a focused confirmation overlay, but the security
contract belongs to the product runtime and must also define deterministic
behavior for RPC, JSON, print, embedding, cancellation, and reconnect paths.

This task is not satisfied by adding a modal after `ToolCallStart`. The current
tool path emits `ToolCallStart` before the asynchronous `before_tool_call` hook
and then executes the tool when the hook does not block it. Authorization must
be resolved before the runtime reports that a tool has started and before any
tool side effect is possible.

The existing delegation-confirmation path is related but incomplete as a TUI
interaction reference. A `DelegationConfirmationRequired` event currently
adds command instructions to the transcript; users must run `/delegations` to
open an inline menu. It does not automatically open a modal and does not use
the generic `pi-tui` overlay surface.

### Security Model

Keep operation capabilities and invocation authorization as separate layers:

- a capability snapshot defines the maximum tools and resources an operation
  may access;
- an authorization policy decides whether a particular invocation is allowed
  immediately, denied, or requires an interactive decision;
- an approval may narrow or exercise an existing capability but must never
  grant a tool, path, shell, plugin, or resource capability absent from the
  operation snapshot;
- capability generation changes and revocation remain authoritative while an
  authorization request is pending.

The initial decision surface is:

- `AllowOnce` for the exact pending invocation;
- `AllowForOperation` for a bounded operation-local grant matching an explicit
  tool and resource scope;
- `Deny` with an optional user-visible reason.

Persistent `AlwaysAllow` policy editing is outside the initial delivery. It
requires a separate configuration, scope-pattern, confirmation, and audit
contract and must not be approximated by silently persisting an operation
grant.

### Authorization Policy And Risk Inventory

The product layer must own structured authorization policy and tool-risk
metadata. The first implementation may classify built-in tools through a
central product-owned inventory, but it must provide an extension path for
plugin-declared risk rather than relying indefinitely on scattered tool-name
checks.

Recommended initial defaults are:

| Invocation | Default decision |
| --- | --- |
| Workspace-local `read`, `grep`, `find`, and `ls` | Allow |
| Filesystem reads outside the granted workspace scope | Ask |
| `write` and `edit` | Ask |
| `bash` | Ask |
| Plugin tools declared read-only and local | Allow |
| Plugin tools with mutation or external side effects | Ask |
| Plugin tools without risk metadata | Ask |

Authorization previews and policy evaluation must use normalized effective
paths and execution parameters, not only raw model arguments. Shell previews
must identify the effective command and working directory. Plugin and shell
arguments, environment data, diagnostics, snapshots, and events must preserve
existing secret-redaction rules.

Delegation continues to use `DelegationPolicy` and its own confirmation modes.
Its confirmation UI should later reuse the same modal interaction primitives,
but delegation authorization must not be collapsed into generic tool-risk
policy.

### Runtime Contract

Define typed product-owned request and decision records with stable identities.
A pending request must carry enough information to validate a decision and to
render a safe preview, including at minimum:

- authorization, operation, turn, and tool-call identity;
- tool name and structured risk classification;
- normalized invocation preview and requested resource scope;
- capability generation and request timestamp;
- redacted display data distinct from authoritative execution arguments.

The runtime path must become:

```text
model emits tool call
        |
        v
validate operation capability and authorization policy
        |
        +--> allow ------> ToolCallStart ------> execute
        |
        +--> deny -------> structured tool error
        |
        `--> ask --------> AuthorizationRequired
                              |
                              v
                         await typed decision
                              |
                    +---------+---------+
                    |                   |
                  allow                deny
                    |                   |
                    v                   v
              ToolCallStart      structured tool error
                    |
                    v
                 execute
```

Waiting for a decision must be asynchronous, cancellation-aware, and owned by
the active operation. It must not block a Tokio worker or move tool execution
into an adapter. Decisions must be rejected when they are stale, duplicated,
refer to another operation or tool call, or no longer match the active
capability generation.

The generic `pi-agent-core` hook boundary may be extended with the execution
identity and cancellation context necessary to host an asynchronous gate.
Product policy, pending queues, product events, persistence decisions, and UI
semantics stay in `pi-coding-agent`. `pi-agent-core` must not gain TUI types,
product risk policies, or approval button semantics.

Tool event ordering must distinguish a proposed or authorization-pending call
from an executing call. `ToolCallStart` and the product `Tool.Started` event
must mean that authorization has completed and execution is beginning.

Parallel tool batches require deterministic authorization behavior. No call
that requires approval may start before its decision. The initial
implementation may serialize approval-required calls while retaining parallel
execution for calls that are already authorized, provided transcript ordering,
cancellation, and terminal results remain deterministic.

### Product Events, State, And Recovery

Add typed product events for at least:

- authorization required;
- authorization approved;
- authorization denied;
- authorization cancelled, expired, or interrupted.

The event contract must expose redacted preview data and stable request
identity without exposing secret-bearing authoritative arguments. Session
facts should record request and resolution for audit and replay where useful.
The live runtime remains the authority for whether a suspended invocation can
still resume.

UI and RPC snapshots must include active pending authorizations so a reconnect
can reconstruct the decision surface. The first implementation does not need
to resurrect a suspended tool future after process restart; unresolved
requests found during recovery may be completed as interrupted or denied, but
must not disappear or be replayed as approved.

Operation abort, session shutdown, capability revocation, and terminal prompt
failure must resolve or cancel all associated pending requests. No request may
outlive its operation.

### Adapter Behavior

The interactive adapter must automatically open an authorization surface when
it receives an authorization-required event. It must not require the user to
notice transcript instructions or invoke a slash command.

The interactive surface must show, when applicable:

- tool name, risk category, profile or operation, and pending queue position;
- normalized filesystem target, shell command, working directory, or plugin
  action summary;
- a bounded diff or content preview for `write` and `edit`;
- `Allow once`, `Allow for operation`, and `Deny` actions.

The surface must take keyboard focus while active. `Escape` must not approve a
request; the exact dismiss-versus-deny behavior must be explicit and tested.
`Ctrl+C` continues to abort the owning operation. Multiple requests are queued
and shown deterministically rather than stacked as competing dialogs.

Before the full-screen migration is complete, the interactive adapter may use
a bounded inline confirmation fallback. After `V03-002` supplies rectangular
layout, focus, and overlay composition, the same state and commands must render
as a true modal overlay without changing runtime semantics. Resize, terminal
cleanup, and keyboard-only operation remain mandatory. Mouse actions may be
added through the typed hit-testing work in `V03-002` but are not required for
the authorization workflow to be complete.

Delegation confirmation should be migrated from transcript command guidance
and the manually opened inline menu to the shared automatic modal interaction
pattern. Existing `/delegations` and explicit approve/reject commands may
remain as discoverable and recovery-oriented alternatives.

RPC must provide typed commands to list, approve, and deny pending requests.
Print and JSON modes must never wait indefinitely for interactive input: they
use an explicit non-interactive policy, defaulting to denial for invocations
that require confirmation. Any `allow-all` mode must require an explicit user
choice and still cannot bypass capability checks.

### Delivery Phases And Dependencies

1. Define risk metadata, authorization policy, request/decision types, pending
   state, and capability-generation validation.
2. Add the execution gate, correct tool event ordering, denial results,
   parallel-call rules, cancellation, and operation-local grants.
3. Add product/session events, snapshots, recovery behavior, and RPC commands.
4. Deliver the keyboard-complete inline interactive fallback and automatic
   event-driven opening.
5. Integrate the authorization surface with the full-screen overlay substrate
   from `V03-002` and migrate delegation confirmation to the shared pattern.
6. Complete redaction, reconnect, cancellation, parallel execution, adapter,
   and terminal-lifecycle regression coverage.

The policy, event, and pending-state contracts do not depend on completion of
`V03-002`. The execution gate may begin once the applicable waiter
cancellation and terminal-outcome contract from `V03-004` is settled. The
keyboard-complete inline fallback does not depend on full-screen layout.
Full-screen modal presentation depends on the overlay and focus substrate from
`V03-002`. RPC commands consume the shared protocol projection work in
`V03-003`. None of these phases may create an adapter-local authorization or
abort path to work around an unfinished dependency; any unavoidable temporary
bridge must be recorded as execution debt and removed in Wave 4.

### Acceptance Criteria

- A tool invocation requiring authorization cannot execute or emit
  `Tool.Started` before a valid approving decision.
- Capability denial cannot be overridden by any authorization decision.
- The interactive adapter automatically presents every pending request and can
  approve or deny it without slash-command entry.
- `AllowOnce` affects only the selected invocation, and `AllowForOperation`
  cannot escape its operation, tool, or declared resource scope.
- Denial produces a structured tool result that lets the model continue or
  terminate according to normal agent-loop rules.
- Abort, shutdown, revocation, and prompt failure leave no executable orphaned
  request or unresolved waiter.
- Stale, replayed, duplicated, cross-operation, and generation-mismatched
  decisions are rejected deterministically.
- Parallel batches do not start approval-required calls early and preserve
  stable tool-result ordering.
- RPC clients can discover and resolve pending requests through typed protocol
  commands and snapshots.
- Print and JSON modes never hang waiting for authorization and preserve
  stdout/JSONL cleanliness.
- Full-screen mode presents a focused, bounded, resize-safe modal overlay;
  inline mode retains a keyboard-complete fallback.
- Authorization previews, product events, snapshots, transcripts, diagnostics,
  and test failures do not expose credentials or secret environment values.
- Delegation confirmation automatically uses the shared confirmation
  interaction while preserving its existing policy and durable state model.

### Test Plan

- blocked tool fixtures proving no filesystem, shell, plugin, or external side
  effect occurs before approval;
- allow-once, operation-grant, denial, stale decision, duplicate decision, and
  capability-revocation tests;
- cancellation while waiting, session shutdown, prompt failure, and reconnect
  snapshot tests;
- sequential, parallel, and mixed authorized/pending tool-call tests with
  deterministic event and result ordering;
- provider-loop tests proving denial is returned as a tool result and the model
  can continue;
- RPC list/approve/deny protocol and JSONL-cleanliness tests;
- print and JSON non-interactive policy tests;
- virtual-terminal tests for automatic opening, focus routing, queue order,
  keyboard actions, resize, clipping, diff preview, and terminal cleanup;
- `scripts/tui-smoke.sh` coverage for an offline authorization round trip;
- redaction tests for commands, paths, environment-derived values, plugin
  arguments, product events, snapshots, and diagnostics;
- delegation confirmation regression tests for automatic presentation and
  explicit command fallback.

### Non-Goals

- Treating a capability snapshot as proof that every invocation is already
  user-approved.
- Implementing authorization solely in the TUI or executing tools from an
  adapter callback.
- Showing a confirmation after tool execution has begun.
- Persisting permanent allow rules in the initial implementation.
- Moving product authorization policy, delegation policy, or modal semantics
  into `pi-agent-core` or `pi-tui`.
