# TypeScript/Wasm Extension Kernel Replacement Plan For 0.4.2

## Status And Scope

Target workspace version: **`0.4.2`**. Status: **in progress and dependency-reviewed;
unblocked by completed `0.4.1`**. Workspace packages remain at `0.4.1` until
all implementation, debt, and release gates below close.

This release establishes only the minimum viable TypeScript/Wasm extension
framework. It does not migrate current contributions for feature parity or claim
a complete end-user extension product. Lua/native extension paths and obsolete
trait-object provider infrastructure are still removed to avoid redundant
runtimes. Built-in Rust AI providers owned by `pi-ai` are unaffected.

See the [`0.4.x` roadmap](0.4.x-architecture-extension-platform-roadmap.md).

## Fixed Contract

```text
authoring language = TypeScript
installed artifact = Wasm Component
runtime ABI = versioned WIT + language-neutral schemas
host/core implementation = Rust, not an extension language
```

Installed extensions run without Node.js. The host never directly executes
TypeScript/JavaScript, loads native Rust plugins, accepts extension trait objects,
or selects among runtime kinds.

## Objectives And Non-Goals

- Manifest v2, WIT, generated TypeScript bindings, and minimal SDK;
- package/digest/lock/quarantine/install baseline;
- `GrantRecord -> ExtensionInstanceGrant -> OperationCapabilityLease`;
- minimal immutable compiled-component cache and isolated invocation
  Store/Instance;
- one real lease-backed Host call with deadline, cancellation, epoch, fuel,
  memory, and output bounds;
- deletion of Lua/native extension-tier and obsolete contribution-provider
  infrastructure without implementing replacement contribution parity;
- deletion of generic Flow production/compatibility scaffolding superseded by
  the typed runners completed in `0.4.1`;
- no advanced services/state, Workbench, public Extension CLI, or complete
  Extension DX in the current train.

The former `0.4.3` through `0.4.5` Extension plans are explicitly Skipped. They
must be replanned rather than inferred as follow-up obligations.

## Detailed Acceptance

Task-ledger row order is the implementation order. Detailed sections remain in
stable numeric ID order for cross-version traceability.

`EKR-006` includes its own minimal DTO/core-versus-extension dispatch prototype
as the acceptance evidence for `ADR-012`. This avoids a dependency cycle:
`EKR-004` consumes the accepted handler-reference boundary for the full Wasm
vertical slice, while `EKR-006` does not claim the Wasm runtime itself complete.

### `EKR-001`: Contracts And Package Baseline

- Manifest declares identity, API ranges, component artifact/digest, WIT world,
  dependencies/lock, activation, permissions, contributions, and limits.
- Manifest cannot select runtime/language or self-declare source/trust.
- Quarantine validates byte/package concerns: layout, paths, links, duplicates,
  entry/size limits, digests, dependency lock integrity, and resources.
- Version manifest, WIT, Host API families, contributions, and SDK separately.
- Grant-backed workspace activation in `EKR-003` activates at most one version
  of an extension ID; dependency permissions never transfer.
- High-level ADR decisions are accepted here, but concrete WIT/schema details
  remain release candidates until the `EKR-004` vertical slice proves package,
  admission, invocation, error, cancellation, and disposal behavior.

Progress evidence: ADR-007 and ADR-008 are accepted. Candidate contract sources
live under `contracts/extensions/0.1.0/` with independent Manifest v2,
contribution-schema v1, and WIT `0.1.0` versions plus checked hashes. A strict
private Rust Manifest v2 parser rejects unknown/forged runtime, language, source,
trust, grant, and lease fields; validates semantic versions/ranges, component
path/world/digest, dependencies, contribution IDs/revisions, resources, and
limits. `scripts/extension-contracts.sh` verifies hashes, JSON structure,
forbidden schema fields, and parser tests. Dependency lock v1 now binds the root
identity and exact dependency versions/digests to Manifest ranges. Directory
quarantine rejects symlinks, unknown and case-colliding paths, non-regular files,
excess depth/entries/bytes, digest mismatches, and incomplete resource
inventories. Its result is explicitly non-admissible until the immutable store
revalidates and publishes it. The package store now owns staging creation,
content-addressed whole-package digests, dependency-presence/identity checks,
idempotent installation, atomic publication, and read-only installed trees;
activation never reopens the acquisition path. Exact installed dependencies must
exist before a dependent package can be installed, so cycles cannot enter the
store. `EKR-001` is complete. Host-supported permission checks and explicit
workspace activation belong to the grant owner in `EKR-003`; Component
import/export validation belongs to the real-engine boundary in `EKR-004`.

### `EKR-002`: TypeScript Contract

- Generate declarations/SDK types from WIT/schema authoritative sources.
- Define supported TypeScript/ECMAScript/bundling subset and reject Node built-ins,
  native addons, dynamic loading, and undeclared WASI.
- A minimal TypeScript fixture compiles, bundles, componentizes, and validates.
- Provide repository-owned offline `build-fixture`, `validate-fixture`, and
  `run-conformance` workflows with locked compiler, bundler, componentizer, WIT
  generator, Wasm validator, inputs, and artifact hashes. This is the supported
  internal precursor to the public CLI delivered in `0.4.5`.
- Record whether each tool is vendored, preinstalled, or restored from a locked
  cache so offline release validation is reproducible rather than aspirational.
- Installed execution requires no Node.js/build tooling.
- Handwritten bindings cannot become a second ABI source.

Completion evidence: `sdk/typescript` pins TypeScript `7.0.2`, esbuild `0.25.12`,
Jco `1.25.2`, componentize-js `0.21.0`, and json-schema-to-typescript `15.0.4`
through `package-lock.json`. Node.js `>=20` is a preinstalled runner; npm tools
are restored only through `npm ci --offline` from a populated integrity-checked
cache into `target/extension-sdk/toolchain`. `scripts/extension-sdk.sh` exposes
the required `build-fixture`, `validate-fixture`, and `run-conformance` commands.
Each run generates WIT and Manifest/contribution/lock declarations from the
authoritative contracts, strict-typechecks the SDK/fixture, produces one ES2022
bundle, componentizes with all ambient WASI disabled, parses embedded WIT,
rejects residual imports/dynamic or native loading/runtime code generation, and
checks source/generated/bundle/Component hashes. No generated artifact or
toolchain dependency is written outside the repository `target/` tree.

### `EKR-003`: Grants And Leases

- Define the host-supported permission catalog and reject unsupported requested
  permissions before activation.
- Persist explicit workspace activation snapshots resolved only from immutable
  packages; reject dependency cycles and multiple active versions of one ID.
- Requested permissions, persisted grants, instance grants, and operation leases
  are distinct.
- Every privileged Host API validates lease, operation, generation, scope,
  deadline, and cancellation per call.
- Revoke blocks admission, cancels old leases, closes registrations, and discards
  uncommitted late results.
- Secrets/authorization data are redacted everywhere.
- No raw repositories, provider clients, channels, Flow registries, adapter state,
  or core handler IDs are exposed.

Progress evidence: candidate GrantRecord v1 and workspace activation v1 schemas
now version the host-owned durable records independently. The supported catalog
currently contains scoped model invocation, process execution, UI interaction,
and workspace read/write permissions. Session association remains lease scope;
generic session state/facts stay in `0.4.3`. `ExtensionGrantRegistry`
installs per-workspace/per-extension generations and mints operation-bound leases
that validate identity, workspace/session scope, permission, deadline, operation
cancellation, instance revocation, and late-result generation on every call.
Workspace activation resolves only immutable content-addressed packages, requires
an independent exact grant for every dependency, rejects unknown permissions and
multiple versions of one ID, writes a bounded `0600` atomic activation record,
and rejects symlinked restart state. Trusted embedding APIs now own store staging,
installation, grant decisions, and activation; `PluginLoad` reloads the durable
snapshot inside its existing transaction and advances runtime capability
generation. Package/store module dead-code allowances are removed, closing
`EKR-D009`. Typed workspace/model/structured-process/UI Host family handles now
return bounded authorization DTOs and contain only frozen leases, never product
services. WIT `0.1.0` declares the same four capability imports; generic session
state remains deferred. Instance revoke wakes active lease cancellation, blocks
new admission, invalidates generation-bound registration handles, and fences
late results. Grant, lease, registration, and authorized-call Debug surfaces
redact source/scope/permission and guest payload material. Focused matrices and
source guards cover allow/deny, wrong identity/scope, deadline, operation cancel,
revoke, replacement, idempotent reload, registration closure, late results,
input bounds, traversal, and raw-service absence. `EKR-003` is complete; `EKR-004`
must consume these handles in the real WIT dispatcher rather than recreate them.

### `EKR-004`: Wasm Runtime

- Validate Component imports/exports against the accepted WIT world and reject
  ambient WASI or undeclared Host API families before compilation/admission.
- First prove one end-to-end vertical slice from Manifest/package quarantine
  through grant, operation admission, Wasm invocation, typed result/error,
  active cancellation, and instance disposal; freeze the concrete WIT/schema
  only after that slice passes.
- Cache immutable compiled components by engine/version/target/digest.
- Each invocation receives isolated Store/Instance/resources and is destroyed.
- Guest memory is non-durable and never shared across workspaces.
- Async host futures honor cancellation/deadline; guest compute uses epochs/fuel;
  memory/host-call/output bounds are enforced.
- Cancelled/trapped/OOM/out-of-fuel/invalid instances are destroyed and not pooled.
- Detached guest tasks and ambient WASI are rejected.
- Pin Wasmtime/toolchain versions and record any MSRV increase.

Completion evidence for the reduced minimum-framework scope: production pins
Wasmtime `46.0.1` with async Component Model/Cranelift and raises the workspace
MSRV to Rust `1.94`. Bindgen consumes the authoritative WIT directly. A package
digest keyed immutable Component cache is prepared before operation admission;
each invocation then creates and drops a fresh Store/Instance with manifest
memory, fuel, deadline, and output limits. A 5 ms epoch callback repeatedly
validates the frozen lease's operation, scope, deadline, revoke, and cancellation
state. The TypeScript fixture imports `host-ui.interact`; the Rust binding routes
that call through `UiHostHandle::authorize_interact` and returns only the bounded
authorized operation-local bytes. `scripts/extension-runtime.sh` builds the
fixture offline under `target/extension-sdk`, passes quarantine/content-addressed
installation, prepares the Component before minting the lease, invokes typed WIT,
and validates structured output. The first debug cold compile took 187 seconds,
which confirmed why compilation cannot live inside the admitted deadline; warm
invocations use the in-memory cache. Full runtime/parity/performance hardening is
still Skipped under `EKR-008`.

### `EKR-005`: Current Contribution Parity

Status: **Skipped** by the 2026-07-19 Extension scope freeze. The acceptance
inventory below is retained as historical scope, not a release obligation.

- Owned DTOs cover tools, commands, Prompt observation/interception hooks, UI
  actions, dialogs, and keybindings.
- `ExtensionHandlerRef` always dispatches through operation admission and Wasm/WIT.
- Structured input/output, progress, cancellation, typed errors, and bounds are
  preserved where current behavior needs them.
- Interactive, RPC, and print/JSON behavior remains clean.

### `EKR-006`: Core/Extension Boundary

- Accept `ADR-012` before contribution migration consumes its handler-reference
  model.
- Built-in Rust behavior is product code, not a first-party extension tier.
- Shared projection uses explicit `CoreHandlerRef` or `ExtensionHandlerRef`.
- Core does not execute through Wasm/extension permissions.
- Extension code cannot address core handlers or receive raw Rust traits.

Completion evidence: ADR-012 is accepted. The private product-owned projection
now uses the exhaustive `HandlerTarget::Core(CoreHandlerRef)` or
`HandlerTarget::Extension(ExtensionHandlerRef)` model without `Deserialize` or
executable authority. Validated Manifest v2 inventories project all six
contribution families to immutable package-bound extension references containing
extension ID, whole-package digest, contribution kind, handler ID, and schema
revision. Workspace activation rejects any package projection that reaches the
core branch and propagates projection errors instead of dropping handlers.
Focused dispatch, malformed-reference, package activation, and source-boundary
tests prove exhaustive routing, language-neutral serialization, fail-closed
identity/digest/revision validation, raw-service absence, and that extension
Manifest/WIT contracts cannot name core handlers. `EKR-006` is complete;
`EKR-004` consumes extension refs through real admission/WIT. Product-owned core
dispatch and full contribution parity remain Skipped with `EKR-005`.

### `EKR-007`: Legacy Removal

Status: **Required**. Contribution parity remains skipped, so obsolete Lua/native
contributions are deleted rather than forwarded through compatibility shims.

Remove `mlua`, `PluginSource::Lua`, Lua constants/loaders/contribution providers/
Host APIs/fixtures/diagnostics, first-party Rust extension concepts,
dynamic/native plugin implications, Rust trait-object extension registries,
dual-runtime dispatch, and temporary legacy forwarding shims. Boundary tests
reject reintroduction. Built-in Rust AI provider registration in `pi-ai` remains.
Release notes must describe the intentional extension break; no Lua/native
extension compatibility runtime or automated migration tool is required.

Completion evidence: `mlua`, runtime/source selection, Lua discovery/loaders,
Lua fixtures, native Rust contribution-provider traits, provider registries,
metadata/source types, and their injection tests are deleted. `PluginLoad` now
only reloads the durable minimum Wasm activation snapshot. A production-source
boundary guard rejects the removed dependency and provider/runtime symbols;
the empty `PluginService`, prompt-hook plumbing, and plugin capability set were
also removed instead of being retained as forwarding shells. Built-in model
providers remain owned by `pi-ai`.

### `CLC-042-001`: Superseded Flow Removal

- Reconfirm every production workflow uses its typed runner and no caller needs
  the generic string-action graph.
- Delete generic Flow engine, legacy workflow builders, compatibility exports,
  tests/fixtures that only preserve the retired abstraction, and dead allowances.
- Preserve lower-level Flow primitives only if a current non-product owner has a

- Keep cancellation, max-step, missing-transition, operation identity, terminal,
  and transaction behavior covered through typed-runner tests and boundary
  guards that reject reintroduction.

Completion evidence: `pi-agent-core` no longer contains or exports the generic
`flow` module, string `Action`, graph/node/error types, graph tests, or the
compatibility `AgentTurnFlow`. The production agent loop is solely
`AgentTurnRunner` plus `AgentTurnDecision`. Every fixed product operation now
lives in a `runner.rs` module behind `WorkflowService`; obsolete graph wrappers,
node tests, and dead test-only adapters were removed. Boundary tests reject old
symbols, source paths, and operation `flow.rs` files.

### `CLC-042-002`: Unreachable Contribution Surface Removal

- Delete the public/internal `PluginCommand` operation and outcome, RPC command,
  interactive pending action, keybinding/dialog dispatch, and adapter-only DTOs

- Keep `PluginLoad` only as the durable minimum Wasm activation reload owner.
- Do not replace the deleted command/UI surface: contribution dispatch remains
  Skipped with `EKR-005` and must be explicitly replanned if ever resumed.
- Add absence guards so an unsupported compatibility operation cannot become a
  permanent substitute for a real versioned WIT contribution contract.

## Release Gates

Full workspace checks/TUI smoke plus focused offline TypeScript componentization,
Wasm vertical, permission, package, boundary, API, and protocol tests pass. The
new framework has no runtime Node.js, ambient WASI, or raw service exposure; no
Lua/native extension runtime, obsolete contribution-provider tier, generic Flow
production/compatibility scaffold, or forwarding shim remains. Versions and
changelogs are `0.4.2`/current.

Here "compatibility shim" excludes explicitly owned versioned durable decoders
and negotiated public-protocol compatibility. It forbids temporary legacy
extension/runtime forwarding paths.
