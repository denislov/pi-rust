# Runtime Integrity And Ownership Framework Plan For 0.4.0

## Status And Scope

Target workspace version: **`0.4.0`**.

Status: **in progress since 2026-07-18; the roadmap `0.3.1` train-entry gate,
`RIF-001`, `RIF-005`, `RIF-006`, and `RIF-007` are complete; `RIF-008` is in
progress**.

This is the first release in the
[`0.4.x` architecture and extension-platform train](0.4.x-architecture-extension-platform-roadmap.md).
It repairs operation lifecycle correctness and establishes the owner/framework
seams that every later release consumes. It does not implement the TypeScript/Wasm
extension runtime or Workbench.

Completion requires every `RIF-*`, `REL-040-*`, and `RIF-D*` row to close, all
workspace packages to advance to `0.4.0`, owning changelogs to contain `0.4.0`
entries, and all release gates to pass.

## Objectives

- one admitted root operation has one immutable identity across execution,
  transactions, SessionEvents, ProductEvents, outcomes, snapshots, and adapters;
- ProductEvent-policy roots resolve to exactly one authoritative terminal;
- uncertain commits enter supervisor-owned `RecoveryPending` without a
  speculative terminal;
- projection fails closed on unknown root identities;
- normative architecture, current-state evidence, ADRs, and migration history are
  separated;
- one operation descriptor supplies terminal policy and orthogonal resource
  claims;
- RuntimeHost is a composition root over explicit owners, not a service locator;
- each persistent session has one bounded logical writer;
- durable SessionEvents, ProductEvent outbox records, committed projections, and
  snapshot cursors share a proven consistency point;
- runtime/session performance baselines are captured before later structural work.

## Non-Goals

- Migrating the Agent production loop away from Flow; that belongs to `0.4.1`.
- Adding Manifest v2, WIT, TypeScript, Wasm, or new extension capabilities; those
  begin in `0.4.2`.
- Removing Lua in this release. Lua remains frozen and receives no wider Host API;
  it is replaced and deleted in `0.4.2`.
- Implementing Workbench views or Application Profiles.
- Turning ProductEvents or snapshots into durable session truth.
- Executing provider, tool, extension, publication, or client work inside the
  session writer.
- Advancing later `0.4.x` versions or changelogs during this release.

## Fixed Architecture Results

### Lifecycle State

```text
Admitted -> Running -> Completed | Failed | Aborted
                    -> RecoveryPending -> Recovering
                                       -> Completed | Failed | Aborted
```

`RecoveryPending` and `Recovering` are durable non-terminal runtime states. The
operation supervisor retains ownership across caller exit and restart. Recovery
resolves the original execution to one idempotent terminal. Operator resolution
cannot select normal completion without durable evidence.

Entering `RecoveryPending` returns a typed non-terminal status/receipt to callers
that cannot remain attached. `ADR-011` defines how that receipt maps to Rust,
ProductEvent, Snapshot, RPC, JSON, and reconnect semantics; it is not represented
as a contradictory terminal `RecoveryRequired` outcome.

### Owner Graph

```text
Adapter
  -> OperationSupervisor
  -> Typed Workflow
  -> SessionCoordinator
  -> committed outbox
  -> OutboxPublisher
  -> EventHub
  -> ClientProjectionCoordinator
```

- `RuntimeHost`: construction, lifetime, shutdown coordination;
- `OperationSupervisor`: admission, identity, permits, cancellation, terminal,
  recovery ownership;
- `SessionCoordinator`: per-session writer, transaction, outbox, committed read
  model, snapshot cursor;
- `OutboxPublisher`: committed delivery obligation only;
- `EventHub`: fan-out/pressure/gap only;
- `ClientProjectionCoordinator`: committed projection and transient overlays only.

EventHub cannot mutate sessions; projection cannot repair facts; adapters and
workflows cannot publish authoritative terminals; SessionCoordinator cannot
decide business outcomes.

The supervisor freezes an immutable `FinalizationDecision`/`CommitIntent` and
submits it to the SessionCoordinator. The coordinator validates and persists the
decision with SessionEvents and outbox obligations, returning `Committed`,
`DefinitelyFailed`, or `InDoubt(recovery_id)`. Only the supervisor maps that
result to a terminal or `RecoveryPending`; this handoff is part of `ADR-001` and
`ADR-011`, not an implementation-local callback convention.

### Session Consistency

One persistent session uses one bounded Tokio command actor or ADR-approved
equivalent. Different sessions remain concurrent. Provider/tool/extension calls,
outbox delivery, and slow clients execute outside the writer.

```text
stage SessionEvents + committed ProductEvent outbox
  -> append/commit or persist RecoveryPending evidence
  -> advance committed projection and cursor
  -> publish outbox asynchronously
  -> recover committed unpublished records after restart
```

## Progress Record

- The `0.3.1` entry baseline is tag `v0.3.1` at source commit `870d4bb`; its dated
  release record is commit `180f219`.
- `RIF-005` split the monolithic architecture into a concise index, normative
  principles/runtime/extension/dependency/testing contracts, version-stamped
  current state, an ADR register, and migration records. The former document is
  preserved as non-normative `0.3.1` migration evidence.
- `RIF-006` accepted `ADR-001` through `ADR-005` and `ADR-011`. The
  zero-dependency contract prototype records capability, state/fact, activation,
  Workbench revision/resync, and client-local state evidence. Locked Wasmtime and
  TypeScript/Jco fixtures additionally prove real-engine Store isolation,
  memory/fuel/epoch/deadline behavior, disposal after cancellation, and typed WIT
  Component generation. `ADR-006` through `ADR-010` and `ADR-012` have owners,
  deadlines, required evidence, and blocked tasks.
- Commit `57e6a17` converged PluginLoad identity and Completed/Failed/Aborted
  terminal evidence already present in the working baseline. It is partial
  `RIF-003` evidence only; `RIF-003` remains planned behind `RIF-002`/`RIF-009`.
- `RIF-001` replaced the admission data bag with an immutable
  `OperationExecution` held by the permit. Root execution freezes descriptor
  revision, origin, capability generation, admitted session identity, and root
  lineage; child permits now retain both parent and resolved root lineage.
  PluginLoad, SelfHealingEdit, and BranchSummary durable transactions now consume
  the admitted capability snapshot identity, while Agent/Team contexts receive
  their execution identity at construction and request nested identities from
  the scheduler. Root, child, session-copy correlation, and recovery IDs now use
  explicit allocator APIs; delegation approval persistence reuses the admitted
  approval identity instead of generating another root. Submission commit,
  terminal association, outcome acknowledgement, and abandoned-submission
  cleanup retain the admitted `OperationExecution` rather than detached identity
  and descriptor copies. Allocator-ownership and dispatch boundary tests close
  the task and `RIF-D001`.
- `RIF-007` completed one exhaustive `OperationContract` descriptor table for all
  16 public operations. Internal payload variants map only to a contract key;
  metadata, capability session access, class, and dispatch are derived. The
  descriptor separates revision, lineage, session/runtime access, priority,
  capacity, durability, cancellation, child, outcome, and terminal claims and
  rejects conflicting combinations. Scheduler admission and capability session
  access consume those claims directly; structured child admission is permitted
  only by Prompt/Agent/Team descriptors. The transitional `OperationMetadata`
  projection and its stale compile-fail fixtures were removed.
- `RIF-008` has begun by reducing `CodingAgentSession` to a facade over one
  `RuntimeHost`. Mutable authority is physically grouped under
  `OperationSupervisor`, `SessionCoordinator`, `EventHub`, and
  `ClientProjectionCoordinator`; workflow sources cannot name `RuntimeHost`.
  The first identity-bearing writer command/reply protocol owns
  SetDefaultAgentProfile, ForkSession, SwitchActiveLeaf, and SetSessionTreeLabel
  mutation, including atomic replacement of forked persistence and replay-derived
  owner state. Delegation approval/rejection now atomically persist their fact
  and update the pending queue through the same writer protocol; child execution
  and EventHub publication stay outside the writer. The bounded writer transport
  and migration of the remaining mutation families are still open, so neither
  `RIF-008` nor `RIF-D007` is complete.

## Detailed Acceptance

Task-ledger row order is the implementation order. Detailed sections remain in
stable numeric ID order for traceability to former task/debt inventories.

### `RIF-001`: Runtime-Owned Identity

- Admission creates one immutable root `OperationExecution`.
- Permit, dispatch, cancellation, transaction, events, outcomes, projection, and
  child admission consume the same root identity.
- Child IDs retain stable parent/root lineage.
- Workflows, services, adapters, and plugins cannot generate or replace an
  admitted root ID.
- Boundary tests reject standalone root-ID generators on admitted paths.

### `RIF-002`: Terminal, Recovery, And Projection

- ProductEvent-policy success/failure/abort paths publish one terminal.
- Definite persistence failure resolves Failed; cancellation resolves Aborted.
- Uncertain commit persists recovery evidence, enters `RecoveryPending`, and
  publishes no speculative terminal.
- Unknown local-event operation IDs cannot create roots.
- Only registered root-start/terminal/recovery evidence changes root lifecycle.
- Startup scan, bounded retry/backoff, authenticated inspect/retry/resolve, audit,
  and subsequent-work policy implement `ADR-011`.
- Rust/API/RPC/JSON/Snapshot mappings distinguish non-terminal
  `RecoveryPending` receipts from authoritative terminal outcomes.

### `RIF-003`: Operation-Family Convergence

- Prompt, Compact, BranchSummary, SelfHealingEdit, and PluginLoad persistent
  transactions reuse admitted IDs.
- PluginLoad and SelfHealingEdit cover validation, Flow failure, cancellation,
  record/commit failure, capability update, and uncertain commit paths.
- BranchSummary defines Completed/Failed/Aborted without borrowing Prompt
  terminals.
- Every public operation records terminal policy, lineage, session/runtime access,
  priority, capacity, durability, and scheduling inputs.
- Submitted-task, compatibility run, interactive, RPC, and direct API paths have
  identical lifecycle semantics.

### `RIF-004`: Lifecycle Matrix

The parameterized matrix covers every public operation and applicable success,
validation, provider/Flow failure, cancellation before/during/commit, definite
persistence failure, uncertainty/recovery, dropped ownership, persistent/
transient session, and adapter path. It asserts:

```text
one admitted operation_id across durable facts/events/outcomes
resolved ProductEvent policy => one terminal_operation
RecoveryPending => zero terminal_operation + supervisor ownership
OutcomeAcknowledgement => zero synthetic terminal_operation
unknown operation_id => no projected root
completed case => no running/recovery-pending root
one authoritative descriptor row per operation
```

### `RIF-006`: Decision Gate

- Accept implementation-blocking `ADR-001` and `ADR-011`, including the typed
  finalization handoff, recovery evidence store, restart ownership, operator
  controls, and public non-terminal status mapping, before `RIF-001`/`RIF-002`.
- Accept `ADR-002` through `ADR-005` only after their offline disposable
  prototypes validate the selected train-wide design; no production extension
  runtime ships in `0.4.0`.
- Schedule/own `ADR-006` through `ADR-010` and `ADR-012` before their roadmap
  release gates.
- Add ADR-to-task/test/evidence traceability.
- A decision cannot be accepted after a dependent implementation has already
  fixed its outcome.

### `RIF-007`: Descriptor And Claims

- Extend the lifecycle descriptor in place; do not create another registry.
- Separate session/runtime/lineage/priority/capacity/durability/terminal claims.
- Derive scheduler and capability requirements from descriptors.
- Reject invalid combinations and drift through exhaustive table tests.
- Transitional classes remain derived projections and have removal inventories.

### `RIF-008`: Runtime And Session Owners

- Implement the `ADR-001` allowed call graph and prohibited-edge tests.
- RuntimeHost is not passed as a mutable service locator.
- Every persistent session mutation uses its bounded writer.
- Independent sessions and non-session roots execute concurrently.
- Provider/tool/extension/publication/client work occurs outside the writer.
- Shutdown closes admission, drains owned work, preserves recovery evidence, and
  terminates deterministically.

### `RIF-009`: Outbox And Snapshot Consistency

- SessionEvents and durable-derived outbox records commit in one authoritative
  batch or persist uncertainty evidence.
- The supervisor supplies an immutable finalization decision; the coordinator
  validates and commits it but never re-decides the business outcome.
- Terminal and outbox records have stable semantic identities for idempotent
  replay/redelivery.
- Committed unpublished records recover after restart with semantic idempotence.
- Live optimistic output cannot enter committed history.
- Snapshot read model and event/session cursors are captured at one consistency
  point.
- Prove `UiState(N) = Snapshot(N) + committed ProductEvents after N` for normal,
  concurrent, lagged, reconnect, commit-failure, and restart cases.
- Slow subscribers remain off the commit path; pressure/gap behavior is explicit.

### `RIF-010`: Baselines

Record reproducible offline evidence for admission, independent-session
concurrency, writer queue pressure, session commit, outbox, snapshot capture,
reconnect, recovery scan, and no-extension runtime overhead. Baselines seed
`ADR-009`; timing evidence is separated from deterministic correctness tests.
Record provisional no-major-regression thresholds for each measured path and
fail the release on an unexplained architectural regression.

### `REL-040-001`: Release Convergence

- Close all `RIF-D*` debt and new debt assigned to `0.4.0`.
- Decide SessionEvent, ProductEvent, Snapshot, RPC, and public Rust API impact.
- Update all workspace package versions to `0.4.0` only after implementation and
  debt closure.
- Add root and owning-crate `0.4.0` changelog sections.
- Update public API snapshots, protocol inventories, architecture evidence, and
  migration notes.
- Run focused and full release validation.

interactive, API-boundary, descriptor, owner-edge, writer-pressure, outbox, and


## Release Gates

`0.4.0` is complete only when:

1. every task and debt row is closed;
2. every durable workflow uses one admitted identity;
3. resolved ProductEvent roots have one terminal;
4. intentional unresolved fixtures retain explicit recovery ownership;
5. projection leaves no unintended running root;
6. architecture documents and accepted ADRs match source;
7. one descriptor drives admission, durability, terminal, and projections;
8. runtime owners have no prohibited cycle;
9. different sessions remain concurrent and writer critical sections contain no
   provider/tool/extension/client work;
10. outbox/recovery/snapshot matrices pass;
11. versions, changelogs, snapshots, protocols, formatting, Clippy, tests, and TUI
    smoke are complete.

The `0.3.1` train-entry baseline, finalization handoff, recovery receipt mapping,
and provisional performance guard are explicit parts of these gates.
