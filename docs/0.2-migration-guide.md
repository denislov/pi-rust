# Migrating `pi-rust` from 0.1 to 0.2

`0.2` is a breaking architecture release. The migration goal is not merely to rename imports: consumers must use scoped provider clients, canonical coding-agent operations, typed product events, and complete snapshot cursors.

See also [0.1-to-0.2-api-diff.md](./0.1-to-0.2-api-diff.md) for the inventory-level diff.

## 1. Move imports to stable facades

### `pi-ai`

```rust
// Before
use pi_ai::registry::{AiClient, ApiProvider, ProviderRegistry};
use pi_ai::types::{ContentBlock, Model};

// After
use pi_ai::api::{AiClient, ApiProvider, ContentBlock, Model, ProviderRegistry};
```

Do not recreate a process-global provider registry. Construct or receive a scoped `AiClient`, register providers on it, and carry that client in operation/runtime configuration.

### `pi-agent-core`

```rust
// Before
use pi_agent_core::flow::{Flow, FlowRunOptions};
use pi_agent_core::types::{AgentEvent, AgentTool};
use pi_agent_core::transcript::StoredAgentMessage;

// After
use pi_agent_core::api::{
    AgentEvent, AgentTool, Flow, FlowRunOptions, StoredAgentMessage,
};
```

The old doc-hidden modules are private. If a symbol is not in `pi_agent_core::api`, treat it as an implementation detail rather than importing another internal path.

### `pi-coding-agent`

```rust
// Before
use pi_coding_agent::{CliArgs, CliError, CliRunOptions, run_cli};

// After
use pi_coding_agent::api::{CliArgs, CliError, CliRunOptions, run_cli};
```

Root compatibility re-exports have been deleted. Hidden implementation modules are not a replacement stable surface.

The old `args`, `config`, `error`, `input`, `models`, `print_mode`, `prompt_options`, `protocol`, `request`, `resources`, `runtime`, `session`, and `theme` module paths are private in 0.2. Import their supported contracts directly from `pi_coding_agent::api`; do not change a deleted root re-export into a doc-hidden module import.

## 2. Use a scoped provider client

```rust
use std::sync::Arc;
use pi_ai::api::{AiClient, ApiProvider};

fn configured_client(provider: Arc<dyn ApiProvider>) -> AiClient {
    let client = AiClient::new();
    client.register_provider("example", provider);
    client
}
```

Pass the client through the supported runtime/session options. Do not mutate global provider state in tests or production code. Tests should create independent clients so parallel cases cannot affect each other.

## 3. Submit work through `CodingAgentOperation`

Use the stable session facade and canonical operation enum:

```rust
use pi_coding_agent::api::{
    CodingAgentOperation, CodingAgentOperationOutcome, CodingAgentSession,
};

async fn reload_plugins(session: &mut CodingAgentSession) -> anyhow::Result<()> {
    match session.run(CodingAgentOperation::PluginLoad).await? {
        CodingAgentOperationOutcome::PluginLoad(outcome) => {
            println!("loaded: {:?}", outcome.loaded_plugin_ids);
        }
        other => anyhow::bail!("unexpected outcome: {other:?}"),
    }
    Ok(())
}
```

Do not call internal Flow services, session services, runtime services, or admission controls. The canonical dispatcher owns scheduler admission, operation identity, frozen capability generation, terminal association, and transaction finalization.

## 4. Migrate product-event consumers to typed matching

```rust
use pi_coding_agent::api::{
    CodingAgentProductEvent, CodingAgentProductEventKind,
    CodingAgentWorkflowProductEvent,
};

fn observe(event: &CodingAgentProductEvent) {
    match event.event() {
        CodingAgentProductEventKind::Workflow(
            CodingAgentWorkflowProductEvent::PromptCompleted { .. },
        ) => println!("prompt completed"),
        _ => {}
    }
}
```

Replace usages of removed fields:

```rust
// Removed
event.family
event.kind

// Available for typed routing or display/metrics
event.family_typed()
event.kind_name()
event.event()
```

Do not parse strings such as `"Workflow(PromptCompleted)"`; those compatibility strings are gone.

## 5. Carry the complete snapshot cursor

Always retain the cursor supplied by `CodingAgentSnapshot`:

```rust
let snapshot = connection.state()?;
let cursor = snapshot.cursor.clone();

match connection.reconnect_from_cursor(&cursor)? {
    // handle replay or fresh-snapshot-required
    reconnect => consume(reconnect),
}
```

Do not manufacture a cursor from only the last sequence. A cursor identifies:

- the session stream;
- the snapshot protocol major;
- the event sequence;
- the capability generation.

This prevents replaying a valid sequence against the wrong session or protocol family.

## 6. Update RPC prompt requests

Before:

```json
{
  "type": "prompt",
  "id": "req-1",
  "message": "hello",
  "afterSnapshotSequence": 42
}
```

After:

```json
{
  "type": "prompt",
  "id": "req-1",
  "message": "hello",
  "afterSnapshotCursor": {
    "streamId": "session-id",
    "snapshotProtocolMajor": 2,
    "lastEventSequence": 42,
    "capabilityGeneration": 7
  }
}
```

Use the values received from the server snapshot. The `2` shown above is the release target; do not hard-code it while consuming development builds that still advertise major `1` before M7.

## 7. Update submitted-operation state handling

Remove branches for:

```rust
CodingAgentSubmittedOperationStatus::Accepted
```

The externally visible state begins at `Running` after the canonical atomic admission/commit boundary. Continue handling terminal status and uncertainty explicitly.

## 8. Update capability-revocation handling

Remove branches for:

```rust
CodingAgentProductEventCapabilityRevocation::CancelMatchingOperations
```

Only `FutureOnly` is currently implemented. Active operations retain the frozen generation they were admitted with. Do not assume a capability update silently mutates or cancels an active operation.

## 9. Do not depend on arbitrary plugin Flow extensions

The 0.1 internal extension model had no runtime consumer and conflicted with the architecture Non-Goal forbidding Lua plugins from registering arbitrary Flow nodes/subflows. The following internal concepts were removed:

- `FlowExtension`;
- `FlowExtensionPoint`;
- `FlowExtensionRegistrationHost`;
- flow-extension registry/capability counters.

Use supported plugin tools, commands, hooks, UI actions/dialogs, and keybindings. A future restricted extension point will require a new explicit contract; there is no compatibility adapter for the deleted model.

## 10. Error handling changes

Remove handling for the deleted `CodingSessionError::StaleClientConnection`. Client takeover and stale handles are expressed through the canonical lifecycle/generation rejection path.

Continue treating event gaps and fresh-snapshot-required recovery as normal reconnect outcomes rather than generic transport failures.

## 11. Suggested migration order

1. Change all imports to the three stable `api` facades.
2. Replace global provider setup with scoped `AiClient` construction/injection.
3. Route coding-agent work through `CodingAgentSession::run(CodingAgentOperation)`.
4. Replace event string inspection with typed matching.
5. Store and transmit the complete snapshot cursor.
6. Update RPC request fixtures and clients.
7. Remove deleted enum/error branches.
8. Run compile-fail/API boundary tests before adapting application behavior.
9. Run adapter protocol tests and reconnect/overflow scenarios.

## 12. Validation checklist

For an embedding crate or application:

```bash
cargo check
cargo test
```

For this workspace before the 0.2 release:

```bash
cargo fmt --all --check
cargo clippy --workspace --all-targets --all-features -- -D warnings
cargo test --workspace --all-features
```

Also verify:

- no imports bypass `pi_ai::api`, `pi_agent_core::api`, or `pi_coding_agent::api` for supported contracts;
- no adapter imports Flow/session/runtime/plugin services;
- no RPC request uses `afterSnapshotSequence`;
- no consumer expects top-level product-event `family`/`kind` strings;
- reconnect tests cover wrong stream, wrong snapshot major, retained replay, and fresh snapshot recovery;
- provider tests use scoped fake clients and do not require live credentials.

## Known release-train boundary

The migration contract is drafted before the M7 version bump. At the time of writing, crate manifests and protocol constants still report `0.1.0`/major `1`. M7 must atomically bump versions, refresh fixtures, and rerun the full release validation matrix before publishing `0.2.0-alpha.1`.
