# Migrating `pi-rust` from 0.1 to 0.2

`0.2` is a breaking architecture release. The migration goal is not merely to rename imports: consumers must use scoped provider clients, canonical coding-agent operations, typed product events, and complete snapshot cursors.

See also [0.1-to-0.2-api-diff.md](./0.1-to-0.2-api-diff.md) for the inventory-level diff.

## 1. Move imports to stable facades

### `pi-ai`

```rust
// Before
use pi_ai::registry::{AiClient, ApiProvider, ProviderRegistry};
use pi_ai::types::{ContentBlock, Model};

// After
use pi_ai::api::client::AiClient;
use pi_ai::api::conversation::ContentBlock;
use pi_ai::api::model::Model;
use pi_ai::api::provider::{ApiProvider, ProviderRegistry};
```

Do not recreate a process-global provider registry. Construct or receive a scoped `AiClient`, register providers on it, and carry that client in operation/runtime configuration.

### `pi-agent-core`

```rust
// Before
use pi_agent_core::flow::{Flow, FlowRunOptions};
use pi_agent_core::types::{AgentEvent, AgentTool};
use pi_agent_core::transcript::StoredAgentMessage;

// After
use pi_agent_core::api::agent::AgentEvent;
use pi_agent_core::api::flow::{Flow, FlowRunOptions};
use pi_agent_core::api::tool::AgentTool;
use pi_agent_core::api::transcript::StoredAgentMessage;
```

The old doc-hidden modules and the flat `api` exports are gone. If a symbol is
not in an `agent`, `tool`, `flow`, `execution`, `resources`, `compaction`, or
`transcript` category, treat it as an implementation detail. `api::testing` is
test support, not a production dependency surface.

Tool callbacks now receive an invocation context:

```rust
use pi_agent_core::api::tool::AgentTool;

let tool = AgentTool::new_text(
    "echo",
    "Echo input",
    serde_json::json!({"type": "object"}),
    |context, args| async move {
        if context.cancel_token().is_cancelled() {
            return Err("cancelled".into());
        }
        Ok(format!("{}: {args}", context.tool_call_id()))
    },
);
```

The context also exposes the generic execution scope, turn, and tool name. In
`pi-coding-agent` the scope is the admitted operation ID. Direct Rust closures
remain trusted embedding code; untrusted extensions must use the product plugin
host rather than capture host services in an `AgentTool` closure.

### `pi-coding-agent`

```rust
// Before
use pi_coding_agent::{CliArgs, CliError, CliRunOptions, run_cli};

// After
use pi_coding_agent::api::cli::command::{CliArgs, CliError};
use pi_coding_agent::api::cli::runner::run_cli;
use pi_coding_agent::api::cli::runtime::CliRunOptions;
```

Root compatibility re-exports have been deleted. Hidden implementation modules are not a replacement stable surface.

All old `pi_coding_agent` doc-hidden root modules are private in 0.2, including
`args`, `config`, `error`, `input`, `interactive`, `models`, `print_mode`,
`prompt_options`, `protocol`, `request`, `resources`, `runtime`, `session`,
`theme`, and `tools`. Flat `pi_coding_agent::api::{...}` imports are also
deleted. Choose one of `runtime`, `operation`, `event`, `client`, `view`,
`protocol`, or the explicitly scripting-oriented `cli` category. The
`testing` category is reserved for feature-gated deterministic support. The
`cli` category is itself divided into `command`, `configuration`, `input`,
`print`, `resources`, `runtime`, `runner`, and `theme`; it has no flat exports.

## 2. Use a scoped provider client

```rust
use std::sync::Arc;
use pi_ai::api::client::AiClient;
use pi_ai::api::provider::ApiProvider;

fn configured_client(provider: Arc<dyn ApiProvider>) -> AiClient {
    let client = AiClient::new();
    client.register_provider("example", provider);
    client
}
```

Pass the client through the supported runtime/session options. Do not mutate global provider state in tests or production code. Tests should create independent clients so parallel cases cannot affect each other.

No crate's `api::testing` category is available to normal dependency builds. A
downstream test crate that needs fixtures such as `FauxProvider`, an agent
harness, or `VirtualTerminal` must opt in explicitly on the owning dependency:

```toml
[dev-dependencies]
pi-ai = { path = "../pi-ai", features = ["test-support"] }
```

Do not enable `test-support` in a production dependency declaration.

## 3. Submit work through `CodingAgentOperation`

Use the stable session facade and canonical operation enum:

```rust
use pi_coding_agent::api::operation::{
    CodingAgentOperation, CodingAgentOperationOutcome,
};
use pi_coding_agent::api::runtime::CodingAgentSession;

async fn reload_plugins(session: &mut CodingAgentSession) -> anyhow::Result<()> {
    match session.run(CodingAgentOperation::PluginLoad).await? {
        CodingAgentOperationOutcome::PluginLoad(outcome) => {
            println!("loaded: {:?}", outcome.loaded_plugin_ids);
        }
        other => anyhow::bail!("unexpected outcome: {other:?}"),
    }
    Ok(())
}
```

Do not call internal Flow services, session services, runtime services, or admission controls. The canonical dispatcher owns scheduler admission, operation identity, frozen capability generation, terminal association, and transaction finalization.

## 4. Migrate product-event consumers to typed matching

```rust
use pi_coding_agent::api::event::{
    CodingAgentProductEvent, CodingAgentProductEventKind,
    CodingAgentWorkflowProductEvent,
};

fn observe(event: &CodingAgentProductEvent) {
    let sequence = event.sequence();
    match event.event() {
        CodingAgentProductEventKind::Workflow(
            CodingAgentWorkflowProductEvent::PromptCompleted { .. },
        ) => println!("prompt completed"),
        _ => {}
    }
}
```

The event envelope is opaque outside the crate. Read metadata through
`sequence()`, `operation_id()`, `capability_generation()`,
`parent_operation_id()`, `root_operation_id()`, `session_id()`,
`terminal_operation()`, and `durability()` rather than struct fields. The same
`CodingAgentProductEvent` value is used for live subscription and reconnect
replay; there is no separate internal event envelope to decode.

`terminal_status()` and `terminal_operation()` answer different questions. A
tool or session-write event can have a local terminal status while
`terminal_operation()` remains `None`. Root terminal metadata is present only
when the event is authorized by the admitted operation descriptor; consumers
must not infer root completion from a payload variant such as
`PromptFailed` alone, because Compact failure uses that payload as well.

Durability consumers must handle `DerivedFromSession`, `PersistenceUncertain`,
and `PersistenceFailed` in addition to live, pending, and durable states.
Recovery events are derived from a committed recovery marker and include the
source operation/recovery identity. `WriteFailed(Uncertain)` requires recovery
or refresh; `WriteFailed(Definite)` means the owner established that no commit
occurred. Neither may be treated as `WriteSkipped`.

Use `delivery_class()` to distinguish data, authoritative root terminal,
control projection, and recovery events. A lag or retained-history gap is
fail-closed for every class: request a fresh snapshot instead of selectively
applying terminal or recovery events after a missing prefix.

Replace usages of removed fields:

```rust
// Removed
event.family
event.kind

// Available for typed routing or display/metrics
event.family_typed()
event.kind_name()
event.event()
event.capability_generation()
```

Do not parse strings such as `"Workflow(PromptCompleted)"`; those compatibility strings are gone.

For auditing or correlation, use `event.capability_generation()` as the
operation's admitted generation. Do not compare an in-flight event against only
the latest snapshot cursor generation: future-only updates deliberately allow
older admitted operations to finish. The field is optional for global events
and legacy recovery records that did not persist a generation.

## 5. Carry the complete snapshot cursor

Always retain the cursor supplied by `CodingAgentSnapshot`:

```rust
let snapshot = connection.state()?;
let cursor = snapshot.cursor.clone();

match connection.reconnect_from_cursor(&cursor)? {
    // handle replay or fresh-snapshot-required
    reconnect => consume(reconnect),
}
```

Do not manufacture a cursor from only the last sequence. A cursor identifies:

- the session stream;
- the snapshot protocol major;
- the event sequence;
- the capability generation.

This prevents replaying a valid sequence against the wrong session or protocol family.

## 6. Update RPC prompt requests

Before:

```json
{
  "type": "prompt",
  "id": "req-1",
  "message": "hello",
  "afterSnapshotSequence": 42
}
```

After:

```json
{
  "type": "prompt",
  "id": "req-1",
  "message": "hello",
  "afterSnapshotCursor": {
    "streamId": "session-id",
    "snapshotProtocolMajor": 2,
    "lastEventSequence": 42,
    "capabilityGeneration": 7
  }
}
```

Use the values received from the server snapshot. `0.2.0` advertises
major `2`, but consumers must not hard-code it.

## 7. Update submitted-operation state handling

Replace the removed, previously unimplemented branch:

```rust
CodingAgentSubmittedOperationStatus::Accepted
```

The externally visible state begins at `Running` after the canonical atomic admission/commit boundary. Continue handling terminal status and uncertainty explicitly.

## 8. Update capability-revocation handling

Remove branches for:

```rust
CodingAgentProductEventCapabilityRevocation::CancelMatchingOperations
```

Normal configuration and plugin updates still emit `FutureOnly`; active
operations retain the snapshot frozen at admission. A privileged embedding host
may instead call:

```rust
let outcome = session
    .capability_control()
    .revoke_older_operations();
```

This installs a new generation, rejects later admission of stale snapshots, and
emits `RequestCancelOlderOperations` with
`cancellation_requested_operation_ids`. Cancellation is cooperative: Prompt,
child agent/team work, compaction, and owned long-running tools observe their
tokens, while arbitrary trusted Rust closures cannot be forcibly preempted.

## 9. Do not depend on arbitrary plugin Flow extensions

The 0.1 internal extension model had no runtime consumer and conflicted with the architecture Non-Goal forbidding Lua plugins from registering arbitrary Flow nodes/subflows. The following internal concepts were removed:

- `FlowExtension`;
- `FlowExtensionPoint`;
- `FlowExtensionRegistrationHost`;
- flow-extension registry/capability counters.

Use supported plugin tools, commands, hooks, UI actions/dialogs, and keybindings. A future restricted extension point will require a new explicit contract; there is no compatibility adapter for the deleted model.

## 10. Error handling changes

Remove handling for the deleted `CodingSessionError::StaleClientConnection`. Client takeover and stale handles are expressed through the canonical lifecycle/generation rejection path.

Continue treating event gaps and fresh-snapshot-required recovery as normal reconnect outcomes rather than generic transport failures.

## 11. Suggested migration order

1. Change all imports to categorized paths under the four active `api` facades.
2. Replace global provider setup with scoped `AiClient` construction/injection.
3. Route coding-agent work through `CodingAgentSession::run(CodingAgentOperation)`;
   use `CodingAgentSession::submit` plus `CodingAgentOperationTask::join` for
   runtime-owned AgentInvocation/AgentTeam work. The former PluginCommand API
   was removed in 0.4.2; contribution dispatch is not part of the minimum Wasm
   framework and must not be emulated through an adapter compatibility command.
4. Replace event string inspection with typed matching.
   Preserve `operationId`/`childOperationId` lineage exactly as emitted. In 0.2,
   delegated Agent/Team wrappers and their internal Prompt executions have
   distinct registered IDs, and parent terminal delivery implies that the
   descendant tree has drained or become a durable pending request.
5. Store and transmit the complete snapshot cursor.
6. Update RPC request fixtures and clients.
   Treat successful `invoke_agent`/`invoke_team` responses as background
   admission receipts, retain `data.operationId` for correlation, and do not
   infer that Prompt control targets those roots. A later Prompt can run while
   they remain active; session replacement and shutdown still wait for all
   active operation IDs to drain. For reconnect, use
   `get_state.data.eventStreamId` as the snapshot cursor `streamId`; do not use
   `sessionId`, because session identity and runtime event-stream identity are
   intentionally independent. A state without `eventStreamId` has no live
   stream to reconnect.
   Submitted-operation terminal state also has two explicit policies. For a
   `ProductEvent` terminal, acknowledge the event sequence. For an
   `OutcomeAcknowledgement` terminal, retain the returned acknowledgement ID
   and call `acknowledge_outcome`; do not wait for a synthetic ProductEvent.
7. Remove deleted enum/error branches.

Do not submit `ApproveDelegation` or `RejectDelegation` as background
non-session/control work. They remain session-writing `run` operations because
the decision and any newly adopted pending confirmations are durable session
facts. Callers must handle `Busy` when Prompt or another session writer is
active, then retry with the same pending operation/tool identifiers after the
writer finishes.
8. Run compile-fail/API boundary tests before adapting application behavior.
9. Run adapter protocol tests and reconnect/overflow scenarios.

## 12. Validation checklist

For an embedding crate or application:

```bash
cargo check
cargo test
```

For this workspace before the 0.2 release:

```bash
cargo fmt --all --check
cargo clippy --workspace --all-targets --all-features -- -D warnings
cargo test --workspace --all-features
```

Also verify:

- no flat facade imports or imports that bypass a categorized `api` path;
- no adapter imports Flow/session/runtime/plugin services;
- no RPC request uses `afterSnapshotSequence`;
- no consumer expects top-level product-event `family`/`kind` strings;
- reconnect tests cover wrong stream, wrong snapshot major, retained replay, and fresh snapshot recovery;
- provider tests use scoped fake clients and do not require live credentials.

## Release boundary

The P7 release bump was applied atomically: all workspace packages report
`0.2.0`, and RPC/ProductEvent/UI Snapshot advertise `2.0`. Durable session
writer version remains `1`. Machine-generated public API snapshots were
generated and reviewed through `scripts/release-api-snapshots.sh`; checked-in
freeze manifests live under `docs/api-snapshots/`.

The final convergence and release audit is recorded in
[`0.2-architecture-convergence-record.md`](0.2-architecture-convergence-record.md).
