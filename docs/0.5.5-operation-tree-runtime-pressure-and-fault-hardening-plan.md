# Operation-Tree Runtime Pressure And Fault Hardening Plan For 0.5.5

## Status And Scope

Target workspace version: **`0.5.5`**. Status: **Completed 2026-07-22**.
Implementation starts from the completed `0.5.4` release at commit `d32367d`.
Package, lockfile, binary, and release changelog versions advance together in
`RPF-009` after `RPF-001` through `RPF-008` close; the plan becomes Completed
only after release evidence and the full gate set pass.

This plan hardens the operation tree introduced and exercised by the 0.5.4
awaited delegation runtime. Its primary concern is not adding delegation or TUI
features. It proves that parent Prompt operations, delegation tool futures,
authorization waiters, structured child operations, provider streams, session
writes, ProductEvents, snapshots, and adapters converge under cancellation,
pressure, disconnect, persistence uncertainty, and shutdown.

The plan is subordinate to the normative contracts in
[`architecture/principles.md`](architecture/principles.md),
[`architecture/dependency-rules.md`](architecture/dependency-rules.md),
[`architecture/runtime.md`](architecture/runtime.md), and
[`architecture/testing.md`](architecture/testing.md). It cannot weaken
exactly-once terminal authority, durable recovery ownership, immutable
operation lineage, generation-scoped capabilities, bounded delivery, or
adapter isolation.

## Motivation

0.5.4 corrected the delegation lifetime:

```text
parent Prompt
  -> delegation tool future
       -> optional delegation authorization waiter
       -> structured child Prompt
            -> provider stream
            -> optional child-tool authorization waiter
            -> child terminal
       -> one terminal DelegationToolResult
  -> parent model continuation
```

The deterministic happy-path, rejection, recursive-depth, child authorization,
lineage routing, bounded child projection, reconnect hydration, and fullscreen
navigation matrices pass. The remaining risk is cross-owner failure
composition. A single logical delegation may be observed simultaneously by the
scheduler, cancellation tree, authorization registry, provider stream, session
writer, outbox, EventHub, client connection, and fullscreen projection.

The current architecture explicitly records that provider/live pressure and
reconnect stress remain a later hardening concern. The required runtime testing
contract also calls for active cancellation, persistence failure, commit
uncertainty, dropped owners, bounded queue pressure, retention gaps, reconnect,
and shutdown across applicable public operations. 0.5.5 closes that gap for the
operation-tree path before new multi-agent or background-service scope is added.

## Core Failure Invariant

For every admitted operation tree, every injected failure or race must converge
to exactly one of these outcomes:

```text
Committed terminal
  => exactly one authoritative terminal decision
  => at most one parent-visible delegation tool result
  => no running/recovery-pending residue

Definitely failed before durable obligation
  => typed failure/cancellation
  => no false durable completion
  => no orphan waiter or child

In doubt after a durable boundary
  => zero contradictory terminal claims
  => durable RecoveryPending ownership
  => stable operation/root/capability association
```

No adapter, dropped future, late provider event, duplicate authorization
decision, or shutdown branch may manufacture a second terminal or discard a
durable obligation.

## Architectural Decisions

- The existing operation scheduler and cancellation tree remain the sole owner
  of root/child admission and cancellation. Do not add a delegation-specific
  supervisor or second task registry.
- `AuthorizationService` remains the waiter owner. Fault evidence may add
  crate-private/test-only observations, but it must not expose waiter authority
  through the stable facade.
- Provider faults are injected through scoped faux transports/streamers. Tests
  must remain offline and must not add provider behavior to product code.
- Existing SessionWriter, repository, outbox, EventService, and recovery fault
  controls are reused and extended at their owning boundaries. Do not create a
  parallel persistence or event fault framework.
- SessionEvents remain durable business facts; ProductEvents remain typed
  client delivery. A live child completion cannot override an in-doubt durable
  write with a false committed state.
- Retention exhaustion and sequence gaps trigger explicit fresh-snapshot/resync
  behavior. Silent loss, best-effort continuation, or adapter-local repair is
  forbidden.
- Fullscreen child pages continue to use the generic `pi-tui` render/lifecycle
  contracts. Product child-page semantics stay in `pi-coding-agent`.
- Correctness soak schedules use deterministic clocks, barriers, bounded
  channels, and named timeouts. Host-dependent timing and RSS measurements are
  recorded separately as performance evidence.
- The stable embedding facade and protocol versions change only when a required
  contract cannot be expressed additively. Any change requires migration and
  checked-in API/protocol evidence.

## Non-Goals

- Do not add new agent/team features, background delegation, detached child
  execution, or user-visible workflow graph concepts.
- Do not activate Extension contribution productization, Workbench semantic
  views, extension state/facts, update coordination, or background services.
- Do not activate `pi-mom`, `pi-pods`, or `pi-web-ui` placeholder behavior.
- Do not make adapter disconnect implicitly abort an operation; detach and
  abort remain distinct typed controls.
- Do not persist a second full child transcript solely for pressure testing.
- Do not widen queue capacities to make saturation tests pass. Define and test
  explicit pressure behavior at the owning boundary.
- Do not use live providers, credentials, uncontrolled wall-clock sleeps, or
  flaky timing thresholds as release evidence.
- Do not turn reasoned dead-code/Extension decisions into the primary scope of
  this version. The release audit must still reject new unreasoned exceptions.

## Detailed Acceptance

### `RPF-001`: State/Failure Model And Instrumentation

- Enumerate the externally meaningful operation-tree phases: admission,
  delegation authorization wait, child admission/run, child authorization
  wait, child finalization, parent-result delivery, folded-state persistence,
  committed terminal, definite failure, and recovery pending.
- For every phase, identify the sole owner of cancellation, waiter resolution,
  child guard/task lifetime, terminal decision, persistence, event publication,
  and adapter projection.
- Build a table-driven fault matrix that maps every injection point to its legal
  terminal/recovery outcomes and forbidden residues.
- Add crate-private or test-only snapshots for resources that cannot otherwise
  be audited deterministically, such as pending authorization waiters, admitted
  child guards, and queued fault commands. Do not expose mutation authority.
- Record baseline queue capacities, retained windows, operation counts, and
  deterministic test runtimes under `target/perf-baseline/0.5.5-operation-tree/`.
- Add boundary guards preventing a second scheduler, waiter registry, terminal
  owner, or fault-only production fallback.

### `RPF-002`: Cancellation And Shutdown

- Abort a parent while it waits for delegation confirmation; the waiter and
  tool future must resolve once without child admission.
- Abort during child provider streaming; the provider, child task, child guard,
  delegation tool future, and parent Prompt must converge without late output.
- Abort while child tool authorization is open; the authorization request must
  not survive as an actionable orphan.
- Exercise child completion racing parent abort in both orderings. A committed
  completion cannot be rewritten as cancelled, and cancellation that wins
  cannot later accept child completion.
- Exercise runtime shutdown while each wait/run/finalize phase is active.
  Shutdown closes admission, rejects new children, drains or cancels the owned
  tree according to policy, and leaves no detached Tokio task.
- Prove adapter detach does not abort and explicit authorized abort does.
- Assert post-case resource baselines: zero unowned waiters/children, no running
  root, and no contradictory terminal/recovery state.

### `RPF-003`: Authorization Race Hardening

- Cover approve versus deny, approve versus abort, deny versus abort, duplicate
  approve/deny, expired decision, stale capability generation, wrong operation
  ID, and dropped waiter receiver.
- Cover both delegation authorization and a real child Prompt tool
  authorization. Approval of delegation never grants child-tool authority.
- Reconnect an RPC/fullscreen client while authorization is pending; the same
  operation-scoped request must remain resolvable once or become explicit
  recovery state.
- Reopen and race a legacy pre-0.5.4 pending delegation without creating a new
  awaited delegation path or duplicate folded terminal.
- Decisions that lose a race return typed stale/already-resolved errors and do
  not wake an unrelated operation.
- The waiter registry returns to its pre-test count after every case, including
  dropped futures and shutdown.

### `RPF-004`: Provider-Stream Faults

- Provide deterministic scoped faux stream profiles for: pending before first
  event, failure before first event, failure after assistant delta, truncated
  thinking/tool arguments, stall after child tool execution, duplicate terminal,
  late delta after cancellation, and timeout/cancellation races.
- Preserve provider-neutral ownership: provider mapping faults stay in `pi-ai`,
  generic agent-loop consequences stay in `pi-agent-core`, and product
  operation/session/UI consequences stay in `pi-coding-agent`.
- Map each child stream outcome to exactly one typed delegation `failed` or
  `cancelled` result where durable certainty permits.
- Reject late child deltas after terminal/cancellation; they cannot mutate the
  child page, main row, parent transcript, or parent model context.
- Keep print/JSON/RPC stdout protocol-clean and diagnostics redacted.
- Prove partial child output remains bounded and child-scoped before the clear
  terminal/recovery presentation.

### `RPF-005`: Persistence And Outbox Faults

- Inject definite append failure before a delegation folded fact is durable.
- Inject session fact success followed by manifest refresh failure. The live
  result remains in-doubt until authoritative replay; reopen resolves directly
  to committed when terminal facts prove it, otherwise it retains
  RecoveryPending. A resolved commit must not keep recovery residue.
- Inject durable outbox obligation followed by publication interruption; reopen
  must redeliver with semantic idempotence.
- Inject terminal persistence failure for the automatic delegation path and
  for both approval/rejection decision operations. Child failure,
  cancellation, RPC/interactive adapters, and restored legacy decisions share
  those typed Prompt/decision transaction owners; prove the common routing at
  the operation/boundary layer instead of duplicating adapter-specific store
  fault fixtures.
- Saturate or close the bounded SessionWriter command path during terminal
  delivery without blocking async workers indefinitely.
- Exercise shutdown after a terminal write command is admitted but before its
  reply is observed.
- Cross-check event cursor, session sequence, replay state, manifest, outbox,
  folded delegation, public snapshot, and parent result. Any uncertainty must
  retain original operation/root/capability identity under RecoveryPending.
- Reopen/retry/operator-resolution tests prove idempotence and exactly one final
  terminal when recovery completes.

### `RPF-006`: Event, Snapshot, Reconnect, And Bounded Pressure

- Produce child events faster than a client consumes them and prove slow
  subscribers do not block commit or provider execution.
- Cross the retained ProductEvent window during child streaming and require an
  explicit sequence-gap/fresh-snapshot transition.
- Reconnect before child open, during child output, during authorization, after
  child terminal, and between folded-state commit and live delivery.
- Interleave fresh snapshot creation with new live child events and prove
  replay-before-live ordering without duplicate child transcript items.
- Exceed 32 retained child conversations and define deterministic eviction,
  active-page fallback, main-row preservation, and navigation behavior.
- Exceed 2,048 queued child events and 1,024 transcript items; retained content
  remains bounded, ordered, and explicitly truncated/evicted where required.
- Verify unknown/stale child lineage fails closed or enters the bounded recovery
  bucket; it cannot mutate root state.
- Assert root transcript and parent model context remain free of child message,
  tool, and authorization content throughout pressure and resync.

### `RPF-007`: Fullscreen Child-Page Fault Injection

- Run child streaming and permission overlays through resize, zero-size, 1x1,
  narrow viewport, Unicode/CJK/emoji, large tool output, and image-protocol
  rendering cases.
- Inject fail-on-Nth terminal writes during child-page full redraw,
  differential redraw, overlay redraw, resize, image cleanup, and page return.
- Race Enter/Escape, child completion, authorization resolution, reconnect, and
  child eviction without corrupting focus, scroll, page identity, or composer
  isolation.
- Panic-unwind and runtime-render failures must balance synchronized output and
  restore terminal modes/cursor exactly as the generic `pi-tui` contract
  requires.
- Returning from or evicting a child page restores a valid main viewport and
  does not submit child-page keystrokes into the main composer.
- Extend the offline tmux smoke suite only for lifecycle behavior that cannot be
  proven with the virtual terminal; keep ordinary coverage deterministic.

### `RPF-008`: Soak And Performance Evidence

- Add `scripts/operation-tree-fault-soak.sh` with a fixed, named, offline
  schedule and per-case timeout diagnostics.
- The repeated schedule includes at least: delegation-wait abort,
  child-authorization abort, stalled-provider cancellation, completion/abort
  race, terminal persistence uncertainty, retention gap/reconnect, shutdown
  with a pending child, and child-page bounded pressure.
- Run the new fault schedule for at least 50 iterations with zero leaked
  waiters/tasks/guards, duplicate terminals, unbounded growth, or nondeterministic
  sleeps.
- Retain the 0.5.4 delegation and fullscreen schedules for at least 20
  iterations each.
- Record controlled performance for high-rate child events, 32-child retention,
  maximum retained child transcript, reconnect rebuild, cancellation latency,
  and recovery scan. Use generous regression budgets separate from correctness.
- Record cold TypeScript/Wasm compilation separately from invocation deadline;
  do not classify active JIT/regalloc work as a runtime deadlock.

### `RPF-009`: Version And Release Convergence

- Update `docs/architecture/current-state.md` only with implemented and tested

- Audit stable Rust API, RPC, ProductEvent, UI snapshot, session/outbox schemas,

- Update root/crate changelogs with completed behavior, not planned claims.
- Advance workspace packages, Cargo.lock, binary output, release scripts, and

## Required Fault Matrix

The implementation may expand this matrix, but it cannot remove a row without
documenting an equivalent stronger test at the owning boundary.

| Phase | Injection/race | Required convergence |
| --- | --- | --- |
| Delegation wait | abort/deny/approve/drop | one rejected/cancelled/continued decision; no waiter |
| Child admission | abort/shutdown/capability revoke | no orphan child guard or task |
| Provider start | permanent pending/timeout | cancellation wakes and returns once |
| Provider stream | partial output/error/late delta | bounded child-only output; one failure/cancel |
| Child tool auth | approve/deny/abort/reconnect | child-scoped single resolution; no parent leak |
| Child terminal | completion versus abort | one winning terminal; loser is stale |
| Parent delivery | receiver drop/shutdown | no duplicate delivery or detached execution |
| Folded write | definite writer failure | typed failure without false durable completion |
| Commit boundary | manifest/outbox uncertainty | durable RecoveryPending; zero contradiction |
| Event delivery | subscriber lag/retention gap | commit progresses; explicit resync |
| Snapshot/live | reconnect interleaving | ordered, deduplicated final projection |
| Child retention | capacity overflow/active eviction | bounded deterministic navigation/state |
| TUI render | I/O failure/panic/resize/image | balanced terminal cleanup and valid page |
| Runtime shutdown | any active phase | closed admission, drained/cancelled tree, durable obligations retained |

## Implementation Order

The intended order minimizes false test infrastructure and keeps ownership
clear:

1. Freeze the state/failure matrix and add resource observations (`RPF-001`).
2. Close cancellation/shutdown lifetime gaps before injecting downstream faults
   (`RPF-002`).
3. Add authorization and provider races using the frozen lifecycle
   (`RPF-003`, `RPF-004`).
4. Add durable write/outbox uncertainty and recovery evidence (`RPF-005`).
5. Stress event/snapshot/reconnect and bounded child projections (`RPF-006`).
6. Apply product child-page faults through the existing TUI test boundary
   (`RPF-007`).
7. Freeze repeated schedules and performance evidence (`RPF-008`).
8. Perform version, protocol/API, documentation, and clean release convergence
   (`RPF-009`).

Tasks may be reordered when a discovered bug changes the dependency graph. Any
scope or decision change must update this ledger and its acceptance text before
the release is claimed complete.

## Completion Invariants

0.5.5 is complete only when all of the following are true:

- every admitted operation tree reaches one committed terminal, a definite
  pre-obligation failure, or durable RecoveryPending;
- every accepted delegation produces at most one parent-visible terminal tool
  result across completion, failure, rejection, cancellation, and recovery;
- cancellation and shutdown leave no orphan authorization waiter, child task,
  scheduler guard, running root, or actionable stale request;
- late provider/tool/event activity after terminal cannot mutate model context,
  durable facts, folded state, or UI projection;
- persistence uncertainty never appears as contradictory committed completion;
- EventHub pressure never blocks commit and every retention gap causes explicit
  snapshot/resync behavior;
- child projections stay within fixed capacities under live, reconnect, and
  eviction pressure while the root transcript remains unpolluted;
- fullscreen child pages preserve terminal cleanup, focus, scroll, navigation,
  Unicode width, resize, image, overlay, and composer isolation under injected
  faults;
- the 50-iteration operation-tree fault schedule and retained 20-iteration
  delegation/TUI schedules pass deterministically offline;
- stable API/protocol/schema decisions, migration, version, changelog, release
  evidence, and checked-in snapshots match the implementation;
- the final release gate passes from the complete candidate tree; `git diff
  --check` is clean and the candidate contains only scoped implementation,
  plan, evidence, and the separately queued 0.5.6 document. Creating the
  maintainer commit/tag is a publication action outside this implementation
  plan and was intentionally not performed automatically.
