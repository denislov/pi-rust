# `pi-rust` 0.1 → 0.2 Public API Diff

> Status: final `0.2.0` release input for the breaking architecture convergence project.
>
> Comparison baseline: `605bb8a chore: checkpoint crates before breaking convergence`.
>
> Current release: Rust crates `0.2.0`, RPC/ProductEvent/Snapshot protocol family major `2`.

## Scope and evidence

This document records externally relevant differences between the 0.1 baseline and the current convergence branch. Evidence comes from:

- the public exports at `605bb8a` and at the current `pi_ai::api`, `pi_agent_core::api`, and `pi_coding_agent::api` facades;
- `api_boundary_guards`, `public_api`, `product_runtime_boundary_guards`, `protocol_events`, and RPC wire tests;
- the canonical public event, snapshot, reconnect, and RPC command types.

This source-and-contract diff is paired with machine-readable rustdoc JSON
snapshots generated by `scripts/release-api-snapshots.sh`. The release artifact
contains one snapshot per active library crate, workspace metadata, compiler
identity, and SHA-256 checksums. `cargo-public-api` may be used as an additional
human-readable diff, but is not the authority for whether the machine snapshot
exists.

The normalized public-item projection is generated with the pinned
`public-api 0.52.0` library. The alpha freeze hashes and item counts are stored
under `docs/api-snapshots/0.2.0-alpha.1/`; full JSON remains a release artifact.

## Summary

| Area | 0.1 baseline | 0.2 target | Breaking impact |
|---|---|---|---|
| Stable imports | Stable facades existed, but they were flat and several implementation/root compatibility paths remained reachable | Stable imports use an explicit category below each crate's `api` module | Old root, implementation-module, and flat-facade paths stop compiling |
| Provider registry | `pi_ai::registry::*` was public and doc-hidden | `registry` is private; scoped types remain in `pi_ai::api` | Import-path break; global-style access is unavailable |
| Agent-core modules | Flow, transcript, compaction, resources, proxy, session context, and related modules were public/doc-hidden | Modules are private and their supported contracts are re-exported by `pi_agent_core::api` | Import-path break; supported symbols remain available through the facade |
| Tool invocation | `ToolFn` received arguments and an optional update callback | `ToolFn` also requires `ToolExecutionContext`; `AgentTool::new_text` callbacks receive context plus arguments | Callback signature break; tools gain scope/turn/call/cancellation association |
| Coding-agent root surface | Deprecated root exports and doc-hidden utility modules mirrored CLI/runtime/tool symbols | Root exports are deleted and all 15 implementation modules are private | Import-path break |
| Product event metadata | Public event exposed envelope fields directly and carried deprecated string `family` and `kind` fields in addition to typed payload | The canonical envelope is opaque; sequence, typed payload/family/kind, association, generation, terminal, and durability metadata are available through accessors | Struct field and serialized shape break |
| Capability revocation | Public enum included `CancelMatchingOperations` although no runtime cancellation implementation existed | `FutureOnly` remains the default; privileged `capability_control()` installs a generation and emits `RequestCancelOlderOperations` with exact request targets | Old fake variant removed; new control/outcome/event contract is explicit and cooperative |
| Snapshot cursor | Cursor carried sequence and capability generation | Cursor also requires stream identity and snapshot protocol major; wire uses camelCase | Struct construction and wire break |
| Reconnect | Callers could reconnect with a bare event sequence | Canonical reconnect validates a complete cursor | Method/request migration required |
| Submitted operation state | Included unreachable `Accepted` state | Starts at `Running`, then becomes terminal | Enum variant removal |
| RPC prompt replay | `afterSnapshotSequence` accepted a scalar sequence | `afterSnapshotCursor` requires the full cursor object | RPC request break |
| Plugin Flow extension | Internal model advertised arbitrary Flow extension points with no runtime consumer | Model, capability counter, registry storage, and collector are deleted | Internal non-contract surface intentionally unavailable |

## `pi-ai`

### Removed module path

```rust
// 0.1: no longer supported
use pi_ai::registry::{AiClient, ApiProvider, ProviderRegistry};

// 0.2
use pi_ai::api::client::AiClient;
use pi_ai::api::provider::{ApiProvider, ProviderRegistry};
```

Supported symbols are organized under `model`, `conversation`, `stream`,
`hooks`, `client`, `auth`, `provider`, `error`, `transport`, `compatibility`,
`images`, and `testing`. The breaking changes are that `registry` is private and
flat `pi_ai::api::{...}` imports no longer compile. Provider registration and
lookup must be scoped to an `AiClient`/`ProviderRegistry`.

### Stable facade additions/removals

The categorized facade preserves the supported provider/model/stream contracts,
but their flat import paths are intentionally removed. Provider wire helpers and
test fixtures remain separated from normal model/conversation consumers.

## `pi-agent-core`

### Modules made private

The following 0.1 doc-hidden module paths are no longer public:

- `pi_agent_core::agent`
- `pi_agent_core::agent_turn_flow`
- `pi_agent_core::branch_summary`
- `pi_agent_core::compaction`
- `pi_agent_core::convert`
- `pi_agent_core::env`
- `pi_agent_core::errors`
- `pi_agent_core::flow`
- `pi_agent_core::harness`
- `pi_agent_core::hooks`
- `pi_agent_core::proxy`
- `pi_agent_core::queues`
- `pi_agent_core::resources`
- `pi_agent_core::session_context`
- `pi_agent_core::shell_output`
- `pi_agent_core::transcript`
- `pi_agent_core::truncate`
- `pi_agent_core::types`

Supported contracts from those owners are re-exported through the categorized
`agent`, `tool`, `flow`, `execution`, `resources`, `compaction`, and
`transcript` facades. `testing` contains deterministic owner-test support.
Examples include:

- Flow: `Action`, `Flow`, `FlowError`, `FlowNode`, `FlowOutcome`, `FlowRunOptions`, `NodeId`;
- agent turn: `AgentTurnContext`, `AgentTurnFlow`, node contracts, and focused node functions;
- compaction and branch summary contracts;
- proxy streaming contracts;
- resource loaders;
- session context and in-memory storage;
- transcript/session tree types;
- truncation and shell-capture helpers.
- agent-message conversion/context assembly helpers.

`pi_agent_core::api` is now the only public root module, and it has no flat
exports. Conversion helpers such as `default_convert_to_llm`,
`assemble_context`, `convert_to_context`, and `bash_execution_to_text` live in
their owning `agent`/`execution` category.

### Newly supported facade symbols

The 0.2 facade explicitly adds previously module-scoped supported contracts, including:

- branch-summary types/functions and `BranchSummaryError`;
- compaction estimate/prepare/summarize functions;
- Flow primitives;
- proxy request/stream types and functions;
- session-context types/functions;
- transcript/session metadata and conversion helpers;
- `execute_shell_with_capture`.

These are facade additions, not a promise that their old module paths remain valid.

### Tool callback signature

`pi_agent_core::api::tool` now exports `ToolExecutionContext`. Core constructs
one at the actual sequential or parallel invocation and passes it to every
`ToolFn`. It contains a caller-owned generic scope ID, agent turn, tool-call ID,
tool name, and cancellation token. `AgentConfig::tool_execution_scope` supplies
the generic scope; `pi-coding-agent` sets it from the admitted operation ID.

This intentionally breaks two-argument `ToolFn` closures and one-argument
`AgentTool::new_text` callbacks. There is no legacy adapter because silently
discarding execution identity would preserve the capability escape this release
removes.

## `pi-coding-agent` Rust API

### Root modules made private

The following 0.1 doc-hidden paths are no longer public; supported symbols remain available from `pi_coding_agent::api`:

- `pi_coding_agent::args`
- `pi_coding_agent::config`
- `pi_coding_agent::error`
- `pi_coding_agent::input`
- `pi_coding_agent::interactive`
- `pi_coding_agent::models`
- `pi_coding_agent::print_mode`
- `pi_coding_agent::prompt_options`
- `pi_coding_agent::protocol`
- `pi_coding_agent::request`
- `pi_coding_agent::resources`
- `pi_coding_agent::runtime`
- `pi_coding_agent::session`
- `pi_coding_agent::theme`
- `pi_coding_agent::tools`

Closing these modules also exposed and removed two functions with no repository consumer: `resources::print_diagnostics` and `session::session_root_from_run_options`. They are removals, not facade migrations.

The curated facade is split by downstream use scenario:

- `runtime`: session/runtime lifecycle and errors;
- `operation`: typed work requests and outcomes;
- `event`: product-event envelopes, families, receivers, and durability views;
- `client`: connection, control, snapshot, reconnect, submission, and detach;
- `view`: read-only session/export/capability/profile DTOs;
- `protocol`: versioned RPC/JSONL DTOs, negotiation, and adapters;
- `cli`: explicitly supported command-line scripting contracts, further split
  into `command`, `configuration`, `input`, `print`, `resources`, `runtime`,
  `runner`, and `theme` with no flat CLI exports;
- `testing`: reserved deterministic downstream test support.

There are no flat `pi_coding_agent::api::{...}` exports. Internal service,
repository, Flow, provider-client, plugin-host, and TUI-component types remain
unavailable. `CapabilityRevocationPolicy` is internal; downstream consumers see
only the product-event revocation view.

The client control surface separates authority by use case:

- `CodingAgentOperationControl` exposes exact-operation `abort` only;
- `CodingAgentPromptControl` owns Prompt-specific abort, steer, follow-up, and
  queued-draft submission;
- both are scoped to a client connection generation and operation ID; neither
  exposes the scheduler, cancellation token, channel, or coordinator.

`interactive` and built-in tool implementation tests now compile as owner-crate
internal tests. Supported high-level CLI construction/filtering is under
`api::cli`; individual bash/edit/read/write execution internals and the
interactive scripted harness are not embedding contracts.

### Removed deprecated root exports

The following root imports are deleted. Import the supported items from
`pi_coding_agent::api::cli`:

- `CliArgs`, `CliMode`, `help_text`, `parse_args`;
- `CliError`;
- `PrintModeOptions`, `run_print_mode`;
- `PromptRunOptions`;
- `CliRunOptions`, `PromptInvocation`, `SessionMode`, `SessionRunOptions`;
- `DEFAULT_MODEL_ID`, `DEFAULT_SYSTEM_PROMPT`;
- `build_agent_config`, `effective_no_context_files`, `effective_session_dir`, `select_model`;
- `ResolvedSessionTarget`, `encode_cwd`;
- `builtin_tools`.

The root CLI functions and output type are also consumed through the stable facade:

```rust
use pi_coding_agent::api::cli::runner::{
    CliOutput, run_cli, run_cli_with_options, run_cli_with_options_and_stdin,
};
```

### Product event shape

Removed public fields from `CodingAgentProductEvent`:

```rust
pub family: String;
pub kind: String;
pub sequence: u64;
```

Use:

```rust
event.event();        // &CodingAgentProductEventKind
event.sequence();     // u64
event.family_typed(); // CodingAgentProductEventFamily
event.kind_name();    // stable snake_case variant name for display/metrics
event.parent_operation_id(); // direct parent for child operations
event.root_operation_id();   // stable root across nested child operations
event.session_id();           // only when carried by an authoritative event fact
event.capability_generation(); // Option<u64>, admitted operation generation
```

Consumers should pattern-match the typed family payload enums. Reconstructing control flow from formatted legacy strings is unsupported.

Operation-associated ProductEvents expose the exact capability generation
frozen at admission. This value can differ from the current snapshot cursor
after a future-only capability update. Global events and recovery from legacy
logs without a persisted generation return `None`.

Root `terminal_operation()` metadata is resolved from the admitted operation
kind and its permitted terminal evidence. Payload variants are not sufficient:
for example, `PromptFailed` may terminate either Prompt or Compact depending on
the authoritative admission contract. Local terminal events such as tool or
session-write completion keep `terminal_operation()` absent.

Product-event durability adds `DerivedFromSession` for committed recovery facts,
`PersistenceUncertain` for partial commits, and `PersistenceFailed` for a write
known not to have committed. Consumers with exhaustive enum matches must add
all branches; treating them as `LiveOnly` loses recovery semantics. Session
events now expose `WriteFailed { status: Definite | Uncertain, ... }` instead of
reporting finalization failure as `WriteSkipped`.

The canonical envelope adds `delivery_class()` with `Data`, `Terminal`,
`Control`, and `Recovery`. All classes share the sequenced bounded stream;
sequence gaps require fresh-snapshot recovery rather than priority replay across
missing prerequisites.

### Capability revocation

Removed:

```rust
CodingAgentProductEventCapabilityRevocation::CancelMatchingOperations
```

`FutureOnly` remains the normal update policy: an active operation keeps its
frozen generation. The new privileged
`CodingAgentCapabilityControl::revoke_older_operations` installs a generation,
requests cancellation for active older-generation identities, and returns
`CodingAgentCapabilityRevocationOutcome`. Product events use
`RequestCancelOlderOperations` and carry
`cancellation_requested_operation_ids`.

The wording is deliberate. Cancellation is cooperative and the outcome reports
requests, not proof that arbitrary trusted Rust code has already terminated.
Production scheduler admission carries the frozen generation into the active
identity and rejects a stale snapshot if a revocation won the race before the
identity was registered.

### Operation concurrency and capability projection

The 0.2 runtime no longer treats every root operation as one global busy slot.
Admission now distinguishes:

- one `SessionWriteRoot` per session;
- a bounded set of `NonSessionRoot` operations, with a current default limit of
  four per runtime control instance;
- exclusive `RuntimeWrite` operations;
- unguarded `ReadOnly` and `Control` paths.

`CodingAgentCapabilities` follows the same categories. For example, an active
agent invocation no longer makes Prompt busy merely because both are roots, and
committed-state reads such as export may coexist with a session writer. Plugin
reload remains busy while any root is active because current runtime writes are
exclusive.

`CodingAgentSnapshot::active_operation` remains a single primary-operation
projection for 0.2 compatibility. When independent slots coexist, it prefers
the exclusive runtime write, then the session writer, then the oldest
non-session root. It must not be interpreted as a complete operation registry.

### Snapshot cursor

0.1:

```rust
CodingAgentSnapshotCursor {
    last_event_sequence,
    capability_generation,
}
```

0.2 target:

```rust
CodingAgentSnapshotCursor {
    stream_id,
    snapshot_protocol_major,
    last_event_sequence,
    capability_generation,
}
```

The serialized field names are:

```json
{
  "streamId": "session-id",
  "snapshotProtocolMajor": 2,
  "lastEventSequence": 42,
  "capabilityGeneration": 7
}
```

The numeric major in this example is the current `0.2.0` contract.
Callers must still use the value received in the snapshot rather than hard-code
it, because future incompatible releases will change the family major again.

### Reconnect

Added canonical method:

```rust
connection.reconnect_from_cursor(&cursor)?;
```

It rejects:

- a cursor from a different session stream;
- an incompatible snapshot protocol major.

The older sequence-only reconnect path is not the canonical adapter contract.

### Submitted operation status

Removed unreachable variant:

```rust
CodingAgentSubmittedOperationStatus::Accepted
```

Observable state now starts at `Running` after atomic admission/commit and transitions to terminal evidence or typed uncertainty.

### Removed error surface

`CodingSessionError::StaleClientConnection` was removed with the parallel stale-client registry path. Client lifecycle failures now use the canonical lifecycle/generation rejection contract.

### Operation admission

`CodingAgentOperation` remains the public operation enum. Internally, every variant now has one descriptor containing its admission class, dispatch mode, submitted kind, outcome family, and permitted terminal evidence. Internal scheduler, service, Flow-node, and plugin-registry types are intentionally not exported from the stable facade.

### Runtime-owned operation submission

0.2 adds the stable runtime verb:

```rust
let task: CodingAgentOperationTask = session.submit(operation)?;
let operation_id = task.operation_id();
let outcome = task.join().await?;
```

The migrated operations are `PluginCommand`, `InvokeAgent`, and `InvokeTeam`.
They freeze admission/capability state, retain their root permit and client
terminal association inside the runtime-owned task, and no longer require an
adapter to move the entire `CodingAgentSession` into a spawned future. These
submitted NonSessionRoot operations may coexist with one SessionWriteRoot.

PluginCommand is outcome-only and uses the frozen plugin capability snapshot;
it does not synthesize a ProductEvent or gain permission to mutate durable
session state. Compatibility `run(PluginCommand)` delegates to
`submit(...).join()`.

Outcome-only behavior is no longer inferred from an empty terminal-evidence
list. Every operation descriptor explicitly declares either `ProductEvent` or
`OutcomeAcknowledgement`. The former requires a non-empty permitted-evidence
set and exactly one root terminal event; the latter forbids root evidence and
requires `acknowledge_outcome` rather than an event-sequence acknowledgement.

For these roots, `CodingAgentOperationTask::operation_id()`, submitted-operation
state, root Agent/Team ProductEvents, and `AgentInvocationOutcome::operation_id`
or `AgentTeamOutcome::operation_id` now use the same admitted ID. Child Prompt
and member operations keep separate child IDs exposed by their parent events.

Child IDs are now backed by a runtime-wide guarded registry rather than a
lineage-only check. Delegated Agent/Team wrapper IDs and their internal Prompt
IDs are distinct admitted nodes. A parent cannot publish terminal completion
while descendants remain active; early parent release cancels descendants and
keeps shutdown/root capacity in drain state until the child tree exits. Clients
should continue treating these IDs as opaque correlation keys and must not infer
that a child may outlive its parent.

Other operation classes continue through `CodingAgentSession::run` for now.

`ApproveDelegation` and `RejectDelegation` are explicitly SessionWriteRoot
operations. Approval records a durable fact before running the delegated child
and may persist newly requested confirmations afterward; rejection records the
durable rejected decision. They therefore return `Busy` when another session
writer such as Prompt is active and are intentionally not accepted by
`CodingAgentSession::submit`. Control is reserved for non-durable priority
signals.
Calling `submit` with an operation that has not yet migrated returns
`UnsupportedCapability`; it does not silently fall back to adapter-owned task
execution. `submit` requires an active Tokio runtime.

## RPC and protocol wire changes

### Agent/Team background admission

Successful `invoke_agent` and `invoke_team` responses now include the admitted
root operation ID:

```json
{
  "type": "response",
  "command": "invoke_agent",
  "success": true,
  "data": {
    "operationId": "op-...",
    "profileId": "coder",
    "task": "implement parser"
  }
}
```

The same ID appears in root Agent/Team ProductEvents and the typed terminal
outcome. Agent/Team execution is background work at the RPC adapter boundary,
so a foreground Prompt may be admitted while it is running. Prompt
steer/follow-up/abort still target only the foreground Prompt; session
replacement waits for both foreground and background roots to drain.

Once an RPC runtime session exists, `get_state.data.eventStreamId` is the
canonical snapshot stream ID. Clients combine it with
`snapshotVersion.major`, `snapshotSequence`, and `capabilityGeneration` to
construct `afterSnapshotCursor`. `sessionId` remains the session identity and
must not be substituted for `eventStreamId`; a state without `eventStreamId`
has no live ProductEvent stream to reconnect.

### Prompt reconnect field

Removed:

```json
{"afterSnapshotSequence": 42}
```

Required replacement:

```json
{
  "afterSnapshotCursor": {
    "streamId": "stream-id",
    "snapshotProtocolMajor": 2,
    "lastEventSequence": 42,
    "capabilityGeneration": 7
  }
}
```

### Product events

Consumers must decode the typed event payload. The removed top-level legacy `family`/`kind` string fields are not emitted as compatibility duplicates.

### Protocol major release boundary

RPC, ProductEvent, and UI Snapshot advertise family version `2.0` for
`0.2.0`. Durable session manifest/writer version remains `1`; P6 did not
require an incompatible durable shape.

## Intentionally unavailable internal surfaces

The following are not stable API migrations; they are deliberate removals of internal parallel paths:

- global provider registry bridges;
- retired agent-loop wrappers;
- direct `IntentRouter` operation admission;
- `CodingAgentSession` convenience pass-throughs that bypass canonical `run`/scheduler dispatch;
- replay handles that bypass snapshot/reconnect recovery;
- legacy internal `ClientConnection`/`SubmittedOperation` projection;
- arbitrary plugin Flow extension declarations;
- raw service access from plugin/tool hosts.

Downstream code that reached these internals must migrate to the stable facade and canonical operation/snapshot/product-event contracts rather than request replacement exports.
