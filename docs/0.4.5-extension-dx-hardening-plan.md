# Extension Developer Experience And Platform Hardening Plan For 0.4.5

## Status And Scope

Target workspace version: **`0.4.5`**. Status: **Skipped by the 2026-07-19
Extension scope freeze**.

All `DXH-*` and `REL-045-001` tasks are **Skipped**. General non-Extension
hardening may be replanned independently, but this Extension DX plan is not a
release obligation and cannot be treated as completed evidence.

This release turns the implemented architecture into a coherent developer
platform: complete TypeScript SDK/CLI/harnesses, maintained examples and docs,
final performance budgets, security/compatibility hardening, and train-wide
evidence. It adds no second authoring language/runtime and no new owner bypass.

See the [`0.4.x` roadmap](0.4.x-architecture-extension-platform-roadmap.md).

## Objectives

- publish one generated and versioned `@pi-rust/extension-sdk`;
- provide create/build/validate/inspect/audit/dev/test/package/compat workflows;
- pin compiler/bundler/componentizer/WIT/Wasm validation tools;
- provide deterministic virtual Host API, permission, state, update, recovery,
  and Workbench harnesses;
- maintain TypeScript/Wasm and Application Profile examples;
- enforce `ADR-009` performance budgets across all train paths;
- complete security, fuzz/property, pressure, compatibility, and legacy-absence
  evidence;
- publish stable architecture, migration, troubleshooting, and protocol records.

This release productizes the locked internal build/conformance harness introduced
in `0.4.2` and extended by `0.4.3`/`0.4.4`; it does not postpone the first
reproducible way to build or test an extension until the end of the train.

## Non-Goals

- Another extension language/runtime or direct Node execution.
- Marketplace, billing, signing authority, or automatic distribution service.
- Arbitrary HTML/WebView, raw adapter widgets, or full npm/Node compatibility.
- Weakening correctness/security to meet a benchmark.
- Reopening completed ownership/state/Workbench decisions without superseding ADR.

## Detailed Acceptance

### `DXH-001`: SDK And Templates

- Publish SDK types generated from WIT/schema; generated ABI digest and supported
  version ranges are machine-readable.
- Provide typed extension definition, operations/tools/hooks, grants/leases,
  state/facts, jobs/subscriptions, Workbench, actions, and profile APIs.
- TypeScript API maps cancellation to `AbortSignal`, progress/errors to structured
  types, and never exposes ambient Node/host objects.
- Templates include locked dev dependencies and reproducible build metadata.
- Host runtime remains independent of Node.js/build tooling.

### `DXH-002`: CLI Workflow

Provide offline-capable equivalents of:

```text
extension create
extension build
extension validate
extension inspect
extension permission-audit
extension dev-run
extension test
extension package
extension compatibility-check
```

- Pin TypeScript compiler, bundler, componentizer, WIT generator, Wasm validator,
  and package schema versions.
- Validate forbidden Node/WASI/native/dynamic imports before packaging.
- Emit source-mapped, redacted build/component/Host API diagnostics.
- Dev mode cannot weaken production grants or runtime isolation silently.
- Public commands wrap or replace the earlier internal harness through recorded
  fixture compatibility; they do not create a second build/package format.

### `DXH-003`: Test Harnesses

- Virtual Host API supports grants/denials/revoke/stale generation, cancellation,
  deadlines, progress, quotas, traps, state, facts, jobs, subscriptions, update,
  rollback, and missing-extension replay.
- Virtual Workbench validates snapshots/patches, gaps, paging, limits, actions,
  profiles, and multi-client state.
- Fixtures are deterministic, offline, temporary-directory isolated, and never
  touch developer configuration.
- Conformance fixtures use the same generated SDK and installed package format as
  real extensions.

### `DXH-004`: Documentation And Examples

- Publish minimal tool/command, stateful workflow, background subscription, Review
  Workbench, and Incident/Release Console examples.
- Document authoring/runtime distinction, supported TypeScript/npm subset, WIT/API
  versioning, permissions, state/facts, package/update, UI portability, testing,
  performance, security, and troubleshooting.
- Document all intentional protocol/public API breaks and supported data migration.
- Document the intentional Lua/native-extension break and supported replacement
  path without shipping a compatibility runtime or promising an automated
  migration tool.
- Distinguish unsupported legacy extension runtimes from supported versioned
  durable decoders, schema migrations, and negotiated public protocols.

### `DXH-005`: Performance Budgets

Use accumulated baselines to set release thresholds for:

- no-extension runtime overhead and operation admission;
- independent-session concurrency, writer queue, commit/outbox/snapshot, recovery,
  reconnect;
- component validation, cold/warm invocation, memory, artifact, cancellation,
  trap disposal, cache;
- state/fact/update/job/subscription pressure;
- Workbench snapshot/patch/render/virtualization/lag/multi-client recovery;
- diagnostic/log and recovery-management bounds.

Correctness assertions remain deterministic; environment-sensitive timing uses
controlled regression evidence. Threshold changes require measured ADR updates.

### `DXH-006`: Hardening

- Fuzz/property-test manifest/package/WIT/schema/patch/state/fact decoders and
  lifecycle state machines where practical.
- Test archive traversal/symlink/decompression, digest/lock, forbidden imports,
  capability escape, revoke races, cancellation races, trap/OOM/fuel, output/log
  pressure, update crashes, replay gaps, and profile/action conflicts.
- Audit secrets/redaction across diagnostics/events/snapshots/logs/fixtures.
- Verify no raw service, reverse dependency, alternate runtime, `mlua`, Lua API,
  Rust extension ABI/tier, direct TS/JS execution, or runtime Node requirement.
- Run dependency/security review for the pinned Wasm and TypeScript toolchains.

### `DXH-007`: Train Evidence

- `ADR-001` through `ADR-012` match final implementation and tests.
- All `0.4.0` through `0.4.4` completion records and later fixes remain valid.
- Architecture current-state evidence, protocol inventories, public API snapshots,
  WIT/schema hashes, SDK versions, examples, and docs agree.
- No open train debt, temporary bypass, duplicate owner, or unresolved shim exists.
- Owned compatibility code has an explicit family/version range, test matrix,
  support policy, and removal rule; it is not classified as an unresolved shim.

### `REL-045-001`: Release And Train Completion

- Close all `DXH-D*` debt and any train debt discovered after earlier releases.
- Update every workspace package to `0.4.5` after closure.
- Add root and owning-crate `0.4.5` changelog sections summarizing platform
  stability, migrations, security, performance, and developer experience.
- Run all focused, full, TUI, conformance, security, compatibility, generated
  artifact, and performance gates.
- Complete this plan and the train roadmap completion record.

## Release Gates

`0.4.5` completes only when all tasks/debt and all train-level debt close; all six
version completion records exist; ADRs, architecture, protocols, APIs, generated
schemas/bindings, SDK, examples, versions, and changelogs agree; full workspace,
TUI, TypeScript/Wasm conformance, state/update, Workbench, security, compatibility,
legacy-absence, and performance gates pass offline.

No credentials, live-provider dependency, ambient authority, raw service, reverse
dependency, alternate runtime, direct TS/JS execution, runtime Node.js dependency,
Lua support, Rust extension ABI/tier, unresolved recovery, or unowned temporary
compatibility shim may remain. Supported versioned durable decoders, schema
migrations, and negotiated public-protocol compatibility remain when explicitly
owned and tested.
