# Migration Guide: 0.5.1 to 0.5.2

This release contracts `pi-coding-agent` around product operations, events,
snapshots, protocol contracts, Extension administration, and high-level CLI
execution. It also removes internal workflow and adapter duplication without
changing durable session or wire-protocol semantics.

## CLI Facade Contraction

The stable `pi_coding_agent::api::cli` facade now contains only `runner`.
The following low-level categories have been removed:

- `api::cli::command`: argument parser, help rendering, resolved request and
  diagnostic implementation types;
- `api::cli::configuration`: config paths, auth resolution, settings merge,
  model selection, and model-rotation parsing;
- `api::cli::input`: stdin and `@file` preprocessing internals;
- `api::cli::resources`: context/resource discovery and built-in tool
  construction;
- `api::cli::theme`: theme parsing, token, detection, export, syntax, and
  watcher internals.

These were product implementation details rather than supported embedding
contracts. Applications should invoke the complete CLI through
`api::cli::runner::run_cli` or `run_cli_stdio`. Tests that need controlled
models, tools, sessions, or stdin may continue to use
`run_cli_with_options`/`run_cli_with_options_and_stdin` with the retained
`api::runtime::CliRunOptions` contract.

There is no public replacement for parsing individual flags, resolving config
paths, merging settings files, constructing product built-in tools, or parsing
theme internals. Those concerns remain owned by the product runner.

## Recategorized Types

The following 0.5.1 moves are the canonical 0.5.2 import locations:

| Old import | New import |
| --- | --- |
| `api::cli::runtime::PromptInvocation` | `api::operation::PromptInvocation` |
| `api::cli::runtime::PromptRunOptions` | `api::operation::PromptRunOptions` |
| `api::cli::runtime::CliRunOptions` | `api::runtime::CliRunOptions` |
| `api::cli::runtime::SessionRunOptions` | `api::runtime::SessionRunOptions` |
| `api::cli::runtime::SessionMode` | `api::runtime::SessionMode` |
| `api::cli::print::PrintModeOptions` | `api::protocol::PrintModeOptions` |
| `api::cli::print::run_print_mode` | `api::protocol::run_print_mode` |

The empty `api::testing` module and the unused `test-harness` feature have been
removed. The explicit non-default `test-support` feature remains the only
supported way for downstream deterministic tests to enable test support.

## Runtime And Adapter Changes

`WorkflowService` and fixed-operation step-loop internals were private and have
been deleted. Public callers continue to submit `CodingAgentOperation` through
`CodingAgentSession`; typed outcomes, terminal evidence, cancellation,
recovery, and session durability are unchanged.

Adapters now consume the same public ProductEvent and snapshot contracts and
use typed outcome extraction. No RPC, ProductEvent, or UI snapshot protocol
version changed in this release:

| Protocol family | Version |
| --- | --- |
| RPC | `2.1` |
| ProductEvent | `2.2` |
| UI snapshot | `2.1` |

Existing JSONL clients do not need a wire migration. The protocol snapshot and
stdout/stderr cleanliness tests remain release gates.

## Extension Contract

The TypeScript/Wasm Extension framework is retained unchanged. Manifest/WIT
schemas, immutable package installation, grants and operation leases,
generation/revoke fencing, Host-call validation, per-invocation Wasmtime
isolation, PluginLoad durable facts/outbox records, restart reload, public
trusted-host operations, Wasmtime `46.0.1`, and the Rust 1.94 minimum are all
preserved. Previously skipped Extension productization work remains skipped.
