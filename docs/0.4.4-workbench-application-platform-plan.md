# Workbench And Application Composition Plan For 0.4.4

## Status And Scope

Target workspace version: **`0.4.4`**. Status: **planned and dependency-reviewed;
blocked by completed `0.4.3`**.

This release builds portable semantic views and Application Profiles on the safe
execution/state kernel, then integrates them across TUI, RPC, snapshots, and the
embedding facade. It does not introduce arbitrary HTML, raw TUI widgets, CSS, or
another extension runtime.

See the [`0.4.x` roadmap](0.4.x-architecture-extension-platform-roadmap.md).

## Detailed Acceptance

### `WAP-001`: Protocol Candidates And Composition Decisions

- Publish candidate view/node/action/data-source identities and schema versions;
  final stability waits for both conformance applications and both adapters.
- Use initial `ViewSnapshot` plus typed revision-based `ViewPatch`.
- Freeze client-local versus shared state and gap-resync behavior.
- Accept one profile base/overlay model, precedence, conflicts, missing dependency,
  user override, degraded/unavailable, and mandatory-surface rules.

### `WAP-002`: View Protocol

- Each client open creates a distinct ViewInstance and revision.
- Host validates node IDs, references, schema, actions, depth, nodes, text, rows,
  logs, patch rate, and encoded size.
- Patches use stable IDs and closed insert/remove/replace/update/replace-children
  operations; stale base revision causes `ViewResyncRequired`.
- Extension may submit full snapshot or patch; host owns accepted revision.
- Workbench remains a semantic protocol, not virtual DOM/CSS/arbitrary UI code.

### `WAP-003`: Contributions And Virtualization

- Start with thin vertical slices of both planned conformance applications so
  materially different layouts, data, actions, and workflows exercise the
  candidate protocol before the complete node/contribution inventory is fixed.
- Support bounded Text/Markdown/List/Tree/Table/Form/Tabs/Split/Diff/Log/Progress/
  Badge/Button/Input/Select/Checkbox/Empty/Error semantics.
- Lists/tables/trees/logs use page/cursor/lazy loading and host-capped ranges.
- Add data sources, actions, menus/palette/context actions, dialogs/forms,
  keybindings, settings, theme tokens, transcript/message renderers, status, and
  composer contributions.
- Every callback/fetch/action is an admitted operation with a lease.
- Client lag coalesces invalidation or resyncs; patch buffers never grow unbounded.

### `WAP-004`: Application Profiles

- One active base profile plus host-mandatory surfaces, ordered extension
  contributions, and user preference overlay.
- Precedence: mandatory security policy > explicit user override > active profile
  > extension defaults > built-in defaults.
- ID conflicts fail activation; keybinding conflicts require explicit resolution.
- Missing required extension makes profile unavailable; missing optional extension
  produces declared degraded mode.
- Profiles cannot hide authorization, recovery, failure, durability, cancellation,
  protocol, or terminal-cleanup surfaces.
- Startup operations still use canonical admission and grants.

### `WAP-005`: TUI

- Render semantic nodes without exposing `pi-tui` component types to extensions.
- Preserve terminal cleanup, cursor placement, Unicode width, resize, scrollback,
  focus, and virtualized rendering.
- Focus, selection, scroll, viewport, expansion, and transient form input remain
  client-local unless an action persists business state.
- Add virtual-terminal tests and offline `scripts/tui-smoke.sh` scenarios.

### `WAP-006`: RPC, Snapshot, And Embedding

- Expose view snapshots/patches/actions/forms/settings/permissions/progress without
  TUI representation.
- Each client owns its view instances and ProductEvent cursor.
- Shared data invalidation schedules separate client refreshes; patches are not
  copied between clients.
- Reconnect/gap yields deterministic fresh snapshot/view resync.
- Print/JSON modes define noninteractive behavior for interactive-only views.
- Publish narrow discovery, permission decision, operation, snapshot, and
  lifecycle APIs without internal registries/services.
- This adapter is a peer of `WAP-005`, not a consumer of TUI implementation.
  Shared semantic fixtures must pass independently through both adapters.

### `WAP-007`: Conformance Applications

- A Review Workbench and Incident/Release Console use the same stable extension,
  state, operation, and Workbench contracts.
- They differ materially in layout, data, actions, and workflow.
- Both run offline through TUI and RPC fixtures with denial, revoke, stale grant,
  missing optional extension, patch gap, lag, and restart scenarios.
- Freeze stable Workbench/Profile schemas only after both applications and both
  adapters pass shared fixtures; record any candidate-to-final migration.

### `REL-044-001`: Release Convergence

- Close all `WAP-D*` debt.
- Update every workspace package to `0.4.4` after implementation/debt closure.
- Add root and owning-crate `0.4.4` changelog sections.
- Update Workbench, Application Profile, ProductEvent, Snapshot, RPC, embedding,
  Manifest/contribution, architecture, and migration evidence.
- Run focused/full/TUI/RPC/conformance/security gates.
- Enforce provisional no-major-regression limits for snapshot/patch validation,
  virtualization, TUI/RPC projection, lag, and multi-client resync.
